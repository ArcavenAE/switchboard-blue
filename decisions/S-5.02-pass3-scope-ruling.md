Issued 2026-06-30. These rulings unblock story-writer (S-5.02 v1.5) and
implementer convergence. Two findings required scope-level decisions before

## Ruling 1 — F-C1 (HIGH): AC-007 Tautological Test / sbctl PC-5 Surfacing

**Finding summary.** `TestSbctlSessionsStatus_QualityFieldPresent` asserts
canned daemon-stub bytes round-trip through sbctl's JSON envelope. No
daemon-side `sessions.list` handler is registered; no daemon code derives a
`quality` field for sessions in the S-5.02 diff. The test cannot regress on
production behavior because production never computes the value being asserted.
sbctl is a thin RPC passthrough for `sessions.list` (cmd/sbctl/main.go:60).

**Ruling: Option A — Defer sbctl PC-5 surfacing to S-7.03.**

Rationale:

1. S-5.02 was already scope-expanded (3→5pts) for p99 work in a prior pass;
   further expansion (+3-5pts for a daemon-side `sessions.list` handler) risks
   wave-5 timeline.
2. The console half of BC-2.06.001 PC-5 is already deferred to S-7.03 (per
   story body §Scope Boundary). Deferring the sbctl half to the same story keeps
   the BC PC-5 surfacing delivery coherent — both halves land together.
3. BC-2.06.001 v1.3 already cites S-7.03 in the Stories trace for the
   "console session-list surfacing" half. Making S-7.03 the single delivery
   point for all PC-5 surfacing is cleaner.
4. Sessions quality surfacing is P1, not P0 — no blocking runtime impact.

**Consequences for S-5.02 v1.5 (story-writer executes):**

- Drop AC-007 from S-5.02 v1.5 story body.
- Drop the tautological test `TestSbctlSessionsStatus_QualityFieldPresent`
  from story task list (implementer does NOT write this test).
- Remove the S-5.02 trace from BC-2.06.001 Stories cell (PO-executed in
  BC-2.06.001 v1.4 — see below).
- Add a scope-note in story §Scope Boundary: "sbctl `sessions.list` quality
  surfacing (BC-2.06.001 PC-5) deferred to S-7.03 per Pass-3 Ruling 1."

## Ruling 2 — F-T3-001 (CRITICAL): BC-2.06.003 v1.5 Pending-Quality Sentinel Not Propagated to S-5.02

**Finding summary.** BC-2.06.003 v1.5 (2026-06-30) added PC-3 normative
obligation: when `rtt_p99_ms == "pending"`, the `quality` field MUST emit
`"pending"`. Added EC-006. S-5.02 v1.4 was bumped for version-pin sweep but
did not add an AC for this behavioral obligation. AC-003 currently says
`quality (green/yellow/red derived from status + rtt_p99_ms)` — no `pending`
enumeration. Implementation is correct
(cmd/sbctl/router_status.go:qualityFromPathEntry handles pending first), but
story-spec doesn't trace the AC. Spec drift.

**Ruling: Approved. Story-writer must add the following to S-5.02 v1.5:**

1. **New AC-008:** "Alias `quality` field MUST be `"pending"` when
   `rtt_p99_ms == "pending"` (BC-2.06.003 v1.5 PC-3 + EC-006)."
   Alternatively: extend AC-003 to enumerate `green|yellow|red|pending` and
   note the pending sentinel rule.
2. **EC row in Edge Cases table:** "alias emits quality:pending when p99
   pending" with expected behavior citing BC-2.06.003 EC-006.
3. **Header bump:** §"Command Surface Reconciliation (BC-2.06.003 v1.4)" →
   `v1.5`.
4. **Compliance Rules table:** bump all rows pinning `BC-2.06.003 v1.4` →
   `v1.5`.

No implementation change required — the behavior already exists in
`qualityFromPathEntry`. This is a story-spec catch-up to match the
implementation and BC v1.5.

path. BC-2.06.003 v1.5 EC-006 requires `quality == "pending"` when
`rtt_p99_ms` is pending; this sentinel path is not exercised in VP-062 v1.0.

Changes:

- Add `TestSbctlMetricsJSON_PropTest` case: `"pending p99 path"` with
  `RTTValue{Valid: false}`, assert quality = "pending".
- Update Property Statement: add pending-sentinel assertion as property 5.
- Source Contract section: add EC-006 citation.
