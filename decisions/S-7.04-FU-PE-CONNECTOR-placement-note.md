## Q1 ‚Äî Order-vs-set upstream diff semantics (FO-1, adv F-P1-007)

**Ruling: the emission diff (mode-line re-emission) keeps order-sensitive semantics;
the connection reconciliation diff MUST use set-equal semantics. The two diffs are
lawfully different.**

**Derivation:**

`equalStringSlices` at `router_config.go:81` is positional. Its role in the SIGHUP
reload path is to suppress a spurious `mode=PE` re-emission when the address list is
unchanged. For the emission surface this is a cosmetic optimization ‚Äî a spurious
re-emission is harmless but noisy. The order-sensitive comparison is acceptable here.

The connection reconciliation diff ‚Äî deciding which upstream connections to tear
down and which to establish after a reload ‚Äî is semantically different. Two configs
that list the same addresses in different orders describe the same desired topology.
Tearing down a live connection because address "A,B" became "B,A" is incorrect
per BC-2.09.001 EC-002 ("sessions maintained; new upstream connections established
without interrupting existing paths"). A reorder MUST NOT trigger redial or teardown.

**Concrete ruling:**

1. The emission diff in `runRouter`'s SIGHUP case (`mgmt_wire.go:560`) keeps
   `equalStringSlices` unchanged ‚Äî a positional order change produces a re-emission,
   which is acceptable noise.
2. The connector reconciliation introduced by this story MUST sort both the running
   set and the loaded set before comparing when computing add/remove deltas. The
   recommended primitive is `upstreamRoutersAsSet(addrs []string) map[string]struct{}`
   in `router_config.go` ‚Äî a free function, pure, fully testable. The reconciler
   computes `added = loaded \ running` and `removed = running \ loaded` as set
   differences, then dials added and tears down removed.

   > **v1.2 implementation note (F-P1-007):** `upstreamRoutersAsSet` was briefly
   > present in `router_config.go` but was deleted at f8fdfd5 as production-dead
   > (no callers; reconcile method builds its set inline). The set-equal semantics
   > are implemented solely inside `Connector.reconcile` via an inline
   > `newSet := make(map[string]struct{}, len(newAddrs))` construction ‚Äî no helper
   > function. This is the final production shape.

3. The two comparators may lawfully differ. The spec does not require the emission
   observable and the connection observable to have the same trigger semantics. The
   emission observable is a log-level diagnostic; the connection observable is the
   load-bearing behavioral contract (BC-2.09.001 PC-2).

**Cite:** BC-2.09.001 EC-002; `router_config.go:81` (current equalStringSlices shape).

## Q2 ‚Äî Dial-loop placement and ownership (FO-2)

**Ruling: the dial loop lives in a new `internal/upstreamdial` package as a
`Connector` type. `runRouter` constructs `Connector` at startup and passes the
initial address list. Reload passes a new address list via a channel. The testenv
`RouterHandle` gains construction-time ownership of the connector's control handle,
retiring the transitional `SetSighupCh`/1-arg `SendReloadSignal` post-hoc pattern.**

**Derivation:**

The SIGHUP placement note (Q2) put the reload channel as a `runRouter` parameter
because there was no dial loop consuming upstream addresses. Now that a dial loop
exists, two options apply:

- **(a) Inline in `runRouter`:** a `for addr := range upstreamRouters` block
  plus goroutines inside `runRouter`. This is where `drainCoord` lives today.
  The problem is that `runRouter` is already 195 lines with four lifecycle phases;
  inline dial goroutines would make it untestable independently.
- **(b) New `Connector` type in a dedicated package:** `runRouter` owns the
  lifecycle (start/stop) but the dial logic, reconnect loop, and address-set
  reconciliation are encapsulated. This matches the precedent of `internal/drain`,
  which extracted drain coordination out of `runRouter` for the same reason.

Option (b) is the ruling. See Q4 for package placement details.

**Construction-time wiring (FO-2 obligation):**

The transitional `SetSighupCh` setter (`testenv.go:302`) and 1-arg
`SendReloadSignal` (`testenv.go:326`) are explicitly deferred artifacts from
S-7.04-FU-SIGHUP-RELOAD. Their doc comments state:

> "Construction-time wiring and real state tracking land with the PE-CONNECTOR-era
> testenv integration (S-7.04-FU-PE-CONNECTOR)."

This story fulfills that obligation as follows:

1. **`internal/upstreamdial.Connector`** exposes a `ConnectorHandle` (or equivalent)
   at construction time that carries: a `ReloadAddrs(addrs []string)` method (the
   address update path, driven by `runRouter`'s SIGHUP case instead of re-emitting
   directly), and a `Mode() RouterMode` accessor reflecting the current connection
   state (returns `ModeE` when no upstreams are connected, `ModePE` when ‚â•1 upstream
   is connected).
2. **`testenv.RouterHandle`** is extended at construction: `StartRouter` passes the
   connector handle into the handle so that `RouterHandle.Mode()` delegates to the
   live connector state rather than the stub `r.mode` field. The `SetSighupCh` and
   `SendReloadSignal` seam MUST be retired in this story (remove from `testenv.go`
   or replace with the construction-time equivalent). If retirement creates a build
   break in existing tests (e.g., `router_sighup_test.go`), those tests must be
   migrated to the new seam ‚Äî they are in scope for this story.
3. **`RouterHandle.Mode()` becomes a real accessor**: it reads from the live
   connector's `Mode()` observable rather than the stub enum. The harness skeleton
   in VP-038's proof method calls `eRouter.Mode()` ‚Äî this is the accessor that must
   now return a value reflecting actual connection state, not a stub.

**Seam interface (proposed):**

```go
// In internal/upstreamdial:
type Handle interface {
    // ReloadAddrs reconciles the running set to the new address list.
    // Set-equal semantics (see Q1). May be called from runRouter's select loop.
    ReloadAddrs(addrs []string)
    // Mode returns ModeE when no upstreams are connected, ModePE when >= 1.
    Mode() testenv.RouterMode  // or a local RouterMode type mirrored from testenv
    // Stop cancels all dial goroutines and waits for them to exit.
    Stop()
}
```

Note on `testenv.RouterMode`: since `testenv` is at DAG position 22 (test-helper
composition root ‚Äî not importable by production code), `internal/upstreamdial` MUST
define its own `Mode` type (e.g., `upstreamdial.ConnMode`) with `ModeE`/`ModePE`
values, and `testenv` maps from `upstreamdial.ConnMode` to `testenv.RouterMode` in
the wiring glue. Alternatively, the mode enum can live in `internal/config` or
`cmd/switchboard/main` as it has no dependencies.

**Cite:** `testenv.go:296-339` (SetSighupCh + SendReloadSignal transitional seam
comments), S-7.04-FU-SIGHUP-RELOAD DELIVERY ¬ßForward Obligations FO-2.

## Q3 ‚Äî `upstreamRouters` concurrency design (FO-4, adv P8-O1)

**Ruling: channel-passed snapshot is the correct shape. `upstreamRouters` in the
`runRouter` goroutine remains a local variable. The `Connector` receives address
updates via a buffered `chan []string`, not a mutex-guarded shared slice. The
`Connector` owns all reconnect state internally under its own mutex, following
go.md rule 12.**

**Derivation:**

The adversarial finding P8-O1 from S-7.04-FU-SIGHUP-RELOAD identified the single-
goroutine assumption on `upstreamRouters`. The choice is:

- **(a) Mutex-guarded shared slice:** `runRouter` writes `upstreamRouters` and the
  dial goroutines read it. go.md rule 12 says "never return internal pointers from
  a locked accessor." If we share the slice behind a mutex, any range loop in the
  dial goroutine holds a reference into the shared backing array ‚Äî a violation of
  the rule-12 snapshot contract.
- **(b) Channel-passed snapshot:** `runRouter`'s SIGHUP case, after computing the
  new address list, sends a `[]string` snapshot on a `chan []string` owned by the
  `Connector`. The `Connector`'s reconcile goroutine reads from this channel. The
  channel is buffered (capacity 1, dropping older value if unread ‚Äî i.e., use
  a non-blocking send or `select { case ch <- v; default: _ = <-ch; ch <- v }`).
  `runRouter` never reads `upstreamRouters` from the connector; it maintains its
  own local copy for emission purposes (BC-2.09.001 PC-1 mode line).

Option (b) is the ruling. It is consistent with the `sighupCh <-chan os.Signal`
precedent established in S-7.04-FU-SIGHUP-RELOAD: signals flow in one direction
through channels, state is not shared.

**Reconnect state ownership:**

The `Connector` owns: the `map[string]net.Conn` (or equivalent per-address dial
state), the per-address reconnect backoff timer, and the connected-count atomic that
drives `Mode()`. All of these are accessed only from the `Connector`'s single
reconcile/reconnect goroutine(s). The only cross-goroutine boundary is the address
update channel (from `runRouter` ‚Üí `Connector`) and the `Mode()` read (from
`runRouter`'s startup log, from testenv's `RouterHandle.Mode()` reader).

`Mode()` is safe as an `atomic.Int32` (0=E, 1=PE) loaded with `atomic.Load` from
any goroutine ‚Äî no mutex needed for a single integer observable. go.md rule 12
applies to slice/map returns, not atomic scalar reads.

**Cite:** `mgmt_wire.go:483` (`upstreamRouters` local var), `router_config.go:81`
(`equalStringSlices`), go.md rule 12, S-7.04-FU-SIGHUP-RELOAD DELIVERY FO-4.

## Q5 ‚Äî Reconnect/backoff policy (BC-2.09.001 EC-001)

**Ruling: exponential backoff with schedule-base=`keepaliveInterval` (floor=500ms),
cap=30s, per-attempt jitter ¬±25%; the log contract is verbatim; partial-PE
semantics is "router is in PE mode once ‚â•1 upstream is connected"; no additional
mode line is emitted for partial connectivity.**

> **F-P2-002 ruling block (v1.3):** Adversary pass-2 finding F-P2-002 identified
> a contradiction between this question (Q5) and Q7. Q5 as originally authored
> stated "base=500ms" as the schedule base. Q7 authored keepalive-coupling
> ("multiplicative from this base" where "this base" = `keepaliveInterval`).
> The two were never reconciled; the shipped code (`connector.go:258, :326`)
> implements Q7 semantics. **Ruling: Q7 prevails.** Rationale: coupling the
> reconnect base to `keepaliveInterval` gives operators one tunable that governs
> both dead-connection detection speed and reconnect aggressiveness ‚Äî operationally
> coherent. A flat 500ms base decoupled from keepaliveInterval risks hammering a
> flapping upstream faster than the keepalive confirms the drop, which is the exact
> failure the cap is meant to prevent. `BackoffBase=500ms` is correctly classified
> as a floor constant (minimum value for `keepaliveInterval` inside `nextBackoff`),
> not the schedule base. Q5's original "base=500ms" language is corrected below to
> describe the floor. Q7's language is authoritative for the schedule base.

**Derivation from BC-2.09.001 EC-001:**

> "Router starts in partial PE mode; logs 'upstream router <addr> unreachable'.
> Retries in background. Existing sessions unaffected."

The BC specifies the log string and the retry-in-background guarantee. It does not
specify backoff parameters ‚Äî these are a quality-of-implementation choice.

**Parameters:**

| Parameter | Value | Rationale |
|---|---|---|
| Schedule base | `keepaliveInterval` (resolved from config; prod default 1s) | Couples reconnect cadence to the health-probe interval ‚Äî one operator knob governs both detection speed and reconnect aggressiveness |
| Floor (`BackoffBase`) | 500ms | Prevents the schedule base from falling below a useful minimum even if `keepaliveInterval` is configured very short; `BackoffBase` is a floor constant inside `nextBackoff`, NOT the operative first-retry or reset-on-success value |
| Cap | 30s | High enough to avoid thundering-herd on a sustained upstream failure |
| Jitter | ¬±25% uniform | Breaks synchronization when multiple PE routers lose the same upstream simultaneously; keeps cap effective |
| Reset on success | Yes ‚Äî backoff resets to `keepaliveInterval` (floored at `BackoffBase`) on first successful dial | Consistent with keepalive-coupled base semantics |

**Log contract (BC-2.09.001 EC-001, verbatim):**

```
upstream router <addr> unreachable
```

This line MUST be emitted to `w` (the `io.Writer` passed to `runRouter`) on every
failed dial attempt. The addr is the string from `upstreamRoutersFor`. No additional
error text is required by the BC, but the implementation MAY append `: <net-error>`
for operator diagnostics; that suffix is NOT tested by ACs.

**Partial-PE semantics:**

- `Mode()` returns `ModePE` when the connected count is ‚â•1. The `mode=PE` startup
  line is emitted by `runRouter` when `upstreamRoutersFor(cfg)` is non-empty (already
  shipped by S-7.04). The LIVE operational mode tracks connected count, not configured
  count.
- No additional "partial PE" mode line is emitted. The BC does not define a
  `mode=partial-PE` observable. Operators learn about unreachable upstreams from the
  `upstream router <addr> unreachable` log line per EC-001.
- If ALL upstreams are unreachable after graduation, `Mode()` returns `ModeE`
  (fallback per BC-2.09.001 EC-004: "falls back to E-router-equivalent behavior").
  A `mode=E (no upstream_routers configured)` log line MUST be emitted when the
  connected count drops to zero after having been ‚â•1 (i.e., the final upstream
  connection is lost). This is the EC-004 observable.

**Cite:** BC-2.09.001 EC-001, EC-004; `mgmt_wire.go:526-530` (current mode emission
shape for the startup seam).

## Q6 ‚Äî Session bootstrap over `outerassembler` (BC-2.09.001 PC-2/PC-3)

**Ruling: on successful TCP dial, the connector calls `outerassembler.Assemble`
with an `Envelope` carrying the upstream router's address as `DstAddr` and this
router's node-level identity as `SrcAddr`. "Connection established" for BC-2.09.001
PC-2/PC-3 is defined as: TCP `net.Dial` returns nil error AND the bootstrap wire
frame is sent without error.**

**Derivation from the outerassembler API (assemble.go):**

```go
// Assemble(env Envelope, cf halfchannel.ChannelFrame) ([]byte, error)
```

The `outerassembler.Assemble` function takes an `Envelope` (with SVTN ID, src/dst
address, and HMAC auth key) and a `halfchannel.ChannelFrame` (the framed payload)
and emits a self-delimiting wire-format byte sequence consumable by
`netingress.ReadFrame` + `routing.RouteFrame`.

**What the connect-half sends:**

On dial success the connector sends a control-plane bootstrap frame to the upstream
router to register itself as a connected downstream PE peer. The frame MUST use:

- `Envelope.SrcAddr` ‚Äî this router's node address (derived from its ephemeral Ed25519
  public key and SVTN ID per `frame.DeriveNodeAddress`)
- `Envelope.DstAddr` ‚Äî the upstream router's expected node address (from config ‚Äî
  operators configure this alongside `addr` in the `upstream_routers` entry, or it is
  derived at dial time from the server's self-declared identity in the TLS handshake;
  exact mechanism is the story-writer's elaboration decision)
- `Envelope.SVTNID` ‚Äî the SVTN this PE router is serving (the router's primary SVTN)
- `Envelope.FrameAuthKey` ‚Äî derived via `hmac.DeriveKey` using the shared session key
  agreed during the admission handshake with the upstream

The inner `halfchannel.ChannelFrame` carries a minimal `FrameTypeControl` payload
identifying this as a PE-CONNECT bootstrap frame type (new constant to be defined,
e.g., `frame.FrameTypePEConnect`).

**"Connection established" definition:**

BC-2.09.001 PC-2 ("router establishes connections to the configured upstream
routers") is satisfied when:

1. `net.Dial("tcp", addr)` returns `(conn, nil)`
2. `outerassembler.Assemble(env, cf)` returns `(wire, nil)`
3. `conn.Write(wire)` returns `(n, nil)` with `n == len(wire)`

All three must succeed. A TCP connection that is not yet authenticated is not a
"connection established" event for BC purposes. The connected-count atomic is
incremented only on step 3 success.

**Upstream admission protocol note:**

The full admission handshake (challenge-response per `internal/admission`) with the
upstream router is the long-term design but is DEFERRED to a follow-on story if it
is not feasible within the 8-point scope of this story. The story-writer should
assess feasibility at elaboration time. If deferred: the dial bootstrap sends an
unauthenticated PE-CONNECT frame and the story notes the deferral; if in-scope:
the full `admission.GenerateChallenge`/`admission.AdmitNode` flow is wired at dial
time. Either way, the three-step "connection established" definition above applies.

**Cite:** `internal/outerassembler/assemble.go` (Envelope, Assemble), BC-2.09.001
PC-2/PC-3, `frame.DeriveNodeAddress`, `internal/admission` challenge-response.

## Q7 ‚Äî Keepalive wiring (BC-2.09.003 PC-8)

**Ruling: the keepalive ticker lives inside `internal/upstreamdial.Connector`,
NOT in `runRouter`. The fence against conflation with `sweepDeadline` is the
normative test `TestKeepaliveIntervalNotSweepDeadline` (already passing at
`950285c`). The ticker drives per-connection health probing and reconnect
scheduling, not session eviction.**

**Derivation:**

`keepaliveIntervalFor(cfg)` is computed in `runRouter` at `mgmt_wire.go:482` and
emitted as an observability log line. The resolved value is currently unused at
runtime (the reconnect-side keeper is the unimplemented half of BC-2.09.003 PC-8).

This story closes that gap:

1. `runRouter` passes the resolved `keepaliveInterval time.Duration` into the
   `upstreamdial.Connector` at construction (alongside the initial address list and
   the update channel).
2. The `Connector` uses `keepaliveInterval` as the base cadence for:
   - Sending keepalive probe frames on established upstream connections (to detect
     silently dead TCP connections that do not produce a read error).
   - Scheduling the reconnect attempt after a dial failure. `keepaliveInterval`
     (floored at `BackoffBase=500ms`) is the **operative schedule base**: the first
     retry after a connection drop waits `keepaliveInterval` (or `BackoffBase`,
     whichever is larger); subsequent retries grow multiplicatively from this base
     per the exponential backoff policy in Q5. On reconnect success, the per-address
     backoff state resets to `keepaliveInterval` (floored at `BackoffBase`).
     `BackoffBase` is a floor constant only ‚Äî it is NOT a separate schedule base and
     does NOT replace `keepaliveInterval` as the reset-on-success value.
3. The ticker MUST NOT be wired into `sweepDeadline` (console eviction ‚Äî `access.go`,
   semantically unrelated). The normative fence test
   `TestKeepaliveIntervalNotSweepDeadline` (in `router_config_test.go`) is the
   enforcement point. The test asserts that `keepaliveIntervalFor` and
   `sweepDeadline` return distinct values for the same config and that changing
   `cfg.KeepaliveInterval` does not affect `sweepDeadline`. This test MUST
   remain green across this story's changes with no modification to its assertions.

**Why the ticker lives in the Connector and not `runRouter`:**

The keepalive ticker is per-connection state. `runRouter` manages daemon lifecycle;
the Connector manages upstream connection lifecycle. The ticker belongs with the
state it governs.

**Cite:** `router_config.go:57` (`keepaliveIntervalFor`), `mgmt_wire.go:522-524`
(observability seam), BC-2.09.003 PC-8 normative note ("MUST NOT be routed into
sweepDeadline"), `TestKeepaliveIntervalNotSweepDeadline` (existing normative fence).

## Q9 ‚Äî EC-003/E-CFG-001 rendering nuance (FO-3)

**Ruling: no spec-side narrowing is needed for EC-003 in the reload+dial era. The
existing rendering (E-CFG-003 detail text surfaced under E-CFG-001 format from
`Validate()`) is correct and complete. The cross-BC note is a documentation artifact
only, not a gap.**

**Derivation:**

The FO-3 deferral from S-7.04-FU-SIGHUP-RELOAD states:

> "BC-2.09.001 EC-003 (invalid upstream addr on reload) is satisfied structurally
> via `Validate` but renders under E-CFG-001 with E-CFG-003 detail text."

Unpacking this: BC-2.09.001 EC-003 says "Config reload rejected; E-CFG-003; router
continues as E." BC-2.09.003 defines E-CFG-003 as a specific message template for
invalid upstream address format. When the SIGHUP reload path calls `loaded.Validate()`
and the upstream address is malformed, `Validate()` returns an error in E-CFG-003
format (the message template from BC-2.09.003 PC-6). The SIGHUP reload case wraps
it in the `"config reload failed: <err>; continuing with previous config"` envelope
from BC-2.09.003 EC-004. The resulting operator-visible log line is:

```
config reload failed: config error: upstream_routers[0].addr: 'notvalid' is not a valid host:port. Fix: ...; continuing with previous config
```

This satisfies both BC-2.09.001 EC-003 ("Config reload rejected; E-CFG-003") and
BC-2.09.003 EC-004 (fail-closed reload log format). There is no ambiguity.

The question raised in FO-3 ‚Äî "whether a BC amendment is warranted" ‚Äî is resolved
as NO. The cross-BC note is a documentation artifact that names the BC interaction;
it does not imply a gap. No BC amendment, no spec change.

For the PE-CONNECTOR dial era: if a reload introduces an unreachable (but validly-
formatted) upstream address, that is not a validation error ‚Äî `Validate()` accepts it,
the dial attempt fails, and EC-001 applies ("upstream router <addr> unreachable").
The two error paths (validation-time: E-CFG-003 via Validate; runtime: EC-001 via
dial failure) are disjoint and unambiguous.

**Cite:** BC-2.09.001 EC-003; BC-2.09.003 PC-6, EC-004; `mgmt_wire.go:548-558`
(SIGHUP reload fail-closed sequence).

## Scope Boundary vs S-7.04-FU-DRAIN-WIRE

S-7.04-FU-PE-CONNECTOR and S-7.04-FU-DRAIN-WIRE have a strict dependency boundary:

| This story (PE-CONNECTOR) | S-7.04-FU-DRAIN-WIRE |
|---|---|
| Establishes outbound TCP connections to upstream routers | Broadcasts DRAIN signal to connected nodes via SVTN channel |
| `Connector.Mode()` reflects live upstream connection state | `drainCoord` observers registered to upstream connections |
| Keeps upstreams alive (keepalive ticker per Q7) | Sends DRAIN frames to nodes migrating over live upstream paths |
| Provides the live upstream connections that make drain migration meaningful | Consumes the live upstream connections provided by this story |

**No DRAIN broadcast in this story.** The `drainCoord` from `mgmt_wire.go:492`
gains its observers from S-7.04-FU-DRAIN-WIRE, not here. The existing
`drainCoordHook` test seam in `router_config.go:76` remains unchanged. This
story does not register observers on `drainCoord`.

## Scope Boundary vs S-BL.RESYNC-FRAME

S-BL.RESYNC-FRAME owns the control-plane RESYNC frame protocol ‚Äî the sequence a
node sends to a new upstream router after migration to re-establish session state.
This story's dial loop sends the initial PE-CONNECT bootstrap frame (Q6) and
maintains the TCP connection. It does not implement the RESYNC exchange. The
Connector delivers the raw TCP `net.Conn` as an upstream channel; RESYNC is a
higher-level session protocol that uses that channel.
