# Architect Design Note: SIGHUP Signal-Loop Placement
## Story: S-7.04-FU-SIGHUP-RELOAD

This note answers the six design questions required to unblock scheduling.
All file:line anchors refer to the develop branch at HEAD `62e38d3`.

---

## Q1 — Signal.Notify placement: same channel or dedicated `sighupCh`?

**Decision: dedicated `sighupCh`, independent of the SIGTERM/SIGINT channel.**

**Reasoning from the code:**

`cmd/switchboard/main.go:118` constructs the termination context via
`signal.NotifyContext(context.Background(), syscall.SIGTERM, syscall.SIGINT)`.
`signal.NotifyContext` owns the notification channel internally and cancels
the returned `ctx` on any received signal. Adding SIGHUP to that set would
cancel `ctx` on a reload request — triggering the graceful-drain shutdown
path at `mgmt_wire.go:531` (`<-ctx.Done()`). That is exactly the wrong
behavior for a non-terminating signal.

SIGHUP must therefore be registered on a separate channel, added alongside
(not replacing) the existing `NotifyContext` setup:

```go
// Proposed addition in the "router" case body, after signal.NotifyContext:
sighupCh := make(chan os.Signal, 1)
signal.Notify(sighupCh, syscall.SIGHUP)
defer signal.Stop(sighupCh)
```

`sighupCh` is buffered (capacity 1) per `signal.Notify` best practice —
a missed SIGHUP while the reload is in progress is silently absorbed; the
operator can send another.

The existing signal-goroutine structure (`signal.NotifyContext` at
`main.go:118`) is not modified. The two channels are orthogonal: `ctx` is
cancelled by termination signals; `sighupCh` receives reload events. No
race between them.

## Q2 — Reload-dispatch path into `runRouter`: channel, callback, or atomic config-holder?

**Decision: pass `sighupCh <-chan os.Signal` as an additional parameter to `runRouter`; run loop selects on it.**

**Reasoning from the code:**

`runRouter` at `mgmt_wire.go:400` already has the signature
`func runRouter(ctx context.Context, w io.Writer, cfg *config.Config) error`.
The run loop at `mgmt_wire.go:531` is `<-ctx.Done()` — a blocking select
with a single case today. Extending it to a two-case select is the minimal,
lowest-distortion change:

```go
// Extended run loop (replace the current <-ctx.Done() block):
for {
    select {
    case <-ctx.Done():
        // existing graceful-drain shutdown path — unchanged
        ...
        return nil
    case <-sighupCh:
        // reload path — Q3 below
    }
}
```

Alternatives ruled out:

- **Atomic config-holder** (`atomic.Pointer[config.Config]`): adds a shared
  mutable object visible to multiple goroutines; the goroutines that consume
  config values (keepalive ticker, drain coordinator) are not yet continuous
  background loops — they read config once at startup and the reconnect ticker
  is not implemented yet. An atomic holder buys complexity with no present
  consumer. The `upstreamRoutersFor` fresh-slice contract
  (`router_config.go:88`) was already designed to isolate the startup call
  site from a future reload path; that isolation does not require an atomic
  holder.

- **Callback / function pointer**: adds indirection with no testing benefit
  that the `sighupCh` channel approach doesn't also provide.

- **Internal channel passed through a config-holder**: same distortion as
  the atomic approach, with an extra layer.

The `sighupCh` parameter approach mirrors the pattern used by
`drainCoordHook` (`router_config.go:76`) for test injection but at the
interface level: production code passes the real OS channel; tests pass a
test-controlled channel. The function signature extension is minimal:

```go
func runRouter(ctx context.Context, w io.Writer, cfg *config.Config,
               sighupCh <-chan os.Signal) error
```

`main.go` creates and passes the real `sighupCh`; integration tests supply a
`make(chan os.Signal, 1)` and send to it directly — no `syscall.Kill` or
process-level signal required.

## Q3 — Fail-closed atomicity (BC-2.09.003 EC-004): where does re-parse + Validate happen?

**Sequence (fail-closed, no partial application):**

```
SIGHUP received
    │
    ▼
config.LoadFile(configPath) ──── error? ──► log E-CFG-001 format
    │                                       "config reload failed: <err>; continuing with previous config"
    │ ok                                    return (stay in run loop with old cfg)
    ▼
loaded.Validate() ────────────── error? ──► log E-CFG-001 format (same message)
    │                                       return (stay in run loop with old cfg)
    │ ok
    ▼
diff: upstreamRoutersFor(loaded) vs upstreamRouters (local copy)
    │
    │ changed?
    ▼
re-emit mode startup line to w  ◄─── this is the ONLY state mutation
    │
    ▼
update local `upstreamRouters` variable (cfg pointer NOT mutated)
    │
    ▼
continue run loop
```

**Key atomicity properties:**

1. The re-parse and validate run entirely against a fresh `loaded *config.Config`
   struct. The existing `cfg` pointer (`runRouter` parameter, `mgmt_wire.go:400`)
   is never mutated during reload. If `LoadFile` or `Validate` returns an error,
   `cfg` is untouched — there is no window of partial application.

2. The "new config is applied" moment is a single `fmt.Fprintf` to `w` plus
   an assignment to the `upstreamRouters` local variable. Both are in the same
   goroutine as the `runRouter` run loop; no concurrent mutation.

3. E-CFG-001 is the right error code for a reload failure per BC-2.09.003 EC-004:
   `"config error: <field>: <problem>. Fix: <suggestion>"` — or for a file-read
   error, the wrapped error text naturally satisfies the "actionable error
   message" requirement from BC-2.09.003 PC-1. The log message format must be:
   `"config reload failed: <err>; continuing with previous config"` (verbatim
   from BC-2.09.003 EC-004 description).

4. `configPath` must be captured before entering the run loop — it comes from
   the `--config` flag parsed in the `router` case body (`main.go:98`). Pass it
   into `runRouter` as a `string` parameter, or capture it in a closure. Passing
   as a parameter is cleaner and consistent with the `sighupCh` extension.

   Updated signature:
   ```go
   func runRouter(ctx context.Context, w io.Writer, cfg *config.Config,
                  configPath string, sighupCh <-chan os.Signal) error
   ```

## Q4 — New internal package required?

**No new `internal/` package.**

The entire reload path lands in `cmd/switchboard` (the existing effectful
top-leaf at ARCH-08 §6.5 position 18). Specifically:

- Signal registration and `sighupCh` channel: `cmd/switchboard/main.go`
  (in the `"router"` case body).
- Reload dispatch, diff, re-emit: `cmd/switchboard/mgmt_wire.go` (inside
  `runRouter`'s extended select loop).
- Re-parse and validate: `internal/config` read-only — `config.LoadFile` and
  `(*Config).Validate` already exist at `internal/config/` (position 1 in
  ARCH-08 §6.5). No new functions needed there.
- `upstreamRoutersFor` already exists in `cmd/switchboard/router_config.go`.

**ARCH-08 registration obligation: NONE.** No new package, no registration
required. The story committer must NOT add a §6.4 declaration.

## Q5 — Interaction with drain coordinator and active sessions (BC-2.09.001 PC-4)

**The reload path must NOT touch any of the following:**

| Construct | Location | Why |
|-----------|----------|-----|
| `ingressCtx` / `ingressCancel` | `mgmt_wire.go:459` | Cancelling this context tears down the data-plane accept loop and closes all accepted connections. A config reload is not a shutdown event. |
| `dataWG` | `mgmt_wire.go:464` | Waiting on `dataWG` blocks until all in-flight ingress goroutines exit. Doing this during reload would stall active sessions. |
| `ctx` (parent context) | `runRouter` parameter | Cancelling this cancels the mgmt server, ingress accept loop, and initiates drain. |
| `drainCoord` | `mgmt_wire.go:489` | Signalling the drain coordinator notifies registered observers (node migration). Reload must not trigger a drain. |
| `mgmtWG` / `mgmtSrv` | `mgmt_wire.go:450-451` | The management server keeps serving during a valid reload. |

**What the reload path DOES touch (and why it is safe):**

- `upstreamRouters` (local `[]string` variable, `mgmt_wire.go:480`): reassigned
  to a new slice returned by `upstreamRoutersFor(loaded)`. No goroutine reads
  this variable after startup except the reload path itself — the variable is
  a closed-over value in the run loop goroutine, not shared state.
- `w` (the log writer): `fmt.Fprintf(w, ...)` is called. `w` is `os.Stderr` in
  production and is safe for concurrent use (Stderr is fd 2; concurrent writes
  from the logger and the reload emission may interleave, but that is acceptable
  for a diagnostic log stream).

**The reload is session-transparent:** it re-emits the mode line to `w` and
updates the local `upstreamRouters` variable. Neither the TCP listener, the
in-flight accepted connections, nor the mgmt server are disturbed.

## Q6 — Testability seam: how do tests drive the reload path?

**Mechanism: `sighupCh <-chan os.Signal` parameter (from Q2).**

Tests pass a `make(chan os.Signal, 1)` to `runRouter` and send `syscall.SIGHUP`
(or any `os.Signal` value) to it directly, without issuing a real OS signal to
the process. This is the same in-process injection pattern used by
`TestRunRouter_ForcedExitPastDrainTimeout` via `drainCoordHook`
(`router_config.go:76`) and by `TestRunRouter_SIGTERMLifecycle` via context
cancellation.

**Test conventions that apply:**

- The existing integration tests (`cmd/switchboard/router_drain_test.go`) run
  `runRouter` in a goroutine with a context derived from `t.Context()` (or a
  manually-created context with `cancel`). The same pattern applies here.
- The `internal/testenv` rig (`internal/testenv/testenv.go`) already provides
  `RouterHandle.Restart(t, cfg RouterConfig)` at `testenv.go:269` — a method
  that reconfigures the in-process router handle's mode. The VP-038 integration
  test (AC-004) can use this rig's `peRouterAddr` field and the new `sighupCh`
  channel to assert that after a SIGHUP with a PE config, `env.RouterHandle.Mode()`
  equals `testenv.ModePE` without process restart. This requires `testenv`'s
  `RouterHandle` to hold a reference to the `sighupCh` it was started with, which
  is a small addition inside the test package only (not a production surface change).
- For unit tests that do not use `testenv`, the pattern is:

  ```go
  sighupCh := make(chan os.Signal, 1)
  var wg sync.WaitGroup
  wg.Add(1)
  go func() {
      defer wg.Done()
      _ = runRouter(ctx, &buf, cfg, configPath, sighupCh)
  }()
  // ... wait for startup (scan buf for "mode=E") ...
  sighupCh <- syscall.SIGHUP
  // ... assert buf now contains "mode=PE" ...
  cancel()
  wg.Wait()
  ```

- The `configPath` parameter (from Q3) allows tests to write a modified YAML
  file to a `t.TempDir()` path before sending SIGHUP, matching the
  `TestRunRouter_EmitsUpstreamModePE` convention of supplying config structs
  via the `cfg` parameter. For reload tests, the config is written to disk
  and the configPath points there.

## Summary of Function Signature Change

Current (`mgmt_wire.go:400`):
```go
func runRouter(ctx context.Context, w io.Writer, cfg *config.Config) error
```

After this story:
```go
func runRouter(ctx context.Context, w io.Writer, cfg *config.Config,
               configPath string, sighupCh <-chan os.Signal) error
```

Call site (`main.go:120`) updates accordingly:
```go
return runRouter(ctx, os.Stderr, cfg, *configPath, sighupCh)
```

All existing `runRouter` callers in tests must add the two new parameters.
`sighupCh` in tests is a `make(chan os.Signal, 1)` never written (for tests
that do not exercise reload) or written with `syscall.SIGHUP` (for reload
tests). The `configPath` for non-reload tests may be an empty string — the
reload path only reads the path when SIGHUP is received, so an empty path
during a non-reload test does not trigger any I/O.

## Non-Goals (Explicit)

This story does NOT implement:

- **DRAIN broadcast on reload** — the drain coordinator is not signalled
  during config reload. Reload is mode-transition emission only.
  DRAIN-over-SVTN broadcast is `S-7.04-FU-DRAIN-WIRE`.
- **PE outbound connector** — adding `upstream_routers` to config and
  reloading emits `mode=PE` and updates the `upstreamRouters` local
  variable. It does NOT initiate outbound TCP dials. That is
  `S-7.04-FU-PE-CONNECTOR`.
- **`sbctl router reload` RPC** — BC-2.09.001 Trigger names both SIGHUP
  and `sbctl router reload`. This story covers only the SIGHUP path. The
  sbctl surface is a management-RPC concern deferred per the existing
  `DRIFT-HS006-DRAIN-CLI-MISSING` adjudication in S-7.04-DELIVERY.
- **keepalive reconnect ticker** — `keepaliveIntervalFor` is already applied
  at the startup emission seam by S-7.04. The reconnect ticker itself is
  not part of this story.
