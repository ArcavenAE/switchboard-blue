# S-6.06 Pass-4 Scope Rulings

Product-owner rulings on six contract-precedence questions raised by Pass-4
adversarial review of S-6.06. Each ruling closes one or more findings and
propagates edits to BC-2.05.004, VP-075, S-6.06, interface-definitions.md, and
error-taxonomy.md.

---

## F-L2-001 (HIGH) — RPC Command Namespace: `admin.key.list-keys`

**Finding:** AC-001/AC-006 of S-6.06 use `admin.list-keys`; VP-075's property
statement and proof harness use `admin.key.list-keys`. These are inconsistent.

**Ruling:** The canonical wire RPC name is **`admin.key.list-keys`** — consistent
with the sibling commands `admin.key.register`, `admin.key.revoke`, and
`admin.key.expire` (all use the `admin.key.*` namespace).

`admin.list-keys` in AC-001 and AC-006 of S-6.06 is a stale form; it must be
replaced with `admin.key.list-keys` everywhere.

**Propagates to:** S-6.06 (AC-001, AC-006 body text, Error Code Map command
references); VP-075 (already correct — no change needed); BC-2.05.004 (Trigger,
Canonical Test Vectors — existing references to `sbctl admin list-keys` remain
valid as CLI surface; the wire RPC name is clarified in a note).

**Rationale:** Consistent RPC namespacing makes it unambiguous which subsystem
owns each handler and prevents `admin.*`-namespace collisions when future
non-key admin commands are added (e.g., `admin.svtn.create`).

## F-L2-002 (HIGH) — Handler Gate vs. SVTNManager Gate Precedence

**Finding:** When a console-role caller attempts to revoke a control key via
`admin.key.revoke`, two error codes could apply: E-ADM-009 (handler gate, fires
before dispatch to SVTNManager) and E-ADM-011 (SVTNManager.RevokeKey
revocation-hierarchy violation, reachable only after dispatch). Which fires?

**Ruling:** **E-ADM-009 fires first and is the only code a console caller ever
sees for any `admin.key.*` operation.**

Per BC-2.05.004 PC-1 / DI-001 and ARCH-04 Inv-3, the handler-layer authority
gate runs before any SVTNManager method is invoked. A console-role caller is
rejected at the handler gate (E-ADM-009) before `SVTNManager.RevokeKey` is ever
called. E-ADM-011 (revocation hierarchy violation) is therefore unreachable via
the mgmt RPC path when the handler gate is in place.

**E-ADM-011 scope clarification:** E-ADM-011 is the code returned by
`SVTNManager.RevokeKey` itself when a direct Go API caller (e.g., a unit test)
invokes `RevokeKey` with a lower-tier role for a higher-tier key — bypassing
any RPC handler. It is NOT reachable via `admin.key.revoke` in normal
operation.

**Propagates to:**
- BC-2.05.004: Canonical Test Vectors — update the "Console or readonly key
  attempts to revoke a control key" row to show E-ADM-009 (not E-ADM-011), and
  add a note that E-ADM-011 is SVTNManager-internal (reachable only via direct
  Go API).
- error-taxonomy.md: Add a disambiguation note to E-ADM-011 clarifying its scope
  is the SVTNManager Go API layer only, not the mgmt RPC path.
- S-6.06: The existing AC-006 correctly targets E-ADM-009 — no change needed
  there.

## F-L2-003 (HIGH) — `admin.key.list-keys` Authority Scope

**Finding:** S-6.06 AC-006 gates `admin.list-keys` on control-role authority
(E-ADM-009 for non-control callers). interface-definitions.md line 108 says
"Any admitted role" for `sbctl admin list-keys`. VP-075 property statement
gates all `admin.key.*` commands on control-role. These are inconsistent.

**Ruling:** **`admin.key.list-keys` is a read-only operation; any admitted role
may call it.**

The interface-definitions.md definition ("Any admitted role") is correct and is
the canonical authority for this command.

AC-006 of S-6.06 incorrectly includes `admin.list-keys` (now
`admin.key.list-keys`) in the authority gate. It must be removed from the gate;
a console-role or access-role key calling `admin.key.list-keys` MUST receive
`ok: true` (subject to the SVTN argument being valid), NOT E-ADM-009.

VP-075's property statement scope must be narrowed to `admin.key.register`,
`admin.key.revoke`, and `admin.key.expire` only. `admin.key.list-keys` is
excluded from the control-only gate.

**Propagates to:**
- S-6.06 AC-006: Remove `admin.key.list-keys` from the list of gated commands;
  remove the `TestE2E_AdminListKeys_RoleInsufficient` test requirement.
- VP-075: Narrow property statement to exclude `admin.key.list-keys`.
- BC-2.05.004 PC-1: Clarify that list-keys is readable by any admitted role;
  note that the "management authority" precondition applies to register/revoke/
  expire only.

## F-L2-007 (MED) — `expire` Wire Argument Field Name

**Finding:** S-6.06 AC-005 uses `after` as the JSON wire field name for the
duration argument. BC-2.05.004 PC-3 Canonical Test Vector uses `--at <timestamp>`
(CLI flag). The relationship between the CLI flag and the wire field is undefined.

**Ruling:**
- **Wire field name:** `after` (Go duration string, e.g., `"720h"`, `"30m"`).
  This is what the daemon handler parses from the incoming JSON args.
- **CLI flag:** `--at <RFC3339-timestamp>`. The `sbctl` CLI accepts a
  human-readable absolute timestamp, computes `duration = timestamp - time.Now()`
  client-side, and sends the computed duration as the `after` wire field.
- **Server-side validation:** The handler validates `after` as a Go duration
  string (rejects zero, negative, and >100-year durations per AC-005).

This is a CLI-to-wire translation pattern, consistent with how human-friendly CLI
flags are translated into wire formats throughout the product.

**Propagates to:**
- BC-2.05.004: Add a note to PC-3 (expire postcondition) clarifying that the
  wire field is `after` (Go duration string) and the CLI `--at <RFC3339>` flag
  is translated to duration before sending. Canonical Test Vector row updated
  to note the translation.
- interface-definitions.md: Add a note to the `admin key expire` row clarifying
  that `--at <RFC3339-timestamp>` is translated to the `after` wire duration
  field by the CLI.
- S-6.06: AC-005 already uses `after` — no change needed to the test data.

## F-P4L1-001 (HIGH) — Operator-Key Bootstrap Chicken-and-Egg

**Finding:** A newly deployed control daemon has operator keys (authenticated
via `mgmt.OperatorKeySet`) but no keys registered in any SVTN. The handler
authority gate requires a `RoleControl` key registered in the target SVTN. Until
the first key is registered, no key can pass the gate — a chicken-and-egg
deadlock.

**Ruling:** **Operator-set membership grants RoleControl authority for
`admin.key.register` only, enabling first-key bootstrap into a fresh SVTN.**

Specifically:
- For `admin.key.register`: if the calling key is a member of
  `mgmt.OperatorKeySet` AND no keys are yet registered in the target SVTN (or
  the key itself is not yet in the SVTN admitted set), the handler MAY proceed as
  if the caller has RoleControl authority. This is the bootstrap path only.
- For `admin.key.revoke`, `admin.key.expire`: the caller MUST be registered as
  RoleControl in the target SVTN's admitted set. Operator-set membership alone is
  insufficient for these destructive operations.
- For `admin.key.list-keys`: any admitted role (or operator-set member, per
  F-L2-003 ruling) may call.

The bootstrap authority grant is limited to the first-register operation. Once
at least one control key is registered in the SVTN, subsequent `admin.key.register`
calls require the caller to be registered as RoleControl in the admitted set.

**Implementation note:** The handler layer SHOULD check:
1. Is the caller's key in the SVTN admitted set with RoleControl? If yes, proceed.
2. Else, is the caller's key in `mgmt.OperatorKeySet` AND the operation is
   `admin.key.register`? If yes, proceed (bootstrap grant).
3. Otherwise, deny with E-ADM-009.

**Propagates to:**
- BC-2.05.004 PC-1: Add a clarifying note documenting the bootstrap exception —
  "operator-set membership grants register authority when no control key exists
  in the target SVTN (bootstrap path)."
- BC-2.05.004 Edge Cases: Add EC-005 ("operator-key first-register-into-empty-SVTN
  bootstrap path").
- S-6.06: Add EC to the Edge Cases table for "operator-key first-register-into-svtn"
  and note the handler logic in the Architecture Compliance Rules.

## F-P4L1-003 (MED) — Revoked/Expired Key Residual Authority

**Finding:** A key may have been admitted during the mgmt handshake (operator
key authentication via `mgmt.OperatorKeySet`) but subsequently revoked or expired
in the SVTN admitted set. At handler authority resolution time, if the key's
`revoked=true` or `expiry` has passed, should the handler grant or deny?

**Ruling:** **Deny with E-ADM-009, same as an unregistered key.**

At handler authority resolution time, the handler checks the key's current state
in the admitted set:
- If `revoked=true`: deny with E-ADM-009 (treat as if not registered).
- If `now >= expiry` (key expired): deny with E-ADM-009 (treat as if not
  registered).
- If key is not found in the admitted set: deny with E-ADM-009.

This is fail-closed behavior: only keys that are currently registered, not
revoked, and not expired may exercise admin authority.

**Rationale:** The mgmt connection authentication (Ed25519 handshake) establishes
that the caller owns the private key. The SVTN admission state determines whether
that key is currently authorized for operations. Revocation and expiry should
take effect immediately at the handler layer — not deferred to SVTNManager.

**Propagates to:**
- BC-2.05.004 PC-1: Add a note that "management authority" requires the key to
  be currently registered, not revoked, and not expired. A revoked or expired key
  is treated as having no authority (E-ADM-009), even if it successfully
  authenticated at the mgmt connection layer.
- BC-2.05.004 Edge Cases: Add EC-006 ("revoked key attempts admin operation after
  mgmt authentication → E-ADM-009").
- S-6.06: Add EC to Edge Cases for this scenario.
