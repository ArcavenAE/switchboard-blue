## Q1 ג€” Frame type for the DRAIN message on the wire

**Ruling: use `frame.FrameTypeCtl = 0x03` (existing constant) to carry DRAIN
messages. No new frame-type constant is introduced. The DRAIN payload schema
is a minimal fixed-length control message (see below).**

**Derivation:**

The six canonical frame types at `e940fc2` are (verified `internal/frame/frame.go`:33ג€“39):

```
FrameTypeData      FrameType = 0x01
FrameTypeEmptyTick FrameType = 0x02
FrameTypeCtl       FrameType = 0x03
FrameTypeArq       FrameType = 0x04
FrameTypeFec       FrameType = 0x05
FrameTypePEConnect FrameType = 0x06
```

`FrameTypeCtl = 0x03` is the existing discriminator for authenticated control
frames at the router level. Adding a new `FrameTypeDrain = 0x07` would:
(a) require widening `Valid()` from `<= 0x06` to `<= 0x07` ג€” which is the very
guard S-BL.PE-RECEIVE-LOOP erected (blast-radius sweep of 10+ locations) to
prevent accidental over-widening; (b) require a same-commit amendment to ARCH-02
and BC-2.01.004 under the wire-format spec-pair rule (F-P8-008/F-SP14-001
precedent); (c) complicate the S-BL.RESYNC-FRAME consumer path noted in the
PE-RECEIVE-LOOP Non-Goals, which already relies on `FrameTypeCtl` for RESYNC
control frames.

Using `FrameTypeCtl` requires only a discriminator byte inside the payload to
distinguish DRAIN from RESYNC. This is the correct design: the outer-header
`FrameType` field identifies the wire family (`ctl`); the control-message opcode
(drain vs. resync vs. other) lives in the payload header.

**DRAIN payload schema (binding ג€” v1.0):**

The payload carried inside a `FrameTypeCtl` DRAIN frame is a fixed 4-byte
control message:

```
byte[0]   control_type   = 0x01   // DRAIN opcode (reserved: 0x02=RESYNC)
byte[1]   version        = 0x01   // protocol version, allows future extension
byte[2:4] reserved       = 0x0000 // zero-filled, must be ignored by receiver
```

Rationale for this schema:
- Minimum overhead: 4 bytes; outer-header `PayloadLen` = 4.
- `control_type = 0x01` reserves space for RESYNC (`0x02`) without ambiguity.
- Fixed length avoids any variable-length parsing complexity in an
  in-process-goroutine observer function.

**Assembly path:** The DRAIN frame is assembled from scratch (not via
`outerassembler.Assemble`) because the drain observer runs inside `cmd/switchboard`
at the SVTN send path boundary. The observer constructs the 4-byte payload, then
calls `frame.EncodeOuterHeader` + `append(ehdr[:], payload...)` ג€” identical to
the PE-RECEIVE-LOOP receive-side reconstruction pattern (F-SP3-001 byte-contract,
verified in `connector.go` at `e940fc2`). The `Envelope` fields (SVTN, SrcAddr,
DstAddr, FrameAuthKey) are sourced from the per-node connection context stored
in the runRouter connection registry (Q3 below for the registry design).

**Wire-format spec-pair obligation:** Because this story introduces a new
`control_type` discriminator byte in the `FrameTypeCtl` payload schema, the
following same-commit amendments are required:
1. ARCH-02 ֲ§ "Outer Header Format" (frame_type table; *corrected v1.2 ג€”
   F-DW-SP2-009: ARCH-02 has no ֲ§3.1 heading ג€” the frame_type table lives
   under the "Outer Header Format (44 bytes, fixed ג€” DI-007, BC-2.01.004)"
   H2 at ARCH-02:67; cite the heading, not a section number*): add a note
   that `ctl (0x03)` payloads carry a `control_type` byte discriminator;
   enumerate opcode `0x01 = DRAIN`.
2. BC-2.01.004 Postcondition 2 (outer-header layout): add the same control_type
   note under the `frame_type = ctl` row.

This is a NARROWER obligation than the FrameTypePEConnect case (F-SP14-001) ג€”
the frame_type byte itself is unchanged (0x03 already listed); only the payload
schema documentation is new. The same-commit rule (F-P8-008) still applies to
ARCH-02 + BC-2.01.004 as a pair.

## Q2 ג€” DRAIN broadcast origin and send path; FO-RECV-FWD-001 disposition

> **v1.1 SUPERSEDED (F-DW-SP1-001 / Q-SEAM and F-DW-SP1-002 / Q2-AMENDED below):**
> The "FO-RECV-FWD-001 MUST be discharged / interface-set-widening / real ForwardFunc"
> ruling is superseded by Q2-AMENDED. FO-RECV-FWD-001 is NOT consumed by this story;
> the nil `ForwardFunc` stays; `interfaceSet` stays `[]{peIfaceID}`. The "Send path
> seam options (a)/(b)" discussion and option (a) ruling are superseded by Q-SEAM ג€”
> the seam is an `OnAccept` hook on `netingress.Serve`, not a ForwardFunc wiring.
> The v1.0 body text below is preserved for historical record. Read Q-SEAM and
> Q2-AMENDED for the binding v1.1 rulings.

**Ruling: the DRAIN broadcast originates in `cmd/switchboard/runRouter` via a
`drain.Observer` closure registered with `drainCoord.RegisterObserver`. This
story IS the interface-set-widening event. FO-RECV-FWD-001 MUST be discharged
(the nil `ForwardFunc` in the `SetFrameCallback` closure at `mgmt_wire.go`:549
replaced with a live wiring) in this story.**

**Derivation:**

S-BL.PE-RECEIVE-LOOP emitted FO-RECV-FWD-001 (verified `S-BL.PE-RECEIVE-LOOP-DELIVERY.md`:270):

> `mgmt_wire.go` `ForwardFunc fn` is nil ג€” split-horizon always exhausts before
> `fn` is invoked in the single-interface set. When the router's interface set
> widens (second interface added), `ForwardFunc` MUST be wired.

This story adds `netingress` inbound connections as a second logical interface set
entry ג€” each admitted node arriving via `netingress.ServeConn` creates a new
`InterfaceID` in the routing layer. The DRAIN observer's closure also needs to
write frames BACK to admitted nodes over the `netingress` send path. The nil
`fn` at `mgmt_wire.go`:549 is therefore no longer correct once per-node send
paths exist.

**Interface-set widening definition for this story:**

The existing `peIfaceID = routing.InterfaceID(1)` (verified `mgmt_wire.go`:534
*(corrected v1.2 ג€” F-DW-SP2-009: was cited as :528)*)
is the PE upstream receive interface. This story registers per-node inbound
interfaces (one `InterfaceID` per admitted connection arriving from
`netingress.ServeConn`). The `SetFrameCallback` closure MUST be updated to pass
a real `ForwardFunc` that dispatches to the per-node send map.

**Send path seam options:**

Two shapes are viable:

**(a) Goroutine-per-node send channel:** Each admitted node's goroutine (in
`netingress.ServeConn` or a peer connection manager) owns a `chan []byte` send
channel. `runRouter` registers the channel in a `sync.Map[InterfaceID, chan []byte]`
(the "per-node send map"). The `ForwardFunc` closure reads from this map and
writes to the channel. The drain observer closure also reads from this map to
locate the node's send channel and write the DRAIN frame.

**(b) Write-callback per node:** Each node connection registers a
`func([]byte) error` write callback at connection-open time. The drain observer
stores a snapshot of registered write callbacks. `ForwardFunc` dispatches to the
callbacks directly.

**Ruling: option (a) ג€” per-node send channel map.** It is the canonical Go
concurrent-server pattern, matches the existing `netingress` per-connection
goroutine structure, and avoids callback retention (GC-friendlier when nodes
disconnect). The `sync.Map` is safe for concurrent access from the drain observer
goroutine and the routing dispatch goroutine simultaneously.

**Implementation location:** The per-node send map is owned by `runRouter` in
`cmd/switchboard/mgmt_wire.go`. No new internal package is required; the map is
a local `sync.Map` variable in `runRouter`, passed to both the `SetFrameCallback`
closure and the drain observer closure by reference.

**FO-RECV-FWD-001 discharge contract (binding ג€” v1.0):**

The `SetFrameCallback` closure at `mgmt_wire.go`:549 currently reads:

```go
connector.SetFrameCallback(func(hdr frame.OuterHeader, raw []byte) error {
    return arrivalHandler.OnFrameArrival(raw, peIfaceID, []routing.InterfaceID{peIfaceID}, nil)
})
```

After this story, the nil MUST be replaced with a `ForwardFunc` that dispatches

## Q-SEAM ג€” Send-path seam design (F-DW-SP1-001, BLOCKING ג€” added v1.1)

> **This section is new in v1.1 (F-DW-SP1-001, HIGH, BLOCKING).** The v1.0
> note asserted "per-node send channels + InterfaceID allocation + RegisterObserver
> at connection-open" against plumbing that does not exist. `netingress.ServeConn`
> is read-only: it accepts a `RouteFn` callback and reads frames from the conn;
> it never writes to conn, has no OnAccept hook, and never surfaces the conn to
> `runRouter`. `upstreamdial` dials OUTBOUND to upstream routers ג€” it is NOT the
> inbound node-connection path. `netingress.go` was missing from the FCL.
> This section designs the seam from first principles.

### The problem

`runRouter` starts `netingress.Serve` with a `route RouteFn` closure (verified
`mgmt_wire.go`:465ג€“484 at `e940fc2`). `Serve` accepts connections on `dataLn` and
spawns per-connection goroutines running `ServeConn`. `ServeConn` is a receive
loop ג€” it reads frames and dispatches them to `route`; it neither writes to
`conn` nor surfaces the `conn` to the caller. There is no existing hook for
`runRouter` to learn that a new inbound connection has been accepted, to allocate
an `InterfaceID` for it, or to attach a send-channel writer goroutine.

### Seam options evaluated

**Option (a) ג€” `netingress.Serve` gains a per-connection `OnAccept` hook:**

Add an `OnAccept func(conn net.Conn, ifaceID routing.InterfaceID) func()` field
to a new `ServeConfig` struct passed to `netingress.Serve`. `Serve` calls
`OnAccept(conn, allocID)` before launching the `ServeConn` goroutine; the returned
`func()` is called as a `defer` inside the per-connection goroutine (deregister
on close). `runRouter` supplies an `OnAccept` closure that ~~creates the
send channel,~~ *(corrected v1.3, F-DW-SP3-001 ג€” this narrative/evaluative
paragraph predates the ownership-split ruling; netingress creates the send
channel, not `OnAccept` ג€” see "Ownership split" and the API specification
below for the binding v1.3 shape. `OnAccept` receives it already built.)*
registers it in the per-node send map, ~~calls
`drainCoord.RegisterObserver(...)`,~~ *(struck v1.2, completing the
F-DW-SP2-002 sweep ג€” Q-SINGLE-OBS below rules this out: `OnAccept` never
calls `RegisterObserver`. Only a single startup observer is registered,
once, at `drainCoord` construction time *(corrected v1.3 ג€” F-DW-SP3-008:
not "before `runRouter` serves any connection"; see Q-SINGLE-OBS's own
v1.3 correction)*; `OnAccept` only stores the
send channel in the live map that observer iterates.)* and starts a writer
goroutine draining the
channel to `conn`.

Pros: `netingress` retains ownership of accept loop; write-path goroutine is
started by `runRouter` and closed by `runRouter` (clean lifecycle). ARCH-08
impact: `netingress` must import `routing.InterfaceID` ג€” that is an import of
`internal/routing` (pos 17) from `internal/netingress` (pos 18). Pos 18 > 17,
so the edge is legal (`netingress` already is downstream of routing in the
topological order). No forbidden edge.

**Option (b) ג€” `netingress.Serve` gains a per-connection `ConnFn` (write-half
passed back to caller):**

`netingress.Serve` calls a caller-supplied `ConnFn func(net.Conn) RouteFn`
before `ServeConn`; the caller can start a writer goroutine against the returned
`net.Conn` write-half and return a per-connection `RouteFn`. `runRouter` constructs
the per-node writer goroutine inside `ConnFn`. No routing import needed in
`netingress`. Con: `net.Conn` is bidirectional ג€” the caller calling `conn.Write`
from the writer goroutine while `ServeConn` calls `conn.Read` requires discipline
(see B13-style platform note below). This is safe on `net.TCPConn` (read and write
are independent paths; no mutex required), but the API leaks the full conn to the
caller.

**Option (c) ג€” `netingress` gains a write-seam internally; `runRouter` gets a
handle registry:**

`netingress.Serve` internally allocates a `chan []byte` per connection and starts
a writer goroutine draining the channel to `conn`. It exposes a `Write(ifaceID,
[]byte) error` method on a `ServeHandle` return value. `runRouter` calls
`handle.Write(id, frame)` in the drain observer. Complex: handle registry adds
indirection and a second internal state machine in `netingress`.

### Ruling (binding ג€” v1.1)

**Option (a) is ruled.** `netingress.Serve` gains an `OnAccept` hook via a
`ServeConfig` parameter (or a functional option). This is the minimal seam that:
- Keeps the per-connection writer goroutine owned by `runRouter` (clean drain
  lifecycle ג€” `runRouter` starts it, closes it when drain completes or conn closes).
- Avoids leaking the raw `net.Conn` write-half to `runRouter` (no B13 concern
  about concurrent read+write ownership confusion ג€” the goroutines are explicit).
- Requires only one new import in `netingress`: `routing.InterfaceID` (pos 17 < 18).
  This edge is LAWFUL under ARCH-08 ֲ§6.5 (forward-direction, no cycle).

### API specification (binding ג€” v1.3, ownership split per F-DW-SP3-001)

```go
// In internal/netingress:

// NodeHandle is the write handle for an accepted node connection, fully
// populated by netingress BEFORE OnAccept is called (netingress owns DATA
// creation ג€” see "Ownership split" below). OnAccept receives it as a
// finished value; it does not construct any of its fields.
type NodeHandle struct {
    IfaceID routing.InterfaceID
    Send    chan []byte    // netingress-created. runRouter stores this in
                             // the per-node send map (Q-SINGLE-OBS's drain
                             // observer sends into it) and the writer
                             // goroutine (below) receives from it. NEVER
                             // closed, by any goroutine, for the lifetime
                             // of the process ג€” enforced by convention
                             // (all call sites honor this), not by the
                             // type system; see the single-closer/
                             // no-send-after-close invariant below.
    Done    chan struct{}  // netingress-created. Closed EXACTLY ONCE, and
                             // always through the doneOnce sync.Once
                             // stored alongside it in nodeConn ג€” never
                             // call close(h.Done) directly. See "Shutdown
                             // ordering guarantee" below for the two
                             // trigger paths that race to close it.
}

// ServeConfig carries optional hooks and allocation parameters for Serve.
type ServeConfig struct {
    // OnAccept is called for each accepted connection, after netingress has
    // allocated the connection's InterfaceID and created its Send/Done
    // channels (NodeHandle is fully populated), and before ServeConn starts.
    // The returned func() (behavior cleanup) is called in a defer inside the
    // per-conn goroutine when ServeConn returns (conn closed or ctx
    // cancelled) ג€” see "Ownership split" below for what belongs in it.
    // If nil, no hook fires and netingress allocates nothing for that
    // connection (zero-cost when unused).
    OnAccept func(conn net.Conn, h NodeHandle) func()

    // IfaceIDSeed is the first InterfaceID netingress allocates for an
    // accepted connection when OnAccept is non-nil; subsequent connections
    // get IfaceIDSeed+1, IfaceIDSeed+2, ... via an internal atomic counter.
    // Callers reserve low IDs (e.g. peIfaceID = routing.InterfaceID(1)) by
    // setting IfaceIDSeed above them. Zero defaults to 2 (preserves the
    // single reserved PE-upstream ID and keeps a zero-value ServeConfig
    // backward-compatible).
    IfaceIDSeed routing.InterfaceID
}

// Serve(ctx, ln, route, logger) is extended to:
// func Serve(ctx context.Context, ln net.Listener, route RouteFn,
//            logger Logger, cfg ServeConfig) error
//
// Backward compat: callers passing a zero ServeConfig (OnAccept == nil) are
// unaffected ג€” Serve allocates no IfaceID/Send/Done and calls no hook.
```

**Ownership split (binding ג€” v1.3, F-DW-SP3-001, HIGH):** `NodeHandle` is
passed INTO `OnAccept` as a parameter ג€” by Go's own calling convention,
that means the CALLER of `OnAccept` (i.e. `netingress.Serve`) must
construct it before the call. The v1.1/v1.2 text described `ifaceID` as
"allocated by runRouter" and had the `OnAccept` closure itself "create
`send`" ג€” both describe `runRouter` populating a value it receives as an
already-built parameter, which cannot compile as written. This is a
data/behavior split, not a relocation of both halves to one side:

- **netingress owns DATA creation.** Inside `Serve`, when a connection is
  accepted and `cfg.OnAccept != nil`: allocate `ifaceID` from the atomic
  counter seeded by `cfg.IfaceIDSeed` (default 2); create
  `send := make(chan []byte, 32)` and `done := make(chan struct{})`;
  populate `h := NodeHandle{IfaceID: ifaceID, Send: send, Done: done}`;
  call `cfg.OnAccept(conn, h)`.
- **`runRouter`'s `OnAccept` closure owns BEHAVIOR.** It wraps `h.Send`/
  `h.Done` in a `*nodeConn` (see "Send-map value type" below), stores it
  in the per-node send map under `h.IfaceID`, fires
  `nodeConnHook(nodeConnRegistered, h.IfaceID)` (Q-AC002), and starts the
  writer goroutine (below) using `conn` and the same `*nodeConn`. It
  returns a behavior-cleanup `func()` that removes the map entry and
  fires `nodeConnHook(nodeConnRemoved, h.IfaceID)` ג€” it does NOT close
  `h.Done` directly (see "Shutdown ordering guarantee" below for exactly
  who triggers that close, and when).

This resolves the contradiction while keeping each side's ownership
symmetric: netingress creates the per-connection data; `runRouter` reads
it, reacts to it, and decides the connection's fate, but never mutates a
field's identity (it never re-creates `send`/`done`, only stores and
selects on the ones netingress handed it).

**Send-map value type (binding ג€” v1.3, F-DW-SP3-001/005):** because the
`done`-close trigger now has two independent call sites (this
connection's own teardown, and the router-wide drain-shutdown flush pass
ג€” see "Shutdown ordering guarantee" below), the per-node send map's value
type is no longer a bare `chan []byte`:

```go
type nodeConn struct {
    send     chan []byte
    done     chan struct{}
    doneOnce *sync.Once
}
```

stored as `sync.Map[routing.InterfaceID, *nodeConn]` (`Store(h.IfaceID,
&nodeConn{send: h.Send, done: h.Done, doneOnce: new(sync.Once)})` inside
`OnAccept`). Q-SINGLE-OBS's `sendMap.Range` callback reads
`nc := value.(*nodeConn); ch := nc.send` instead of a bare channel ג€” see
the reconciled Q-SINGLE-OBS text below.

> **v1.3 SUPERSEDED (completes the F-DW-SP2-002 sweep; extended by
> F-DW-SP3-001/005):** The "Writer goroutine ownership" steps below (v1.2)
> still described the pre-fix world in two respects this version corrects:
> (1) step 1 had `OnAccept` itself CREATE `send`/`done` ג€” corrected by the
> "Ownership split" above (netingress creates them; `OnAccept` receives
> them already built, inside `h`). (2) step 4's cleanup closed `done`
> unconditionally and unilaterally, as the ONLY possible trigger ג€” it is
> now one of two independent triggers (see "Shutdown ordering guarantee"

> `sync.Once`, and the writer's `done` case now flushes queued frames

> **v1.2 SUPERSEDED (F-DW-SP2-002, HIGH):** The "Writer goroutine ownership"
> steps below (v1.1) had the cleanup `func()` `close(send)` to unblock the

> - **Close race:** Q-SINGLE-OBS's single drain observer concurrently runs

>   `Signal`'s `defer recover()` (Q5/AC-003) then swallows the panic and
>   aborts the whole `Range` call, so every node ordered after the
>   disconnecting one in map-iteration order silently misses its DRAIN

> - **Per-connection RegisterObserver:** step 3 below contradicted
>   Q-SINGLE-OBS, which already rules (same v1.1 pass) that a SINGLE

**Single-closer / no-send-after-close invariant (binding ג€” v1.3, extends
F-DW-SP2-002 per F-DW-SP3-001/005):** the per-node `send` channel is NEVER
closed, by any goroutine, for the lifetime of the process ג€” unchanged from
v1.2. The per-node `done` channel IS still closed exactly once, but now
has TWO possible triggers racing to close it: (a) this connection's own
teardown (conn closed by the node, or by `ingressCancel` during shutdown)
via the behavior-cleanup `func()` `OnAccept` returns, and (b) the
router-wide drain-shutdown flush pass ("Shutdown ordering guarantee",
below), which proactively closes every still-live node's `done` BEFORE
`ingressCancel` runs, to force writers to flush the DRAIN frame before
their connections tear down. Two independent triggers racing to close the
same channel is exactly the double-close hazard F-DW-SP2-002 eliminated
for a single-trigger world ג€” generalized here via a per-node `doneOnce
*sync.Once` (created alongside `done`, stored in the `nodeConn` entry):
BOTH triggers call `nc.doneOnce.Do(func() { close(nc.done) })` rather than
`close(nc.done)` directly. Exactly one of the two calls performs the
actual close; the other is a harmless no-op. `send` becomes unreferenced
(GC-eligible) once its map entry is deleted and its writer goroutine has
exited; any frame the drain observer still manages to enqueue on it in the
race window between `Range` visiting the entry and cleanup deleting it is
a harmless best-effort send into an abandoned buffered channel ג€”
consistent with the existing best-effort delivery ruling (Q3.P1). No
consumer reads it after that point, and nothing re-closes or re-sends on
it, so there is no double-close and no send-on-closed panic, structurally.

**Writer goroutine ownership (binding ג€” v1.3, F-DW-SP3-001/005):** The
`runRouter` `OnAccept` closure:
1. Receives `h` (fully populated by netingress per the Ownership split
   above ג€” `h.Send`/`h.Done` are the actual channels netingress created,
   not copies) and `conn`. Wraps them as
   `nc := &nodeConn{send: h.Send, done: h.Done, doneOnce: new(sync.Once)}`
   and stores it in the per-node send map (`sendMap.Store(h.IfaceID, nc)`);
   fires `nodeConnHook(nodeConnRegistered, h.IfaceID)`.
2. Increments the router-scoped `writerWG` (`sync.WaitGroup`, one per
   `runRouter` invocation ג€” see "Shutdown ordering guarantee" below) by 1,
   then starts a writer goroutine:
   ```go
   go func() {
       defer writerWG.Done()
       for {
           select {
           case frame := <-nc.send:
               if _, err := conn.Write(frame); err != nil {
                   return // conn closed or write error; exit loop
               }
           case <-nc.done:
               // Flush any frame(s) already queued (e.g. the DRAIN frame
               // enqueued by the single observer's Range call, per the
               // Shutdown ordering guarantee below) before exiting ג€” do
               // NOT drop them silently (F-DW-SP3-005).
               for {
                   select {
                   case frame := <-nc.send:
                       if _, err := conn.Write(frame); err != nil {
                           return
                       }
                   default:
                       return
                   }
               }
           }
       }
   }()
   ```
3. Does NOT register a per-connection `drain.Observer` (unchanged from
   v1.2 ג€” see Q-SINGLE-OBS, which already rules a single startup observer
   instead).
4. Returns a behavior-cleanup `func()` that: removes the `InterfaceID`
   entry from the per-node send map; fires
   `nodeConnHook(nodeConnRemoved, h.IfaceID)`; calls
   `nc.doneOnce.Do(func() { close(nc.done) })` ג€” one of the two possible
   `done`-close triggers (see the invariant above; harmless no-op if the
   router-wide shutdown-flush pass already closed it first). `send`
   itself is NEVER closed ג€” see the invariant above.

**Concurrent write safety (B13 note):** The writer goroutine is the SOLE writer
to `conn`. `ServeConn` is the sole reader. `net.TCPConn` guarantees independent
read/write paths; no mutex is required between them.

### Shutdown ordering guarantee (binding ג€” v1.3, F-DW-SP3-005)

**The problem:** Q3.P1's best-effort delivery means the single drain
observer returns as soon as it has QUEUED the DRAIN frame to every live
node's `send` channel (a non-blocking `select`/`default`) ג€” `Wait`
returning `nil` says only "queuing finished," not "every writer goroutine
has flushed its queued frame to the wire." The shutdown block
(`cmd/switchboard/mgmt_wire.go`, verified at `ef1ee1e`:638ג€“644) calls
`ingressCancel()` immediately after `drainCoord.Wait(drainCtx)` returns,
which tears down `ingressCtx` and, via `netingress.Serve`'s own
ctx-cancel handling, closes every accepted connection's underlying
`net.Conn`. Nothing orders the writer goroutine's `conn.Write(DRAIN)`
before that close ג€” under scheduling pressure the `net.Conn` can close
before the writer even reaches its `case frame := <-nc.send:` branch, and
the node reads EOF instead of the DRAIN frame. This is the AC-004/AC-001
flake source F-DW-SP3-005 identifies.

**Ruling ג€” insert a bounded flush phase between `Wait` and
`ingressCancel()`:**

```
Signal ג†’ Wait ג†’ close all live nodes' done (router-wide flush pass) ג†’
writerWG.Wait() bounded by a short flush deadline ג†’ ingressCancel()
```

Concretely, the shutdown block gains two new steps between the existing
`drainCoord.Wait(drainCtx)` call and `ingressCancel()`:

```go
// (existing) drainCoord.Signal(drainCtx); if derr := drainCoord.Wait(drainCtx); ... ; drainCtxCancel()

// NEW ג€” router-wide flush pass: tell every still-live writer goroutine to
// drain its queued frame(s) and exit, BEFORE the connections are torn down.
sendMap.Range(func(key, value any) bool {
    nc := value.(*nodeConn)
    nc.doneOnce.Do(func() { close(nc.done) })
    return true
})

// NEW ג€” bounded wait for writers to finish flushing. writerWG has no
// native timeout, so race it against a short deadline; on expiry proceed
// anyway (best-effort, consistent with Q3.P1 ג€” a stalled/non-reading node
// must not block router shutdown indefinitely).
flushDone := make(chan struct{})
go func() {
    writerWG.Wait()
    close(flushDone)
}()
select {
case <-flushDone:
case <-time.After(drainFlushTimeout):
    routerLogger.Log("runRouter: drain flush deadline exceeded; proceeding with shutdown")
}

ingressCancel() // (existing, now runs only after the flush phase)
dataWG.Wait()   // (existing)
```

`drainFlushTimeout` is a small fixed constant ג€” PROVISIONAL value (e.g.
200ms); the MECHANISM (bounded-select flush phase before `ingressCancel`)
is BINDING, the exact duration is left to the story-writer/implementer to
tune against real write-syscall latency for a 48-byte frame. It is
deliberately NOT `cfg.DrainTimeout` ג€” that budget is already spent by the
time this phase runs; this is a second, much shorter bound purely for
flushing already-queued, already-small frames to a socket that should
accept them near-instantly under normal conditions.

**Why `sync.Once` and not a direct `close(nc.done)` here:** this
router-wide pass and each connection's own behavior-cleanup (Writer
goroutine ownership, step 4, above) are two INDEPENDENT triggers that can
both fire for the same node ג€” e.g. a node that happens to disconnect on
its own at the same moment the router is draining. Racing two
unconditional `close(nc.done)` calls is exactly the double-close panic
F-DW-SP2-002 eliminated for a different pair of callers; `doneOnce`
(created alongside `done`, stored in the same `nodeConn` entry)
generalizes that fix to any number of trigger sites, present or future,
without reintroducing the hazard.

**FCL:** this is production code inside `runRouter`
(`cmd/switchboard/mgmt_wire.go`, FCL row 1 ג€” no new file). `writerWG` is a
new `sync.WaitGroup` local to `runRouter`, alongside the existing
`dataWG`/`mgmtWG`.

**Reconciles AC-001/AC-004 (Q3-AMENDED) and VP-037 Stage 1 (Q4-AMENDED):**
both discharge traces' final "read from the inbound conn; assert DRAIN
frame received" step already assumed the frame would be observable before
the connection tore down ג€” that assumption was NOT previously backed by

> `cmd/switchboard/mgmt_wire.go` at `ef1ee1e`: the netingress accept loop
> goes live in Phase (f) (`ingressCtx`/`ingressCancel`, the
> `netingress.Serve` goroutine) BEFORE `drainCoord := drain.New(drainWindow)`
> is even constructed ג€” accept-loop start strictly precedes
> drain-coordinator construction textually, so "before serving any
> connections" is false as an ordering claim. The correct statement:
> **the single observer is registered at `drainCoord`-construction time,
> which is guaranteed to occur before `drainCoord.Signal`** (the only
> point at which registration ordering matters ג€” `RegisterObserver`
> no-ops after `Signal`, per `drain.go`). Registration-vs-accept-loop
> ordering is immaterial for a separate reason: the per-node send map (and
> the `IfaceIDSeed` counter state) are zero-value-usable ג€” `sync.Map`'s
> zero value needs no explicit initialization step, so `OnAccept` can
> safely store entries into it from the moment Phase (f) starts, whether
> or not the observer has been registered yet. The only real invariant is:
> the send map and any counter state must exist (be declared, even as
> zero values) before Phase (f) so `OnAccept` can capture them by
> reference; observer registration may happen any time before `Signal`.
> **No phase reorder is mandated by this correction** ג€” the existing
> Phase (e)/(f)/drain-construction ordering in `mgmt_wire.go` is left
> as-is; only the prose describing it is corrected. All three sites
> (listed above) are corrected in place by this pass.

### The problem

`drain.Drain.RegisterObserver` appends to `[]Observer` with no deregister API
(verified `drain.go`:82ג€“93 at `e940fc2`). Per-connection registration leaks
observers on churn (a node connects, drains its observer, reconnects ג€” the stale
observer from the first connection fires again at next drain, sending a DRAIN
frame to a closed channel).

### Ruling

**Single-observer registered ONCE, at `drainCoord` construction time,
iterating the live per-node send map at `Signal` time.** Specifically:

1. `runRouter` registers a SINGLE `drain.Observer` function with `drainCoord`
   immediately after constructing it. This registration happens once, and
   is guaranteed to precede `drainCoord.Signal` *(corrected v1.3 ג€”
   F-DW-SP3-008: not "before any connections arrive"; see the correction
   banner above)*.

2. The single observer closure captures a reference to the per-node send map
   (`sync.Map[routing.InterfaceID, *nodeConn]` as of v1.3 ג€” F-DW-SP3-001/005;
   see Q-SEAM's "Send-map value type").

3. At `Signal` time, the observer first fires the test-only
   `drainObserverFiredHook` (Q-AC003, new v1.3 ג€” F-DW-SP3-006), then
   iterates the live map:
   ```go
   observer := func(ctx context.Context) {
       if drainObserverFiredHook != nil {
           drainObserverFiredHook()
       }
       sendMap.Range(func(key, value any) bool {
           nc := value.(*nodeConn)
           frame := assembleDrainFrame(...)
           select {
           case nc.send <- frame:
           default: // channel full ג€” best-effort (never closed, per the
                    // single-closer/no-send-after-close invariant in
                    // Q-SEAM v1.3 ג€” F-DW-SP2-002/F-DW-SP3-001/005; this
                    // send cannot panic)
           }
           return true
       })
   }
   drainCoord.RegisterObserver(observer)
   ```

4. The observer then blocks on `drainCtx.Done()` (or returns immediately ג€”
   per Q3 best-effort ruling, observer returns after dispatching to all send
   channels, context deadline is the bound).

**Consequence for `drain.Drain` API:** No `deregister` API is needed.
`drain.Drain.RegisterObserver` retains its current shape. The single-observer
pattern eliminates the stale-observer problem without requiring API changes to
`internal/drain`.

**Consequence for Q-SEAM OnAccept:** The `OnAccept` closure in Q-SEAM registers
the per-node send channel in the live map (and deregisters on cleanup), but does
NOT call `drainCoord.RegisterObserver`. Only the single startup observer is registered.

## Q-CTL-GUARD ג€” Receive-side ctl payload_len guard (F-DW-SP1-008 ג€” added v1.1)

**Ruling: `cmd/switchboard` MUST length-check ctl frame payload before reading
`payload[0]`. Guard home and pin test specified. BINDING.**

### The problem

BC-2.01.008 EC-002 (v1.0, PO ruling): `payload_len < 4 ג†’ E-PRT-002`. A node
(or attacker) can send a `FrameTypeCtl` frame with `PayloadLen = 0`. The outer
header parses cleanly. Without a length check, `payload[0]` panics (index out of
range) in a goroutine with no recover, crashing the router.

### Guard specification (binding ג€” v1.1)

**Location (firmed up ג€” binding v1.2, F-DW-SP2-003):** The ctl frame receive
path lives EXCLUSIVELY in the netingress `route` RouteFn closure at
`mgmt_wire.go`:473ג€“475 (`route := func(hdr frame.OuterHeader, payload []byte)
error { return routing.RouteFrame(hdr, payload, router) }`) ג€” the closure
`runRouter` passes to `netingress.Serve`. It does NOT live in the PE
`FrameFn` closure at `mgmt_wire.go`:549ג€“551
(`connector.SetFrameCallback(...)`). The v1.1 "route RouteFn closure OR the
per-connection dispatch code" hedge is resolved: it is the `route` closure
(or dispatch code called directly from it in the same story-added ctl
branch), full stop. The `FrameFn` closure is FROZEN per Q2-AMENDED (nil
`ForwardFunc`, single-element `interfaceSet`, no code change this story) ג€”
it already passes any `FrameTypeCtl` frame it happens to see straight into
`OnFrameArrival` unmodified, and this story does not touch that path. A
guard or dispatch requirement written against the `FrameFn` closure (e.g. a
story AC citing "PC-4"/"PSI-3" against that closure) is a scope-conflation
error: the ctl payload guard and `control_type` dispatch both belong on the
`route` closure path, not the `FrameFn` path.

**Terminal-consumer basis for the unconditional guard (binding ג€” v1.3,
F-DW-SP3-004; PO ruling BC-2.01.008 v1.1 Inv-2 consumed):** BC-2.01.008's
Precondition 2 requires "the router is the addressed terminal consumer of
this frame" before the `control_type` schema applies. This guard, wired
directly onto the netingress `route` closure, applies to EVERY `ctl`
frame `route` sees, with NO runtime "is this addressed to me" check ג€” that
is legitimate ONLY because BC-2.01.008 Inv-2 (new v1.1) establishes that
netingress-arriving `ctl` frames are terminal-consumer BY CONSTRUCTION
under the current architecture: `netingress` is exclusively the
node-facing accept loop, and no inter-router relay path exists anywhere
in this codebase (`internal/routing.SVTNRoute` validates forwarding-table
entries but never relays frame bytes to another connection) ג€” so a `ctl`
frame arriving here can never be in transit to a downstream router. Inv-2
is explicitly scoped to today's architecture and carries a REVISIT
TRIGGER: if a future story introduces inter-router forwarding/relay (a
topology where a `ctl` frame could arrive via a router-facing connection
addressed to a downstream router), this guard's unconditional posture
must be re-examined against Precondition 2's general terminal-consumer
language ג€” Invariant 1's DI-001 carve-out and Precondition 2 remain the
general rule; Inv-2 only certifies that the netingress-specific instance
collapses to "always true" today. Story-writer: carry this revisit
trigger forward into any follow-on story that adds router-to-router
connections.

The guard and dispatch are placed in the `route` RouteFn closure (or
dispatch code it calls directly) before any `payload[0]` access:

```go
if hdr.FrameType == frame.FrameTypeCtl {
    if len(payload) < 4 {
        // E-PRT-002: control frame truncated. No `conn` is in scope here
        // ג€” the route RouteFn closure signature is func(hdr, payload)
        // error; there is no per-connection dispatch seam with conn
        // access (F-DW-SP3-003) ג€” do not add one for this log line.
        routerLogger.Log(fmt.Sprintf(
            "netingress: E-PRT-002: ctl frame payload_len=%d < 4; discarding",
            len(payload)))
        // silent-ignore: no connection close per BC-2.01.008 EC-002
        return nil // drop, keep reading
    }
    controlType := payload[0]
    switch controlType {
    case 0x01: // DRAIN ג€” this story's dispatch target; no-op on receipt
               // (DRAIN is router-originated broadcast per Q2-AMENDED; a
               // router does not act on an inbound DRAIN byte from a node)
    default:
        // Unknown/forward-compat control_type (includes the reserved-but-
        // undispatched 0x02 RESYNC opcode until S-BL.RESYNC-FRAME lands).
        // BC-2.01.008 PC-4 (v1.1, F-DW-SP3-002 / PO ruling): silent-ignore
        // means NO logging of any kind on this path ג€” not even a
        // diagnostic/informational line ג€” and no connection close. Do NOT
        // add a log call to this arm; an earlier draft of this guard
        // logged an "unrecognized ctl control_type" line here, which
        // PC-4 v1.1 explicitly forbids (unbounded log growth across a
        // rolling upgrade ג€” see BC-2.01.008 PC-4's rationale). This is
        // intentionally asymmetric with the E-PRT-002 branch above, which
        // DOES log ג€” a truncated payload is corruption, diagnostic-worthy;
        // a well-formed-but-unrecognized opcode is ordinary
        // forward-compatible protocol evolution, not an anomaly.
    }
    return nil
}
```

**Disposition:** Silent-ignore (no connection close) for BOTH arms, but the
LOGGING behavior is asymmetric (BC-2.01.008 PC-4 v1.1, F-DW-SP3-002 ג€” PO
ruling): the short-payload guard DOES log (BC-2.01.008 EC-002: "discard
with E-PRT-002 log") while the unknown-`control_type` default arm does
NOT log at all (PC-4's strengthened no-logging rule, including no
diagnostic/informational line). The node can send further frames after
either case ג€” neither closes the connection.

**Pin tests (binding ג€” v1.3, respecified per F-DW-SP3-002/003):**

Both live in `cmd/switchboard/router_drain_wire_test.go` (the new file from
Q6 FCL), targeting the `route` closure path (NOT the `FrameFn` path):

`TestRouter_CtlFrame_ShortPayload_NoConnClose`:

`TestRouter_CtlFrame_UnknownControlType_SilentIgnore` (respecified v1.3,
