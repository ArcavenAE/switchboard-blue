# S-5.02 Pass-4 Scope Rulings

Issued 2026-06-30. These rulings address the BLOCK result from Pass-4 across all
three adversarial lenses. One CRITICAL scope gap requires story decomposition;
three HIGH findings require spec alignment; one LOW finding requires BC wording

## Ruling 1 — F-C1 (CRITICAL): No Daemon-Side RPC Handlers Exist

**Finding summary.** The entire daemon-side path for `paths.list`, `router.metrics`,
and `router.status` is absent from the worktree. No `PathsListResponse`, `PathEntry`,
or `RouterMetricsResponse` types exist in `internal/metrics`. No handler is registered
in `internal/mgmt` or `cmd/switchboard` for these RPC methods. Tasks 11 and 12 in
S-5.02 v1.5 ("Wire PathSnapshot.P99RTTMs → rtt_p99_ms" and "Wire Degraded →
status: degraded") are unfulfilled. The story's end-to-end behavior — an operator
running `sbctl paths list` against a real daemon and seeing real per-path metrics —
is INERT. Tests prove only that sbctl's JSON envelope correctly passes through
whatever bytes a stub daemon responds with.

**Ruling: Option B — Defer daemon-side wiring to S-W5.04 (new story).**

Rationale:

1. S-5.02 has already been repointed twice (3→5pts) and propagated through three
   adversarial passes. Expanding scope a third time during Pass-4 erodes the
   Wave-5 close timeline.
2. The client-side delivery is real and valuable: operators will have a functional
   sbctl surface once daemon handlers land. The structural pattern is identical to
   BC-2.06.001 PC-5 split (S-5.01 quality state machine / S-7.03 surfacing): a
   clean per-story BC ownership boundary.
3. S-W5.04 cleanly carries the BC-2.06.003 daemon-side concern as a separate story
   with its own adversarial cycle and VP-047 integration test coverage.
4. The §Scope Boundary in S-5.02 already documents one such split (sessions.list
   → S-7.03). A second split is structurally consistent with that precedent.
5. VP-047 (integration test: per-path field presence) is non-blocking for Wave-5
   TDD per the Architecture Compliance Rules in S-5.02 v1.5. Deferring it to
   S-W5.04 does not block Wave-5 convergence.

**Consequences for S-5.02 v1.6 (story-writer executes):**

- Add §Scope Boundary note: "daemon-side `paths.list`, `router.metrics`, and
  `router.status` RPC handlers and response types (`PathsListResponse`, `PathEntry`,
  `RouterMetricsResponse`) deferred to S-W5.04 per Pass-4 Ruling 1
  (decisions/S-5.02-pass4-scope-ruling.md)."
- Reframe AC-001 through AC-006 and AC-008 to acknowledge stub-only test posture:
  - AC-001: "sbctl client correctly serializes and forwards `paths.list` request
    and envelopes the daemon response into the BC-2.06.003 PC-1 schema"
  - AC-002: "sbctl client correctly serializes `router.metrics --svtn=<id>` request
    and envelopes the BC-2.06.003 PC-2 schema"
  - AC-003: "sbctl router status alias is a CLI dispatch shim; JSON structure
    (minus quality) is identical to paths list response; alias adds quality column
    per BC-2.06.003 PC-3"
  - AC-006: note that daemon unreachable path is fully exercised (no stub needed)
- Note in the Token Budget that Tasks 11 and 12 are now owned by S-W5.04.
- VP-047 defers to S-W5.04 (remove from vp_traces or annotate as deferred).
- acceptance_criteria_count remains 7 (ACs do not change count; they change scope).

**New story S-W5.04 minted today (see §Appendix: S-W5.04 stub).**

## Ruling 2 — F-H1/F-H2/F-M5 (HIGH): Bucket-Layout Drift

**Finding summary.** The implementation `rttHistogramBuckets` adds 25ms-wide buckets
at 100/125/150/175ms, yielding buckets [100,125), [125,150), [150,175), [175,200)
where ARCH-03 v1.6 §Bucket layout specifies two 50ms-wide buckets: [100,150) and
[150,200). Story task list and BC-2.06.003 PC-1 reference "rolling sample buffer
maintained by the PathTracker" rather than the canonical fixed-bucket histogram.
ARCH-03 is the canonical source for bucket layout and must not be bumped.

**Ruling: Option (a) — reconcile implementation to ARCH-03 canonical layout.**

The canonical right-edge sequence is:
  {25, 50, 75, 100, 150, 200, 300, 400, 500, 750, 1000, 1250, 1500, 1750, 2000, +Inf}
(16 buckets, Bucket 4 = [100,150), Bucket 5 = [150,200), matching ARCH-03 v1.6.)

Consequences for implementer fix-burst:
- Rewrite `rttHistogramBuckets` (or equivalent) to match the 16 canonical right-edges.
- Rewrite AC-005 test bounds to derive from the canonical layout: test cases must
  cover (a) all samples in 0–25ms bucket, (b) samples in [0,25ms) and [100,150ms)
  bucket, (c) a distribution where p99 falls in [150,200ms) bucket.
- The approximation error bound in AC-005 and ARCH-03 is "≤ max_bucket_width for
  the bucket containing the true p99." For buckets 0–3 (width 25ms) this is 25ms;
  for buckets 4–5 (width 50ms) this is 50ms. AC-005 must specify the bound per
  bucket, not a single global bound.

No ARCH-03 changes. No BC-2.06.003 changes for this finding (PC-1 text references
the field semantics, not the histogram layout).

## Ruling 5 — F-LO1 (LOW): BC-2.06.003 PC-1 Wording Alignment

**Finding summary.** BC-2.06.003 v1.5 PC-1 says "`rtt_p99_ms` — p99 of per-path RTT
samples over the observation window (float64); computed from the rolling sample buffer
maintained by the PathTracker". ARCH-03 v1.6 says "histogram counts are never reset."
These are different semantics: "rolling sample buffer" implies a bounded window;
"never reset histogram" accumulates indefinitely.

**Ruling: Update BC-2.06.003 PC-1 to align with ARCH-03 v1.6 (canonical).**

PC-1 `rtt_p99_ms` description replaces "rolling sample buffer maintained by the
PathTracker" with "fixed-bucket histogram maintained by the PathTracker (counts
never reset; approximation error ≤ bucket width)". PO-executed in BC-2.06.003 v1.6
(see below). No behavioral change — the existing EC-003 ("pending" when < 10 samples)
is retained.
