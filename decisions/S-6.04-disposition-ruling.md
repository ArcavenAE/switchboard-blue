# S-6.04 Disposition Ruling: Closed — Subsumed by S-7.04-FU-SIGHUP-RELOAD

## Decision

**S-6.04 is CLOSED as subsumed by S-7.04-FU-SIGHUP-RELOAD.**

No residual scope remains in S-6.04. Every obligation it carried — behavioral contract
postconditions, edge case EC-004, and all three acceptance criteria — is fully covered
by S-7.04-FU-SIGHUP-RELOAD. S-6.04's frontmatter is updated to `status: closed-subsumed`.

## Rationale

S-6.04 was created as a backlog anchor (Wave 4 audit, 2026-06-28) for the SIGHUP config
reload path that S-6.01 deferred. S-7.04-FU-SIGHUP-RELOAD was created (2026-07-06) as
the S-7.04 DELIVERY's formal deferral artifact for AC-001 (SIGHUP runtime reload), and
its scope covers the entire SIGHUP/config-reload surface for the router daemon.

The two stories target exactly the same subsystem boundary
(`cmd/switchboard/main.go` signal loop + `internal/config` re-parse path) and the same
behavioral contract (BC-2.09.003 Inv-3, EC-004). No narrowing is possible because
S-6.04 carries no scope that is NOT already in S-7.04-FU-SIGHUP-RELOAD's scope.

## VP-028 and VP-029 Disposition

### Finding: Both VPs are already proven; their scope does not cover SIGHUP reload

**VP-028** (`verification_lock: true`, `proof_completed_date: 2026-07-06`, version 1.3):
Property statement — "∀ duration d: d < 5ms ∨ d > 50ms ⟹ (&Config{...}).Validate()
returns a non-nil error ∧ code E-CFG-001 ∧ field 'tick_interval' in message."
This is a startup-validation property on `Config.Validate` behavior. It is discharged by
`TestVP028_AnyValidationErrorIsECFG001` (internal/config/config_test.go:572) and
`TestConfigValidate_RejectsOutOfRangeTickInterval` (config_test.go:91).

**VP-029** (`verification_lock: true`, `proof_completed_date: 2026-07-06`, version 1.3):
Property statement — "∀ required field F in {listen_addr, tick_interval}: zeroing F while
other fields are valid causes Validate() to return non-nil E-CFG-001 error naming F."
This is also a startup-validation property on `Config.Validate`. Discharged by
`TestVP029_ErrorMessageNamesFieldAndSuggestion` (config_test.go:647) and
`TestVP028_AnyValidationErrorIsECFG001` (config_test.go:572).

**Conclusion:** VP-028 and VP-029 are fully proven against the `internal/config` module's
startup validation behavior. Their property statements do not include the SIGHUP signal
path or the daemon-level reload integration logic. They were listed in S-6.04's frontmatter
as traces because the fail-closed reload property (Inv-3) builds on Config.Validate — but
the VPs themselves are narrowly scoped to the validation function, not the signal handler
or the reload integration test.

**No transfer is required.** VP-028 and VP-029 remain anchored to the config validation
infrastructure (internal/config), where they are PROVEN. Their obligations are discharged.
S-7.04-FU-SIGHUP-RELOAD's fail-closed reload property (BC-2.09.003 Inv-3/EC-004) will be
covered by the integration test the implementer writes for AC-002, which is a behavioral
integration test — not the unit-level startup-validation property covered by VP-028/VP-029.

### Spec Note: BC-2.09.003 Verification Properties Table Drift

The BC-2.09.003 Verification Properties table (§"Verification Properties") lists VP-028
and VP-029 as covering "Config reload failure leaves daemon on previous config". This is
an overstatement relative to the actual VP documents, which scope only to
`Config.Validate` behavior. This is pre-existing authoring drift in the BC; it predates
this ruling and is not a defect introduced here. Remediation (narrowing the BC table row or
adding a new VP for the reload integration property) is left to the spec-steward or the
story-writer when elaborating S-7.04-FU-SIGHUP-RELOAD. No change to VP-028 or VP-029
documents is required.
