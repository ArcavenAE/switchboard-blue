# Adjudication: W3-R3-F2 — BC-2.05.008 EC-006 Ownership and SW305 Family

## 1. Finding Restatement

**W3-R3-F2 (Wave 3, Gate Adversary Pass-3, run at 10dd880):**
BC-2.05.008 EC-006 describes a repeated HMAC failure alert mechanism. Story S-W3.05
(PR #16, sha fa6345e) implements the FailureCounter type and wires RecordHMACFailure
into RouteFrame. The adversary flagged this as an ADJUDICATION-DEPENDENT scope-boundary
finding: the mechanism is implemented, but the BC text at gate time did not fully
specify the injection seam, the interface type, or the routing-layer call semantics.
The adversary asked: does BC-2.05.008 EC-006 now reflect what was built, or does the
implementation diverge from intent?

**Related open items (SW305 family, same gate pass):**
- SW305-M2 (MED): WithFailureCounter accepts the unexported `hmacFailureRecorder`
  interface, not a pinned `*admission.FailureCounter` concrete type. Is this ratified
  or should it be reverted to a concrete type?
- SW305-M3 (MED): WithNow clock seam and threshold<=0 guard are not in the BC contract
  (BC-2.05.008). Should the BC pin these?
- SW305-M4 (MED): Integration test for fire-once end-to-end does not exercise 6th/7th
  failures through RouteFrame (i.e., suppression after threshold crossing is not verified
  at the routing integration level). Is a new test story needed?

## 2. Spec-vs-Implementation Comparison

### 2.1 EC-006 — What the BC Says (post-v1.3, current)

BC-2.05.008 v1.3 EC-006 states:

> `RouteFrame` calls `router.failureCounter.RecordHMACFailure(fmt.Sprintf("%x", hdr.SrcAddr))`
> immediately before returning `ErrHMACVerificationFailed`. The `failureCounter` field is
> typed as `hmacFailureRecorder` (unexported interface: `RecordHMACFailure(string)`);
> `*admission.FailureCounter` is the production implementation injected via
> `WithFailureCounter`. Tests may inject a fake satisfying the interface. When
> `RecordHMACFailure` detects that the sliding-window count for the srcAddr hex string
> has reached >=5 within 60 seconds, it emits E-ADM-017 ... This mechanism is fully
> specified in BC-2.05.005 PC-3 ... `RouteFrame` itself has no alert logic — it is a
> call-through.

BC-2.05.008 v1.3 PC-5 (postcondition 5) states:

> On every `ErrHMACVerificationFailed` return path (postconditions 2 and 4),
> `RouteFrame` calls `router.failureCounter.RecordHMACFailure(srcAddrHex)` BEFORE
> returning, where `srcAddrHex` is the lowercase hex encoding of the 8-byte
> `hdr.SrcAddr` field (e.g. `fmt.Sprintf("%x", hdr.SrcAddr)`). ... The `failureCounter`
> field is typed as the unexported `hmacFailureRecorder` interface
> (`RecordHMACFailure(string)`); `*admission.FailureCounter` is the production
> implementation injected via `WithFailureCounter`. Test doubles may inject any
> satisfying type.

### 2.2 What the Implementation Does (routing.go, PR #20 / sha 418de54)

`internal/routing/routing.go` defines the unexported interface:

```go
type hmacFailureRecorder interface {
    RecordHMACFailure(srcAddr string)
}
```

The `Router` struct carries:

```go
failureCounter hmacFailureRecorder // nil = no counter; set via WithFailureCounter
```

`WithFailureCounter` accepts `hmacFailureRecorder` (any satisfying type). In RouteFrame:
- PATH-A (no forwarding entry): nil-guards then calls
  `r.failureCounter.RecordHMACFailure(fmt.Sprintf("%x", hdr.SrcAddr))` before return.
- PATH-B (tag mismatch): same nil-guard + call before return.
- Success path: no call.

`internal/admission/failure_counter.go` (PR #16 / sha fa6345e):
- `FailureCounter` type with `RecordHMACFailure(srcAddr string)`, sliding-window, drain-only
  re-arm, append-skip, LRU source cap, `WithNow` clock seam, constructor panics on
  threshold<1 / windowDuration<=0 / nil-logger.
- All 17 acceptance criteria fully implemented and covered by tests
  (`failure_counter_test.go`, `failure_counter_adversarial_test.go`,
  `failure_counter_property_test.go`).

### 2.3 Divergence Assessment

**There is no divergence.** The implementation precisely matches BC-2.05.008 v1.3 (which
was written to ratify the intended design before this gate adjudication was formally
issued). The BC was amended to v1.3 during story delivery (modifier note dated 2026-06-27
is in the BC frontmatter), and the implementation is consistent with every normative
statement in that version.

The nil-guard on `r.failureCounter` in routing.go is an implementation detail that
preserves backward compatibility when no counter is injected (e.g. tests that don't
care about E-ADM-017). This nil-guard is not a spec divergence: BC-2.05.008 PC-5 says
"RouteFrame calls ... RecordHMACFailure" — it implicitly assumes the counter is wired
(the BC does not enumerate the no-counter case as a postcondition because the spec
assumes proper construction of the daemon). The nil-guard is a defensive programming
measure consistent with `WithFailureCounter` being optional.

## 3. Disposition

### 3.1 W3-R3-F2 (EC-006) — RATIFY

**Disposition: RATIFY. No Wave-4 correction story required.**

Rationale:
1. BC-2.05.008 was already amended to v1.3 (before this adjudication was formally
   issued) to specify exactly the implemented semantics: `hmacFailureRecorder` interface,
   `WithFailureCounter` injection, nil-guard-free normative call, srcAddrHex encoding,
   call-through with no alert logic in RouteFrame.
2. The implementation in routing.go (PR #20) and failure_counter.go (PR #16) matches
   every postcondition, invariant, and edge case in BC-2.05.008 v1.3.
3. VP-059 v1.2 covers the BC-2.05.008 EC-006 seam explicitly in its Source Contract
   section ("BC-2.05.008 — RouteFrame Wire-Layer HMAC Enforcement / Postcondition 5 /
   Edge Case EC-006: RecordHMACFailure is a call-through from RouteFrame").
4. The adversary classification of F-2 as "ADJUDICATION-DEPENDENT scope-boundary, NOT
   a raw code defect" is confirmed correct — the code is clean and the BC reflects the
   design intent.

**EC-006 is ratified as specified in BC-2.05.008 v1.3. No version bump needed (v1.3 is
the current version and already contains the ratified semantics).**

### 3.2 SW305-M2 — RATIFY the unexported-interface design

**Disposition: RATIFY. No spec or implementation change required.**

The question was: should `WithFailureCounter` accept `*admission.FailureCounter`
(concrete) or `hmacFailureRecorder` (interface)?

The interface design is correct and should be pinned in the spec. Rationale:

1. **Architectural correctness.** `internal/routing` (position 5) is permitted to import
   `internal/admission` (position 4) per ARCH-08 §6.5. However, pinning a concrete type
   would make the routing package depend on the specific implementation details of
   `admission.FailureCounter` — reducing testability and coupling two boundary packages
   more tightly than necessary.

2. **Test double expressibility.** The `hmacFailureRecorder` interface allows test doubles
   in `routing_test.go` without importing `internal/admission`. This is the established
   Go pattern for injectable boundary dependencies (consistent with how `Logger` is
   handled in the same package).

3. **BC already specifies this.** BC-2.05.008 v1.3 PC-5 and EC-006 already state the
   interface name, its signature, and the concrete production type. The spec is not
   under-specified — the adversary's observation was that the BC text predating v1.3
   did not pin this. v1.3 does.

4. **No revert warranted.** Reverting to `*admission.FailureCounter` would reduce
   testability and violate Go's prefer-interfaces-for-injection idiom without any
   compensating benefit.

**BC-2.05.008 v1.3 already captures this correctly. SW305-M2 is closed by ratification.**

---

### 3.3 SW305-M3 — RATIFY WithNow and threshold guard as BC-2.05.005 scope; no change to BC-2.05.008

**Disposition: RATIFY the split scope. No change to BC-2.05.008.**

SW305-M3 asks whether `WithNow` (clock seam) and the threshold<=0 constructor guard
should appear in BC-2.05.008.

They should not. Rationale:

1. **`WithNow` is a FailureCounter API concern, not a routing concern.** `WithNow` is
   a constructor option on `admission.FailureCounter`, documented in BC-2.05.005 PC-3
   (v1.4 addition). `RouteFrame` never sees the clock seam — it only calls
   `RecordHMACFailure(string)` on the `hmacFailureRecorder` interface. The clock seam
   is an internal implementation detail of the FailureCounter type that the routing
   layer is not aware of and should not be aware of.

2. **Constructor validation guards are FailureCounter-owned.** The threshold<1,
   windowDuration<=0, and nil-logger panics are specified in BC-2.05.005 PC-3 and
   captured in S-W3.05 AC-013. These are programmer-error guards on the
   `NewFailureCounter` constructor. BC-2.05.008 has no constructor to guard — it
   specifies routing behavior. These guards are not a gap in BC-2.05.008.

3. **VP-059 covers both.** VP-059 v1.2 proof harness includes `TestFailureCounter_ConstructorValidation`
   with threshold=0 and windowDuration=0 panic sub-cases. The threshold<=0 guard is
   verified.

4. **No spec gap exists.** BC-2.05.005 v1.8 PC-3 and S-W3.05 AC-013 together fully
   specify the constructor validation contract. Adding it to BC-2.05.008 would create
   redundancy and ambiguity about which BC is authoritative.

**SW305-M3 is closed. WithNow and threshold guards belong in BC-2.05.005 / VP-059, where
they already reside. No change to BC-2.05.008.**

### 3.4 SW305-M4 — DEFER to Wave 4 as a test-writer task (not a BC gap)

**Disposition: DEFER to Wave 4. A targeted test story is warranted but is not a
product-owner spec gap.**

SW305-M4 asks whether the integration test (`TestRouteFrame_WithFailureCounterInterface`,
S-W3.05 AC-009) should exercise the fire-once suppression invariant end-to-end through
RouteFrame (i.e., verify that a 6th or 7th HMAC failure through RouteFrame does NOT
cause a second E-ADM-017 before window drain).

Assessment:

1. **The spec is not deficient.** BC-2.05.008 PC-5 specifies that `RouteFrame` calls
   `RecordHMACFailure` on every failure path. The fire-once invariant is BC-2.05.005
   PC-3's responsibility and is verified by VP-059 properties (a) and (b), which are
   unit-tested at the `FailureCounter` level (S-W3.05 AC-004, AC-006, AC-017b).

2. **However, the integration seam is not end-to-end tested.** AC-009 verifies that the
   interface is wired (the call happens) and that it is NOT called on success. It does
   not verify the full end-to-end behavior: 5 RouteFrame failures → E-ADM-017; 6th
   failure → no second E-ADM-017. This gap is a test coverage gap, not a spec gap.

3. **The BC does not need to change.** The existing BC-2.05.008 EC-006 test vector row
   already states: "5 consecutive HMAC failures from same src_addr within 60s → After
   5th: E-ADM-017 emitted by FailureCounter; RouteFrame called RecordHMACFailure 5
   times." This implies the suppression contract (6th call through RouteFrame should not
   produce a second E-ADM-017 in the same window). The behavior is specified; the test
   gap is in coverage.

4. **Action:** Create a Wave-4 test task (test-writer scope) to add an integration test
   to `routing_test.go` that:
   - Injects a real `*admission.FailureCounter` (with `WithNow` clock injection) into
     a `Router` via `WithFailureCounter`.
   - Calls `RouteFrame` with bad HMAC frames from the same srcAddr 5 times → asserts
     exactly 1 E-ADM-017 alert.
   - Calls `RouteFrame` a 6th and 7th time → asserts still exactly 1 E-ADM-017 alert
     (no re-fire within the same window).
   This test exercises the routing↔admission integration seam for fire-once suppression.

**SW305-M4 action: add to Wave-4 drift table as a test-writer task. No BC amendment
required.**

cause: BC-2.05.008 v1.2 (pre-gate) did not fully specify the injection interface

(SW305-M4), which is test-writer scope, not a product-owner spec amendment.

No Wave-4 correction story is needed for the core EC-006 mechanism.
