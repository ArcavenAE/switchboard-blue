## Lens 2 — Test Rigor

**Result: 1 HIGH finding, 1 MED finding.**

### F-P4L2-HIGH-1 — AC-002 test-name mis-anchor + false attestation in v1.5 changelog

**Confidence: HIGH.**

AC-002 in story v1.6 names the test `TestSVTNManager_Destroy_KeyPurgeUnderConcurrentAdmission`
as the oracle for the key-purge postcondition and concurrent-admission race property.

Inspection of `internal/svtnmgmt/svtnmgmt_test.go` at cc78688 finds no function with that
exact name. The actual symbols at the relevant lines are:

- `TestSVTNManager_Destroy_KeyPurgePostcondition` (line 1628) — validates that all admitted
  keys are absent from the key set after `Destroy` completes (quiescent-state purge oracle).
- `TestSVTNManager_Destroy_ConcurrentAdmissionIsRaceFree` (line 1835) — launches concurrent
  goroutines attempting admission during and after `Destroy`; asserts no data race and no
  torn key set (concurrency-race oracle).

The v1.5 changelog entry under finding `(F-M1)` claims:
> "AC-002 test name corrected to `TestSVTNManager_Destroy_KeyPurgeUnderConcurrentAdmission`
> (matches implementation)."

That attestation is false: the stated symbol does not appear in the implementation. The v1.5
fix-burst introduced a test name that does not exist in code. This is a governance-truth
failure: the story certifies a test name that cannot be verified or located by a downstream
reviewer.

**Severity rationale — HIGH, not MED:**
- The false attestation is in a changelog entry explicitly claiming "matches implementation."
- A reviewer trusting the changelog cannot verify the AC-002 oracle without independently
  grepping the impl.
- The false test name would prevent `just test -run TestSVTNManager_Destroy_KeyPurgeUnderConcurrentAdmission`
  from matching any test, silently passing as a vacuous run.
- Two separate impl symbols cover the two distinct obligations in AC-002; the story collapses
  them into one nonexistent symbol, obscuring the two-oracle structure.

**Fix locus:** AC-002 test annotation: replace `TestSVTNManager_Destroy_KeyPurgeUnderConcurrentAdmission`
with `TestSVTNManager_Destroy_KeyPurgePostcondition` (quiescent oracle) AND
`TestSVTNManager_Destroy_ConcurrentAdmissionIsRaceFree` (race oracle). Add a v1.5 RETRACTION
changelog entry. Architecture Compliance Rules table row for "Destroy purges all admitted keys
atomically under lock" must also be updated to cite the two actual test names.

### F-P4L2-MED-1 — AC-002 body wording conflates two distinct postcondition oracles into one assertion

**Confidence: HIGH.**

Even setting aside the test-name mis-anchor, the AC-002 body currently reads:
> "— asserts admission-blocking postcondition (the session-layer-observable consequence within
> the story's architectural boundary)."

This description is accurate for `TestSVTNManager_Destroy_ConcurrentAdmissionIsRaceFree` but
not for `TestSVTNManager_Destroy_KeyPurgePostcondition`. The purge-postcondition test asserts
that keys are absent from the set in quiescent state; the concurrent-admission test asserts
race-freedom. These are distinct properties with distinct failure modes.

A single AC-002 body that blends both into "admission-blocking postcondition" obscures which
test covers which obligation. When `TestSVTNManager_Destroy_KeyPurgePostcondition` fails, the
reviewer cannot tell from AC-002 alone whether it is the purge property or the race property
that broke.

**Severity rationale — MED:** The behavioral coverage is correct; both tests exist and cover
real obligations. The defect is in the wording clarity that distinguishes oracle type. No
implementation change required.

**Fix locus:** AC-002 body: add a parenthetical or sub-bullet distinguishing the
quiescent-purge oracle from the concurrent-race oracle, tied to each named test symbol.

## [process-gap] — Changelog false-attestation shield

The v1.5 changelog entry under `(F-M1)` asserted: "AC-002 test name corrected to
`TestSVTNManager_Destroy_KeyPurgeUnderConcurrentAdmission` (matches implementation)."
That symbol does not exist. The changelog is meant to be an audit trail, but a false
"matches implementation" claim in a changelog defeats its auditing purpose: future passes
that trust the changelog will not re-verify the impl-symbol link unless they perform
an independent grep.

**Recommendation:** Consider adding a lightweight rule (or pass-N-only axis) that every
changelog line claiming "corrected to match implementation" MUST be re-verified against the
impl at the next adversarial pass. This would catch this class of false attestation before
it propagates. Filed as GitHub issue against drbothen/vsdd-factory (POL-001-verify-attestations,
cross-ref #382, #396).
