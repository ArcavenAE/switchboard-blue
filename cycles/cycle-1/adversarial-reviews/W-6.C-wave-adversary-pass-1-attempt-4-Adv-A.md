**Evidence:**

- Grep `Detach|console` inside `svtnmgmt.go` returns only the one deferral comment line (770).
- `.factory/cycles/cycle-1/wave-schedule.md:175-177` states integration expectation ("SVTN destroy MUST cascade-detach any active console sessions attached to sessions inside the destroyed SVTN").
- `.factory/holdout-scenarios/wave-scenarios/wave-6.md` — grep for `cascade|destroy.*attach|attach.*destroy|SESSION-DRAIN|out.of.scope` returned zero hits; scenarios 1–8 exercise FEC + presence + console attach/switch independently but never assert a destroy→detach cascade.

**Finding:** None (Observation only — see §4-O1).

### Q2 — `--json` envelope consistency

**Evidence:**
- Single canonical writer path: `cmd/sbctl/main.go:95-107` (`writeSuccess`) and `main.go:111-123` (`writeError`) — same success shape (`{ok:true, error:null, data:...}`) and same error shape (`{ok:false, error:{code,message}, data:null}`) confirmed by `cmd/sbctl/client_test.go:1042-1118`.
- Both admin (`admin.go:262`, `admin.go:402`, `admin.go:446`, `admin.go:492`) and console (`console.go:102`, `console.go:120`, `console.go:151`) dispatch through the shared `connectAndRun(...)` client, which routes success/error to `writeSuccess`/`writeError` (`cmd/sbctl/client.go:378-391`).
- Post-RPC plain-text corruption risk explicitly gated on `!useJSON` at `cmd/sbctl/admin.go:268-273` for the destroy path; console handlers emit nothing post-`connectAndRun`, so no corruption vector.

**Finding:** None. Envelope shape is structurally identical across both story surfaces.

### Q3 — Confirm-gate reuse

**Evidence:**
- Grep `--confirm|--yes|E-CFG-006` in `cmd/sbctl/console.go` → zero hits.
- The five-path confirm-gate (`runDestroyConfirmGate`) at `admin.go:245-330` is specific to `admin svtn destroy` (destructive irreversible operation). `admin key revoke` has a distinct `--confirm` semantics (control-to-control revocation, ADR-004, `admin.go:418`).
- Console `attach`/`detach`/`switch` are non-destructive (per `console.go:107-108` — "Detach does not close the session"). No holdout scenario or BC requires a confirmation gate on console operations.

**Finding:** None. The confirm gate is correctly scoped to admin destroy; absence in console is expected, not divergent.

**Finding:** None. No collision detected between the two story surfaces.

### Q5 — Auth chain integration (ADR-004 + ADR-012)

**Evidence:**
- `cmd/switchboard/console_handlers.go:41` — `BuildConsoleHandlers(cs *session.ConsoleServer, ks *admission.AdmittedKeySet)` — console handlers accept the shared `AdmittedKeySet`.
- `cmd/switchboard/console_handlers.go:65-77` — `verifyConsoleCallerRole` checks `entry.Role != admission.RoleControl && entry.Role != admission.RoleConsole` before dispatching attach/detach/switch — inline authorization gate.
- `cmd/switchboard/main_test.go:132-138` — end-to-end challenge/response: `admission.GenerateChallenge` → `ed25519.Sign(nodePriv, ch.Nonce[:])` → `admission.ChallengeResponse{NonceSig: nonceSig}` → `admission.AdmitNode(ch, resp, nodePub, svtnID, ks)`. This is the ADR-012 challenge-response wired through the same admission surface that S-6.05 destroy uses via `resolveAndVerifyCallerRole` (`cmd/switchboard/admin_handlers.go:498`).
- The `mgmt.WithHandshakeTimeout(...)` decorator is applied identically to both admin and console mgmt sockets (`cmd/switchboard/metrics_wire_test.go:56`, `admin_handlers_e2e_test.go:77`).

**Finding:** None. S-7.03's console remote-control path integrates with the ADR-004 admission key registry and the ADR-012 handshake/challenge-response, using the same `admission.AdmittedKeySet` that S-6.05 destroy consults. No bypass.

## 4. Observations (non-blocking)

**O1 (LOW, informational).** Wave-schedule §"Cross-story integration surface" (`.factory/cycles/cycle-1/wave-schedule.md:175-177`) states as a MUST: *"SVTN destroy MUST cascade-detach any active console sessions attached to sessions inside the destroyed SVTN."* Implementation (`internal/svtnmgmt/svtnmgmt.go:770-771`) explicitly defers the cascade to `S-BL.SESSION-DRAIN`, citing story S-6.05 v1.5 AC-002 as out-of-scope. The holdout scenarios do NOT exercise this cascade (grep confirmed). Because the deferral is documented in-story and in-code, and the wave-6 holdout does not assert it as an acceptance criterion, this is a spec-vs-schedule tension rather than a spec-vs-code defect. Recommend the wave-schedule text be softened (e.g., "MUST eventually cascade..." with pointer to S-BL.SESSION-DRAIN) at the next schedule edit to avoid future L1 reviewers flagging it as a discrepancy. Not blocking for W-6.C convergence.
