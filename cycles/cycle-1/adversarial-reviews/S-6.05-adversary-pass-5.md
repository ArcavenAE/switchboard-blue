### F-P5L1-MED-1 — Confirm-Gate Semantics section contradicts §125 interactive-mode canonical rule

**Confidence: HIGH.**

The Confirm-Gate Semantics section of story v1.7 (lines 141–152) describes two paths:

1. Flag-supplied: `--confirm=SVTN-<8-hex>` static shape check before RPC dispatch.
2. `--yes` bypass with stderr warning (cannot combine with `--confirm`; E-CFG-006 exit 2).

The section makes no reference to an interactive-mode path when `--confirm` is omitted.

AC-003 body states:
> "sbctl admin svtn destroy --name=<svtn-name> --confirm=<svtn-short-id>…"
which correctly cites §117 (the table row showing `--confirm` as required in the
flag-supplied path). However the phrase "requires explicit confirmation" in AC-003
is ambiguous: it could be read as "exit non-zero when confirmation is absent" (static-only
abort model, matching the Confirm-Gate Semantics section) or as "require a user
confirmation gesture which may be provided interactively" (three-path model per §125).

`interface-definitions.md §125` (lines 125–129) is unambiguous:
> "**`--confirm=<svtn-short-id>`** — When the flag is omitted, the command enters
> interactive mode and prompts `Type SVTN-<short-id> to confirm:` before proceeding."
§129 (confirmation flow summary) reinforces this three-path model.

The Confirm-Gate Semantics section (lines 141–152) does NOT describe the interactive path.
It describes only: (a) flag-supplied static check, and (b) `--yes` bypass. This is
inconsistent with §125/§129.

Inspection of the implementation at `cmd/sbctl/admin.go` (cc78688) confirms the
Confirm-Gate Semantics section describes the as-built behavior: the `--confirm` flag is
required; omitting it exits with a static error rather than entering interactive mode.
The implementation does NOT implement the §125 interactive-mode fallback.

**Adjudication — spec is authoritative; impl is non-compliant:**

`interface-definitions.md §125` is the normative canonical rule (L2 Domain Spec).
BC-2.07.001 and S-6.05 are downstream artifacts that must conform to it.
The Confirm-Gate Semantics section as written describes the non-compliant impl behavior
and presents it as the spec. The spec-drift is therefore:

- **Non-compliant artifact:** `cmd/sbctl/admin.go` (interactive-mode path not implemented)
- **Non-compliant artifact:** tests asserting static-only abort on absent `--confirm`
- **Defective spec prose:** Confirm-Gate Semantics section, which omits Path 2 (interactive mode)

The spec-side fix (this burst) is to rewrite the Confirm-Gate Semantics section to describe
the §125/§129 canonical three-path flow, add a SPEC-IMPL DRIFT block documenting that
the impl deviates, and retain Task 11 as UNFULFILLED. The impl/test correction is a
separate implementer fix-burst and must not be made in this spec-only burst.

**Fix locus (spec-side):**
- Rewrite Confirm-Gate Semantics to describe the three-path flow:
  1. Flag-supplied: `--confirm=SVTN-<8-hex>` static shape check.
  2. Interactive-mode (flag omitted): prompts `Type SVTN-<short-id> to confirm:` per §125.
  3. `--yes` bypass: stderr warning, cannot combine with `--confirm` (E-CFG-006, exit 2).
- Add `> [!WARNING] SPEC-IMPL DRIFT (F-P5L1-MED-1)` block: implementation does not
  implement interactive-mode fallback; `--confirm` currently required; Task 11 annotated
  UNFULFILLED; fix tracked in drbothen/vsdd-factory#430 (POL-005 candidate).
- Update AC-003 cite to `§117 and §125` (not §117 alone).
- Task 11: retain, annotate as UNFULFILLED with note that `--yes` flag is also unimplemented.
- Pass-6 will remain NOT CONVERGENT until impl and tests conform to §125.
