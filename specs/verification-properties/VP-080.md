# VP-080: Router-Side Discovery Ingest Discards Non-Increasing Sequence Even After HMAC Passes

## Property Statement

∀ sequence of `PRESENCE_ADV` datagrams received by the router-side
discovery ingest path (`internal/discovery`, router-mediated relay per
`ARCH-03` §"Session Discovery" v1.7) for a single `(SVTNID, NodeAddr)`
pair, each individually HMAC-valid (i.e. `router.DiscoveryAuthKeyFor`
lookup succeeds and HMAC verification passes against the resulting
`DiscoveryAuthKey`):

1. **Cold-start acceptance.** The first datagram received for a given
   `(SVTNID, NodeAddr)` pair since router start (or since the router has
   no prior last-seen `Sequence` recorded for that pair — including
   immediately after that node's admission) is accepted regardless of
   its declared `Sequence` value.
2. **Replay/stale discard.** Any subsequent datagram for the same
   `(SVTNID, NodeAddr)` pair whose declared `Sequence` is not strictly
   greater than the last-accepted `Sequence` recorded for that pair is
   discarded: the discovery registry is NOT updated for that datagram,
   and it is NOT relayed to other admitted nodes on the SVTN — even
   though its HMAC verifies. Exact byte-for-byte replay of a
   previously-accepted datagram is the degenerate case of this property
   (`Sequence == last-accepted`).
3. **Forward acceptance.** Any subsequent datagram for the same
   `(SVTNID, NodeAddr)` pair whose declared `Sequence` IS strictly
   greater than the last-accepted `Sequence` is accepted: the registry
   is updated per the datagram's content, the datagram is relayed, and
   the router's recorded last-accepted `Sequence` for that pair advances
   to the new value.
4. Properties 1–3 hold identically regardless of the specific numeric
   `Sequence` values involved — only relative ordering matters.
   `uint32` wraparound behavior is explicitly OUT OF SCOPE for this
   property and this story (see rulings doc Forward Obligation); no
   wraparound case is asserted here.

This property enforces the SEC-DW-07 adjudication in
`S-BL.DISCOVERY-WIRE-rulings.md` v1.1 (Ruling 1, "Replay / freshness"
subsection) and the BC-2.03.001 Postcondition 2 amendment drafted there
for product-owner execution (new `sequence` field + router-side discard
rule). It closes the residual this story would otherwise introduce: an
attacker who passively captures one valid, HMAC-authenticated
advertisement (no admission bypass or router compromise required —
multicast traffic is observable on the local segment) can currently
replay it indefinitely, defeating BC-2.03.002 Postcondition 5's
staleness-expiry guarantee.

> **Replay-resistance field (SEC-DW-07, Ruling S-BL.DISCOVERY-WIRE-1):**
> each advertisement additionally includes a monotonic `sequence` value,
> unique-and-increasing per (access node, SVTN), incremented on every
> outbound advertisement (state-change or heartbeat-triggered). The
> router discards any HMAC-verified advertisement whose `sequence` is
> not strictly greater than the last-accepted value for that (SVTN,
> node) pair, even though HMAC passed — closing the replay window that
> would otherwise let a captured, still-valid frame be re-injected
> indefinitely and defeat Postcondition 5's staleness-expiry guarantee.
> Cold-start (router restart, or first frame from a newly-admitted node)
> accepts unconditionally, bounding the residual replay window to at
> most one heartbeat interval — the same bounded-not-perfect posture
> this project already accepts for admission-layer nonce replay
> (`nonceTTL=60s`).

- **Ruling:** `S-BL.DISCOVERY-WIRE-rulings.md` v1.1, Ruling 1 §"Replay /
  freshness (SEC-DW-07, MED, CWE-294 — my adjudication, flagged for the
  human gate)" — full finding, DI-003 analysis, and cost/benefit
  rationale for adding this field rather than accepting the residual.
- **Architecture:** `ARCH-03-routing-engine.md` v1.7 §"Session
  Discovery" — router-only multicast membership; the router is the sole
  point where this discard decision can be enforced (access/console
  nodes never receive advertisements directly from one another, so only
  the router-side ingest path is a meaningful replay-defense boundary).
- **Adjacent BC:** BC-2.03.002 Postcondition 5 — "sessions no longer
  advertised... do not appear after the next heartbeat cycle" — the
  postcondition this property protects from indefinite defeat via replay.

## Proof Method

| Method | Tool | Bounded? | Coverage |
|--------|------|----------|----------|
| integration | `go test` against the router + discovery ingest test harness (extends the `TestDiscovery_VP045_SVTNIsolation_MultipleScopes` family, `internal/discovery/discovery_test.go`) | yes | Four sub-cases: cold-start acceptance, replay/stale discard (including exact-replay degenerate case), forward acceptance, and no-relay-on-discard side-effect assertion |

**Method justification.** This property is fundamentally about
wiring-correctness across two packages (`internal/discovery`'s router-side
ingest path and `internal/routing`'s `DiscoveryAuthKeyFor` HMAC-key
lookup), and about ordered, stateful behavior across multiple received
datagrams over time — not a single pure-function call. VP-009 (admission
nonce replay) used `proptest` because `admission.NonceStore` is a
standalone in-memory component testable in isolation; no equivalent
isolated component is committed to for the discovery replay-discard map
yet (implementer's choice of concrete data structure is unconstrained by
this ruling — see rulings doc Implementation Constraints). Integration
testing against the actual router+discovery ingest pipeline is the
proof method that is falsifiable at design time without presupposing
that shape, and it additionally proves the property holds through the
real HMAC-verify → sequence-check → registry-update ordering, not just
an isolated discard-decision function. This mirrors VP-078's integration
classification for another cross-cutting wiring property.

### Test Scenarios

1. **Cold-start acceptance:** router has no prior last-seen `Sequence`
   for `(SVTNID, NodeAddr)` (fresh router state, or immediately following
   that node's admission). Send one HMAC-valid advertisement with an
   arbitrary `Sequence` value (including 0). Assert: accepted, registry
   updated, relay attempted.
2. **Replay/stale discard:** having established a last-accepted
   `Sequence = N` via scenario 1, send a second HMAC-valid advertisement
   for the same `(SVTNID, NodeAddr)` declaring `Sequence = N` (exact
   replay — same bytes retransmitted) and, separately, `Sequence < N`.
   Assert both are discarded: registry NOT updated (still reflects the
   `N`-sequence content), no relay side effect observed, last-accepted
   `Sequence` state unchanged.
3. **Forward acceptance:** from the same established state
   (`last-accepted = N`), send an HMAC-valid advertisement with
   `Sequence = N+1`. Assert: accepted, registry updated to the new
   content, relay attempted, last-accepted `Sequence` advances to `N+1`.
4. **No-relay-on-discard:** re-assert scenario 2's relay non-occurrence
   explicitly via a relay-call spy (fails the test if invoked for the
   discarded datagram) — distinguishing "silently dropped" from
   "accepted but not yet relayed due to an unrelated fault."
