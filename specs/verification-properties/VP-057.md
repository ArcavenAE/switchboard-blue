# VP-057: Node Private Keys Never Appear as Literal Bytes in Any Emitted Frame

## Property Statement

Across all frame types and all admission/control message types: no field,
header, or payload byte of any frame ever contains the private half of a
node's admission keypair as a literal byte sequence.

The property has two parts:

**Part (i) — proptest sampling:** For arbitrary node admission keypairs and
arbitrary frame types, the raw bytes of every emitted frame do not contain
the 32-byte (Ed25519) or 64-byte (Ed25519 with seed) private key as a
contiguous literal subsequence.

∀ NodeKey K (Ed25519), ∀ FrameType T ∈ {DATA, EMPTY_TICK, ADMISSION_CHALLENGE,
  ADMISSION_RESPONSE, CONTROL_DRAIN, CONTROL_KEY_REG, CONTROL_KEY_REVOKE}:
  let frame = Emit(K, T, testPayload)
  assert !bytes.Contains(frame.Bytes(), K.PrivateKey().Bytes())
  assert !bytes.Contains(frame.Bytes(), K.Seed())        // first 32 bytes of private key

**Part (ii) — inline proof sketch (HKDF non-exposure):**

The HMAC key for frame authentication is derived per ARCH-04 §"HMAC keying"
(ADR-001, lines 167–170). Canonical derivation:

```
HKDF-Extract(salt=svtn_id, ikm=node_admission_pubkey) → PRK
HKDF-Expand(PRK, info="switchboard-frame-auth", length=32) → frame_auth_key
```
Source: ARCH-04 §"HMAC keying" (ADR-001, lines 167–170).

HKDF is a one-way PRF; the public key is the IKM (input keying material).
The private key is never an input to HKDF or HMAC. The HMAC tag therefore
cannot reveal the private key: (a) the private key is not an input to the
derivation; (b) HKDF output is computationally indistinguishable from random
given only the public key and the output. This is documented in ADR-001.

**Limitation note:** Exhaustive enumeration over all possible frame byte
sequences is infeasible. The proptest samples across all 7 frame types with
100 keypair draws (gopter default). This provides coverage against accidental
serialization of the private key in any known code path. It does not provide
a formal proof of absence in adversarial or unknown code paths; that guarantee
comes from code review enforced by postconditions in BC-2.05.007 and ADR-001.

## Rationale

VP-007 (`Admission: private key never in wire structs`) verifies that the
private key is not present in the admission-specific wire structs at the
type level (i.e., the `AdmissionChallengeResponse` struct does not have a
field of type `PrivateKey`). This is a structural check.

VP-057 adds a runtime sampling check: after encoding any frame through the
full `internal/admission` code path (keypair → sign → frame bytes), the
resulting byte slice does not contain the private key as a literal byte
sequence. This catches accidental serialization bugs (e.g., a logging path
that writes `fmt.Sprintf("%v", keypair)` to a frame payload) that a
structural check would miss.

BC-2.05.007 postcondition 1 ("The private key bytes are not present in any
outgoing network frame") is the direct target of this VP.

## Proof Harness Skeleton

```go
package admission_test

import (
	"bytes"
	"crypto/ed25519"
	"crypto/rand"
	"testing"

	"github.com/leanovate/gopter"
	"github.com/leanovate/gopter/prop"

	"github.com/arcavenae/switchboard/internal/admission"
	"github.com/arcavenae/switchboard/internal/frame"
)

// allFrameTypes lists every frame type the admission package may emit.
var allFrameTypes = []frame.Type{
	frame.TypeData,
	frame.TypeEmptyTick,
	frame.TypeAdmissionChallenge,
	frame.TypeAdmissionResponse,
	frame.TypeControlDrain,
	frame.TypeControlKeyReg,
	frame.TypeControlKeyRevoke,
}

// TestPropPrivateKeyNotInFrameBytes verifies that the private key bytes do not
// appear as a literal contiguous subsequence in any emitted frame's bytes.
//
// Limitation: this is a sampling test (100 keypairs × 7 frame types = 700
// samples). It is not an exhaustive proof. The complementary proof that the
// HMAC tag cannot reveal the private key is provided in the inline sketch below
// and documented in ADR-001.
func TestPropPrivateKeyNotInFrameBytes(t *testing.T) {
	properties := gopter.NewProperties(nil)

	// genKeyPair generates a fresh Ed25519 keypair for each sample.
	genKeyPair := gopter.CombineGens().FlatMap(
		func(_ interface{}) gopter.Gen {
			return func(genParams *gopter.GenParameters) *gopter.GenResult {
				pub, priv, err := ed25519.GenerateKey(rand.Reader)
				if err != nil {
					panic(err)
				}
				return gopter.NewGenResult(admission.KeyPair{Public: pub, Private: priv}, gopter.NoShrink)
			}
		},
		gopter.TypeOf(admission.KeyPair{}),
	)

	properties.Property(
		"private key bytes not present as literal subsequence in any emitted frame",
		prop.ForAll(
			func(kp admission.KeyPair) bool {
				for _, ft := range allFrameTypes {
					f, err := admission.EmitFrame(kp, ft, []byte("test-payload"))
					if err != nil {
						// Emission error is not a VP failure; log and skip this type.
						continue
					}
					frameBytes := f.Bytes()

					// Check for the raw private key (64 bytes: seed + public key).
					if bytes.Contains(frameBytes, []byte(kp.Private)) {
						return false
					}
					// Also check for the seed alone (first 32 bytes).
					seed := []byte(kp.Private)[:32]
					if bytes.Contains(frameBytes, seed) {
						return false
					}
				}
				return true
			},
			genKeyPair,
		),
	)

	properties.TestingRun(t)
}

// InlineProofSketch_HKDFDoesNotExposePrivateKey documents why the HMAC tag
// cannot reveal the private key.
//
// The frame authentication key is derived per ARCH-04 §"HMAC keying" (ADR-001):
//
//   HKDF-Extract(salt=svtn_id, ikm=node_admission_pubkey) → PRK
//   HKDF-Expand(PRK, info="switchboard-frame-auth", length=32) → frame_auth_key
//   hmac_tag = HMAC-SHA256(frame_auth_key, frame_bytes_without_tag)
//
// Arguments for non-exposure:
// 1. The private key is NEVER an input to HKDF or HMAC.
//    The IKM is the PUBLIC key. Code: internal/hmac/derive.go:DeriveFrameAuthKey.
// 2. HKDF-SHA256 is a one-way PRF: given (output, salt, info), recovering IKM
//    requires inverting HMAC-SHA256, which is computationally infeasible.
// 3. HMAC tags are message authentication codes. Given (tag, key, message),
//    recovering the key requires inverting HMAC-SHA256 again.
// 4. An observer who sees the frame bytes and HMAC tag can compute:
//      frame_auth_key ← brute force from HMAC (infeasible)
//      pubkey         ← brute force from HKDF   (infeasible)
//      privkey        ← brute force from pubkey (infeasible for Ed25519)
//
// This argument is documented in ADR-001 ("HMAC trust model") and referenced
// by BC-2.05.007 invariant 1 (DI-002).
//
// This function is intentionally unexported and never called — it exists as
// a documented proof sketch in the test file.
func inlineProofSketch_HKDFDoesNotExposePrivateKey() {} //nolint:unused // proof sketch
```
