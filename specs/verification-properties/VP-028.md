# VP-028: Config.Validate Rejects Out-of-Range tick_interval

## Property Statement

∀ duration d:
  d < 5ms ∨ d > 50ms
  ⟹ (&Config{ListenAddr: <valid>, TickInterval: d}).Validate() returns a non-nil error
      ∧ the error carries code E-CFG-001 (internal/config/config.go:43)
      ∧ the error message names field 'tick_interval' (internal/config/config.go:210)

A tick interval outside the allowed range [5ms, 50ms] is a configuration
error that must be surfaced explicitly by field name, not silently clamped
or ignored. Zero is treated as missing/invalid per BC-2.09.003 EC-002
(internal/config/config.go:203–204).

## Source Contract

- **BC:** BC-2.09.003 — Router startup fails cleanly on malformed config with actionable error message
- **Postcondition/Invariant (derived):** BC-2.09.003 Postcondition 1 — "The daemon exits with a non-zero exit code before accepting any connections" when given a malformed config. This VP tests a specific malformed-config subcase: when `tick_interval` is outside [5ms, 50ms], `(*Config).Validate()` must return a non-nil error with code E-CFG-001 naming field `tick_interval`. The out-of-range tick_interval bound is config-validation territory per ASM-001 (tick interval bounds are operationally defined constraints in ADR-008). This VP is a DERIVED property: it specifies what "malformed config → exit non-zero" means concretely for the tick_interval field, not a postcondition literally stated in BC-2.09.003.

## Proof Method

| Method | Tool | Bounded? | Coverage |
|--------|------|----------|----------|
| table-driven | standard testing.T | yes | enumerated out-of-range durations: {-1ms, 0, 1µs, 4ms999µs, 50ms1µs, 100ms, 1s}; asserts Validate returns non-nil error with E-CFG-001 code and "tick_interval" in message |

**Justification for table-driven over proptest:** The out-of-range domain is two bounded intervals (d < 5ms, d > 50ms) with a finite set of structurally distinct cases: zero (missing), sub-millisecond, just-below-min, just-above-max, coarsely-out-of-range, and maximally-large. The existing test `TestVP028_AnyValidationErrorIsECFG001` (internal/config/config_test.go:572) exhaustively covers these cases as a Cartesian product. A property sweep over random durations would only increase execution time without discovering new failure modes that the enumerated cases miss. Table-driven is more appropriate here; proptest would add value only if the bounds were derived from a formula rather than constants.

## Proof Harness Skeleton

```go
package config_test

import (
	"errors"
	"testing"
	"time"

	"github.com/arcavenae/switchboard/internal/config"
)

func TestVP028_OutOfRangeTickIntervalReturnsECFG001(t *testing.T) {
	t.Parallel()

	cases := []struct {
		name     string
		duration time.Duration
	}{
		{"negative", -1 * time.Millisecond},
		{"zero_missing", 0},
		{"sub_microsecond", 1 * time.Microsecond},
		{"just_below_min", 4*time.Millisecond + 999*time.Microsecond},
		{"just_above_max", 50*time.Millisecond + 1*time.Microsecond},
		{"coarsely_above", 100 * time.Millisecond},
		{"one_second", 1 * time.Second},
	}

	for _, tc := range cases {
		tc := tc
		t.Run(tc.name, func(t *testing.T) {
			t.Parallel()
			cfg := &config.Config{
				ListenAddr:   "0.0.0.0:9090",
				TickInterval: tc.duration,
			}
			err := cfg.Validate()
			if err == nil {
				t.Fatalf("expected validation error for tick_interval=%v, got nil", tc.duration)
			}
			var ce *config.ConfigError
			if !errors.As(err, &ce) || ce.Code != "E-CFG-001" {
				t.Fatalf("expected E-CFG-001, got %v", err)
			}
			if !strings.Contains(err.Error(), "tick_interval") {
				t.Fatalf("expected field name 'tick_interval' in error, got: %q", err.Error())
			}
		})
	}
}
```

- `TestValidate_OutOfRange` (internal/config/config_test.go:84): asserts that error messages for out-of-range tick_interval contain "tick_interval" and the invalid value — covers the field-name-in-message clause.
