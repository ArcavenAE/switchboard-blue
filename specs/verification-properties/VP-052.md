# VP-052: N Consecutive Missing Frames Trigger Quality Indicator Downgrade

## Property Statement

∀ QualityIndicator in state Green:
  N=HysteresisCount consecutive calls to OnMissingFrame()
    → Current() transitions from Green to Yellow

∀ QualityIndicator in state Yellow:
  N=HysteresisCount consecutive calls to OnMissingFrame()
    → Current() transitions from Yellow to Red

Downgrade is one level at a time; a single OnMissingFrame call (below the threshold) must not
downgrade. A successful Update() call resets the consecutive-miss counter so a fresh streak
of N is required before the next downgrade.

Deadline timing (was-tick-overdue? logic) is a **caller concern**: the caller decides when
to invoke OnMissingFrame(); internal/metrics is clock-free and remains pure-core. This
design preserves the purity boundary (no Clock injection, no time.Now() inside the package).

## Rationale

BC-2.06.002 requires the quality monitor to detect missing frames and trigger an
indicator downgrade. VP-027 tests monotonicity of degradation transitions (quality
never upgrades spontaneously) but does not inject missing-frame events. BC-2.06.002's
core claim — that a missed tick causes a downgrade — is not exercised by VP-027.

Empty-tick frames are the liveness signal of the protocol (BC-2.01.002). Their
absence is the primary signal for path failure. Without VP-052, an implementation
that reacts only to explicit TLPKTDROP events and never to missed ticks would pass
VP-027 while violating BC-2.06.002.

The 3-measurement hysteresis (canonical value from BC-2.01.002 EC-001 and BC-2.06.001
invariant 3) applies: 3 consecutive missed ticks trigger the downgrade, not 1.

**Option (a) rationale (H-002):** The original skeleton required unbuilt API
(`NewMonitor`, `CheckDeadline`, `RecordTick`, `Clock`) that would require
Clock injection into internal/metrics, violating the pure-core purity boundary.
The as-built implementation exposes `OnMissingFrame()` — the caller signals a
missing frame after performing its own deadline check. This keeps internal/metrics
deterministic and clock-free. VP-052 is fully exercised by the as-built
`metrics_test.go` tests (TestQualityIndicator_MissingFrameDowngrade*) and the
proptest in `metrics_prop_test.go`.

## Source Contract

- **BC:** BC-2.06.002 — Missing expected frame is a degradation signal triggering indicator downgrade
- **Core claim:** "The quality monitor detects missing frames and degrades the
  quality indicator."
- **BC-2.01.002 EC-001:** "After threshold (implementation: ≥3 consecutive missed
  ticks), quality indicator degrades."
- **ARCH-03 thresholds:** Green→Yellow at RTT >100ms OR loss >5%; Yellow→Red at
  RTT >500ms OR loss >20%.

## Proof Harness Skeleton

The as-built implementation in `internal/metrics/metrics_test.go` fully covers
this property. The skeleton below mirrors the delivered tests for completeness.
No `FakeClock`, `NewMonitor`, `CheckDeadline`, or `RecordTick` — those would
require Clock injection into internal/metrics and are not part of the pure-core API.

```go
package metrics_test

import (
	"testing"

	"github.com/arcavenae/switchboard/internal/metrics"
)

// TestMissingFrame_GreenToYellow verifies that HysteresisCount consecutive
// OnMissingFrame calls degrade the indicator from Green to Yellow.
//
// BC-2.06.002 PC-2; VP-052 canonical scenario.
func TestMissingFrame_GreenToYellow(t *testing.T) {
	t.Parallel()

	n := metrics.HysteresisCount
	qi := metrics.NewQualityIndicator()

	// Establish Green: feed n good measurements.
	for i := 0; i < n; i++ {
		qi.Update(50, 1)
	}
	if qi.Current() != metrics.Green {
		t.Fatalf("prerequisite: expected Green, got %s", qi.Current())
	}

	// Fewer than n consecutive missing frames must not downgrade.
	for i := 0; i < n-1; i++ {
		qi.OnMissingFrame()
		if qi.Current() != metrics.Green {
			t.Errorf("after %d missing frame(s): got %s, want Green", i+1, qi.Current())
		}
	}

	// The n-th consecutive missing frame must downgrade Green → Yellow.
	qi.OnMissingFrame()
	if qi.Current() != metrics.Yellow {
		t.Errorf("after %d consecutive missing frames: got %s, want Yellow", n, qi.Current())
	}
}

// TestMissingFrame_YellowToRed verifies that HysteresisCount consecutive
// OnMissingFrame calls in Yellow state degrade the indicator to Red.
//
// BC-2.06.002 PC-2; VP-052 Yellow→Red scenario.
func TestMissingFrame_YellowToRed(t *testing.T) {
	t.Parallel()

	n := metrics.HysteresisCount
	qi := metrics.NewQualityIndicator()

	// Drive to Yellow via n missing frames.
	for i := 0; i < n; i++ {
		qi.OnMissingFrame()
	}
	if qi.Current() != metrics.Yellow {
		t.Fatalf("prerequisite: expected Yellow after %d missing frames, got %s", n, qi.Current())
	}

	// Fewer than n more consecutive missing frames must not downgrade to Red.
	for i := 0; i < n-1; i++ {
		qi.OnMissingFrame()
		if qi.Current() != metrics.Yellow {
			t.Errorf("after %d more missing frame(s): got %s, want Yellow", i+1, qi.Current())
		}
	}

	// The n-th consecutive missing frame must downgrade Yellow → Red.
	qi.OnMissingFrame()
	if qi.Current() != metrics.Red {
		t.Errorf("after %d more consecutive missing frames: got %s, want Red", n, qi.Current())
	}
}

// TestMissingFrame_UpdateResetsStreak verifies that a successful Update resets
// the consecutive-miss counter so a fresh streak of N is required for downgrade.
//
// BC-2.06.002 PC-4; VP-052 streak-reset scenario.
func TestMissingFrame_UpdateResetsStreak(t *testing.T) {
	t.Parallel()

	n := metrics.HysteresisCount
	qi := metrics.NewQualityIndicator()

	// Feed n-1 missing frames (streak in progress, below threshold).
	for i := 0; i < n-1; i++ {
		qi.OnMissingFrame()
	}

	// A good Update must reset the missing-frame counter.
	qi.Update(50, 1)
	if qi.Current() != metrics.Green {
		t.Fatalf("after Update: expected Green, got %s", qi.Current())
	}

	// Feed n-1 more missing frames — must still not downgrade (streak was reset).
	for i := 0; i < n-1; i++ {
		qi.OnMissingFrame()
	}
	if qi.Current() != metrics.Green {
		t.Errorf("n-1 missing frames after streak reset must not downgrade: got %s", qi.Current())
	}
}
```
