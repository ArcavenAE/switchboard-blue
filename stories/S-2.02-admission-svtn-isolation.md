# S-2.02: Tier-1 Admission and SVTN Isolation

> **Execute:** `/vsdd-factory:deliver-story S-2.02`

## Narrative

- **As a** router
- **I want to** admit only cryptographically verified nodes via signed challenge, partition all traffic by SVTN, and ensure private keys never appear on the wire
- **So that** unauthorized nodes cannot inject frames and nodes on SVTN-A cannot observe SVTN-B traffic

## Acceptance Criteria

### AC-001 (traces to BC-2.05.001 postcondition 1)
`AdmitNode(challenge, signed_response, public_key, svtn_id)` succeeds when the signature over the challenge is valid for the presented public key.
- **Test:** `TestAdmitNode_ValidChallenge`

### AC-002 (traces to BC-2.05.001 postcondition 5)
`AdmitNode` returns `E-ADM-001` when the signature over the challenge is invalid.
- **Test:** `TestAdmitNode_InvalidSignature`

### AC-003 (traces to BC-2.05.001 invariant 3)
`AdmitNode` returns `E-ADM-008` when the challenge nonce has already been used (replay prevention).
- **Test:** `TestAdmitNode_ReplayedNonce`

### AC-004 (traces to BC-2.05.002 postcondition 2)
`RouteFrame(frame, admitted_set)` drops the frame and returns `E-ADM-003` if the frame's `src_addr` is not in the admitted set for the frame's `svtn_id`.
- **Test:** `TestRouteFrame_DropsUnadmitted`

### AC-005 (traces to BC-2.05.006 postcondition 1)
`SVTNRoute` never delivers a frame to a node on a different SVTN: a frame with `svtn_id=A` is never forwarded to a node admitted only to `svtn_id=B`.
- **Test:** `TestSVTNRoute_NoCrossContamination`

### AC-006 (traces to BC-2.05.007 invariant 1)
No wire struct or log output produced by `internal/admission` contains private key bytes. Property test verifies that no field in the admission challenge or response structs encodes private key material.
- **Test:** `TestAdmission_PrivateKeyAbsentFromWireStructs`

### AC-007 (traces to BC-2.05.007 postcondition 1)
`GenerateChallenge()` produces a nonce without using or transmitting the node's private key.
- **Test:** `TestGenerateChallenge_NoChallengeContainsPrivateKey`

## Edge Cases

| ID | Scenario | Expected Behavior |
|----|----------|-------------------|
| EC-001 | Node presents key for SVTN it is not registered on | E-ADM-003 returned; frame dropped |
| EC-002 | Duplicate key registration | Last-write-wins per ADR-003; existing entry replaced |
| EC-003 | Admission of revoked key | E-ADM-005 returned |
| EC-004 | LWW re-registration of previously admitted key | Re-registration resets `admitted=false`. The node MUST complete a fresh challenge-response handshake (AdmitNode) before it appears in the active admitted set. This is security-by-default: any key change forces re-authentication. Reset semantic per ADR-003 (LWW) + BC-2.05.001 PC4. |
