# S-7.04-FU-DRAIN-WIRE: DRAIN-over-SVTN Wire Propagation

## Narrative

- **As an** operator gracefully shutting down a PE router
- **I want** connected nodes to receive a DRAIN signal over their SVTN channels before
  the router disconnects
- **So that** nodes can migrate active sessions to alternate routers within the drain
  window, rather than losing them abruptly

## Context

S-7.04 (merged PR #101/#103) delivered `internal/drain.Drain` with `RegisterObserver`,
`Signal`, `Wait`, and the `drainCoordHook` test-injection mechanism. The coordinator's
liveness model (fan-out-then-race under a bounded child context, EC-003 ErrTimeout) is
complete and tested. However, `runRouter` currently has zero registered observers: no
per-node connection identity model exists at the DRAIN observer call site.

S-BL.PE-RECEIVE-LOOP (merged PR #118 @ `e940fc2`, 2026-07-11) delivered the per-connection
receive goroutine, `upstreamdial.Connector.SetFrameCallback`, and the
`FrameArrivalHandler.OnFrameArrival` wiring. It also emitted FO-RECV-FWD-001: the nil
`ForwardFunc` at `mgmt_wire.go`:549 must be replaced when the interface set widens.

**v1.1 clarification (Q2-AMENDED):** FO-RECV-FWD-001 is NOT consumed by this story. The
nil `ForwardFunc` stays; the `interfaceSet` stays `[]{peIfaceID}`. DRAIN is a
router-originated broadcast dispatched directly from the drain observer closure to per-node
send channels ‚Äî it does NOT route through `OnFrameArrival`. The E-FWD-001 invariant is
preserved. FO-RECV-FWD-001 remains deferred to the future data-frame-widening story.

This story lands three deliverables:

1. **DRAIN frame on the wire** ‚Äî A single startup drain observer (registered once at
   `runRouter` setup) iterates the live per-node send map at `Signal` time. For each
   connected node, it assembles a `FrameTypeCtl` (0x03) frame with a 4-byte payload
   (`control_type = 0x01`, DRAIN opcode) and non-blocking sends it to the node's send
   channel. The per-node send map is a `sync.Map` owned by `runRouter`; entries are
   added via the `netingress.ServeConfig.OnAccept` hook and removed on conn close.

2. **netingress OnAccept seam** ‚Äî `netingress.Serve` gains a `ServeConfig` parameter
   carrying an `OnAccept func(conn net.Conn, h NodeHandle) func()` hook. `runRouter`
   supplies the hook to allocate `routing.InterfaceID`, create the per-node send channel,
   start the writer goroutine, and register the channel in the send map. Cleanup
   (channel close + map remove) is deferred on conn close. This is the binding Q-SEAM
   design.

3. **Per-observer panic recovery and EC-003 production-path evidence** ‚Äî `drain.Signal`'s
   fan-out goroutine currently has no `defer recover()`. This story adds the recovery
   wrapper. EC-003 is evidenced via `drainCoordHook` + blocked observer +
   `cfg.DrainTimeout=100ms` (the real drain-timeout injection seam per Q-AC005).

## Design Constraints

### DRAIN Frame Wire Schema (Q1 ‚Äî binding)

**Binding (per placement note Q1).**

The DRAIN signal is carried as a `FrameTypeCtl = 0x03` frame (existing constant, verified
`internal/frame/frame.go` at `e940fc2`). No new `FrameType` constant is introduced. The
DRAIN payload schema is a fixed 4-byte control message (BC-2.01.008 v1.0, schema home):

```
byte[0]   control_type   = 0x01   // DRAIN opcode
byte[1]   version        = 0x01   // protocol version; allows future extension
byte[2:4] reserved       = 0x0000 // zero-filled; receiver MUST ignore
```

Rationale: `FrameTypeCtl` is the existing discriminator for authenticated control frames.
Adding `FrameTypeDrain = 0x07` would require widening `Valid()` past `FrameTypePEConnect =
0x06` ‚Äî the guard S-BL.PE-RECEIVE-LOOP erected. Using `FrameTypeCtl` with a
`control_type` payload byte is the correct design: outer header identifies the wire
family (`ctl`); the control-message opcode lives in the payload. BC-2.01.008 is the
authoritative schema home; BC-2.01.004 Inv-2 carries the terminal-consumer carve-out
cross-reference.

**Assembly path:** The observer closure constructs the 4-byte payload and calls
`frame.EncodeOuterHeader` + `append(ehdr[:], payload...)`. The `Envelope` fields
(SVTN, SrcAddr, DstAddr, FrameAuthKey) are sourced from the per-node connection context
stored in the `runRouter` connection registry using the zero-envelope pattern identical to
S-7.04-FU-PE-CONNECTOR (FO-DRAIN-WIRE-002 notes that full Ed25519 bootstrap is a
session-bootstrap follow-on). `frame.EncodeOuterHeader` is an EXISTING function at
`e940fc2` (not new ‚Äî defined by this story).

**Wire-format spec-pair obligation (same-commit ‚Äî binding):** This story introduces the
`control_type` discriminator byte in the `FrameTypeCtl` payload schema. The following
same-commit amendments are required:

1. ARCH-02 ¬ß "Outer Header Format" (NOT ¬ß3.1 ‚Äî ARCH-02 has no ¬ß3.1, corrected v1.2,
   F-DW-SP2-009) frame_type table: add a note that `ctl (0x03)` payloads carry a
   `control_type` byte discriminator; enumerate opcode `0x01 = DRAIN`.
2. BC-2.01.004 Postcondition 2 (outer-header layout): add the same `control_type` note
   under the `frame_type = ctl` row (with cross-ref to BC-2.01.008 as schema home).

These are a NARROWER obligation than the FrameTypePEConnect case (F-SP14-001) ‚Äî the
frame_type byte itself is unchanged (0x03 already listed); only the payload schema
documentation is new. The same-commit rule (F-P8-008) still applies.

**Forward-compatibility silent-ignore rule (FO-DRAIN-WIRE-001, binding; anchors to
BC-2.01.008 Postcondition 4; guard home firmed v1.2, F-DW-SP2-003):** Any `FrameTypeCtl`
frame arriving at the netingress `route` RouteFn closure (`mgmt_wire.go`, the inbound
per-node path) with an unrecognized `control_type` value MUST be silently ignored:
logged (naming the unrecognized byte) and dropped ‚Äî the `route` closure returns `nil`
directly, with no forwarding to `routing.RouteFrame`/`OnFrameArrival`. This guard does
NOT live in the PE-upstream `FrameFn` closure, which is frozen per Q2-AMENDED and
carries no code change this story. The DRAIN receive path MUST NOT fail on unknown
opcodes. `control_type = 0x02` (RESYNC) is reserved and dispatched by
`S-BL.RESYNC-FRAME` ‚Äî NOT this story. EC-003 in BC-2.01.008 specifies that RESYNC
received before S-BL.RESYNC-FRAME ships is silently ignored.

### Send-Path Seam and Per-Node Send Map (Q-SEAM + Q2-AMENDED ‚Äî binding, amended v1.3)

**Binding (per placement note Q-SEAM and Q2-AMENDED, F-DW-SP1-001/F-DW-SP1-002, amended
v1.2 F-DW-SP2-002/F-DW-SP2-004/F-DW-SP2-008, amended v1.3 F-DW-SP3-001/005 ‚Äî ownership
split + shutdown ordering guarantee).**

**FO-RECV-FWD-001 disposition:** FO-RECV-FWD-001 is NOT consumed by this story. The
`SetFrameCallback` closure in `mgmt_wire.go` MUST NOT change the `interfaceSet` or
`ForwardFunc`. The nil `ForwardFunc` stays nil (correct ‚Äî E-FWD-001 invariant: split-horizon
exhausts before `fn` is reached with the single-element set `[]{peIfaceID}`). FO-RECV-FWD-001

**DRAIN dispatch path:** DRAIN is a router-originated broadcast, NOT a frame arriving
via `OnFrameArrival`. The drain observer closure in `runRouter` dispatches directly to
each node's send channel. The routing layer is not in the DRAIN path.

**netingress seam (Q-SEAM, binding, ownership split firmed v1.3 ‚Äî F-DW-SP3-001):**

`netingress.Serve` gains an `OnAccept` hook via a `ServeConfig` struct parameter.
`NodeHandle` is passed INTO `OnAccept` as a parameter ‚Äî by Go's own calling
convention, the CALLER of `OnAccept` (i.e. `netingress.Serve`) constructs it before
the call. `netingress` owns DATA creation (allocates `ifaceID` from an atomic
counter seeded by `ServeConfig.IfaceIDSeed`; creates `Send`/`Done`; populates
`NodeHandle` before calling `OnAccept`); `runRouter`'s `OnAccept` closure owns
BEHAVIOR only (see "Per-node send map ownership" below) ‚Äî it does not construct
any `NodeHandle` field itself:

```go
// In internal/netingress:

// NodeHandle is the write handle for an accepted node connection, fully
// populated by netingress BEFORE OnAccept is called. OnAccept receives it as
// a finished value; it does not construct any of its fields.
type NodeHandle struct {
    IfaceID routing.InterfaceID
    Send    chan []byte    // netingress-created. NEVER closed by any goroutine,
                             // for the lifetime of the process ‚Äî see the
                             // single-closer/no-send-after-close invariant
                             // below (Q-SEAM v1.2, F-DW-SP2-002).
    Done    chan struct{}  // netingress-created. Closed EXACTLY ONCE, always
                             // through a per-node doneOnce sync.Once ‚Äî never
                             // call close(h.Done) directly. See "Shutdown
                             // ordering guarantee" below for the two trigger
                             // paths that race to close it (v1.3, F-DW-SP3-005).
}

type ServeConfig struct {
    // OnAccept is called for each ADMITTED connection ‚Äî i.e. one that has
    // cleared the CWE-770 concurrency-cap semaphore (netingress.go:191-197)
    // ‚Äî NEVER for a connection this Serve loop sheds (v1.4, F-DW-SP4-003;
    // see "Admission-gating" below). It fires after netingress has allocated
    // the connection's InterfaceID and created its Send/Done channels
    // (NodeHandle is fully populated), and before ServeConn starts. The
    // returned func() (behavior cleanup) is called in a defer inside the
    // per-conn goroutine when ServeConn returns. Every OnAccept call is
```

**Per-node send map ownership (ownership split binding ‚Äî v1.3, F-DW-SP3-001):**

- **Ownership split.** `netingress.Serve` owns DATA creation: it allocates `ifaceID`
  from an atomic counter seeded by `ServeConfig.IfaceIDSeed` (default 2; `peIfaceID = 1`
  stays reserved for the PE upstream interface), creates `send := make(chan []byte, 32)`
  and `done := make(chan struct{})`, populates `h := NodeHandle{IfaceID: ifaceID, Send:
  send, Done: done}`, and calls `cfg.OnAccept(conn, h)`. `runRouter`'s `OnAccept` closure
  owns BEHAVIOR only: it wraps `h.Send`/`h.Done` in a `*nodeConn` (see "Send-map value
  type" below), stores that in the per-node send map under `h.IfaceID`, fires
  `nodeConnHook`, and starts the writer goroutine ‚Äî it never re-creates or mutates the
  identity of `send`/`done`, only stores and selects on the ones netingress handed it.
- **Send-map value type (binding ‚Äî v1.3, F-DW-SP3-001/005):** because the `done`-close
  trigger has two independent call sites (this connection's own teardown, and the
  router-wide drain-shutdown flush pass ‚Äî see "Shutdown ordering guarantee" below), the
  per-node send map's value type is `*nodeConn`, not a bare `chan []byte`:
  ```go
  type nodeConn struct {
      send     chan []byte
      done     chan struct{}
      doneOnce *sync.Once
  }
  ```
  stored as `sync.Map[routing.InterfaceID, *nodeConn]`.
- **Single-closer / no-send-after-close invariant (binding ‚Äî v1.2, extended v1.3,
  F-DW-SP2-002 / F-DW-SP3-001/005):** the per-node `send` channel is NEVER closed, by any
  goroutine, for the lifetime of the process. The per-node `done` channel IS still closed
  exactly once, but now has TWO possible triggers racing to close it: (a) this
  connection's own teardown via the behavior-cleanup `func()` `OnAccept` returns, and (b)
  the router-wide drain-shutdown flush pass, which proactively closes every still-live
  node's `done` BEFORE `ingressCancel` runs (see "Shutdown ordering guarantee" below).
  BOTH triggers call `nc.doneOnce.Do(func() { close(nc.done) })` rather than
  `close(nc.done)` directly ‚Äî exactly one performs the actual close, the other is a
  harmless no-op. This eliminates a close-race against the single drain observer's
  concurrent `sendMap.Range` (Q-SINGLE-OBS): closing `send` while the observer's
  `select { case nc.send <- frame: default: }` targets it would panic ‚Äî `default:` only
  guards a full channel, not a closed one ‚Äî and the resulting panic would be swallowed by
  `Signal`'s `defer recover()` (Q5/AC-005), silently truncating DRAIN delivery to every
  node ordered after the disconnecting one in map-iteration order. `send` becomes
  unreferenced (GC-eligible) once its map entry is deleted and its writer goroutine has
  exited; any frame the drain observer still manages to enqueue on it in the race window
  between `Range` visiting the entry and cleanup deleting it is a harmless best-effort
  send into an abandoned buffered channel (Q3.P1 best-effort ruling) ‚Äî no consumer reads
  it, and nothing re-closes or re-sends on it.

> below fired `nodeConnHook(nodeConnRegistered, ...)` immediately after `sendMap.Store`,

- `runRouter` `OnAccept` closure (writer goroutine ownership, binding ‚Äî v1.4, reordered
  per F-DW-SP4-001; step bodies otherwise unchanged from v1.3, F-DW-SP3-001/005):
  1. Receives `h` (fully populated by netingress ‚Äî `h.Send`/`h.Done` are the actual
     channels netingress created, not copies) and `conn`. Wraps them as
     `nc := &nodeConn{send: h.Send, done: h.Done, doneOnce: new(sync.Once)}` and stores
     it in the per-node send map (`sendMap.Store(h.IfaceID, nc)`).
  2. Increments the router-scoped `writerWG` (`sync.WaitGroup`, one per `runRouter`
     invocation ‚Äî see "Shutdown ordering guarantee" below) by 1, then starts a writer
     goroutine (sole writer to `conn`):
     ```go
     go func() {
         defer writerWG.Done()
         for {
             select {
             case frame := <-nc.send:
                 if _, err := conn.Write(frame); err != nil {
                     return // conn closed or write error; exit loop
                 }
             case <-nc.done:
                 // Flush any frame(s) already queued (e.g. a DRAIN frame enqueued by
                 // the single observer's Range call) before exiting ‚Äî do NOT drop
                 // them silently (F-DW-SP3-005).
                 for {
                     select {
                     case frame := <-nc.send:
                         if _, err := conn.Write(frame); err != nil {
                             return
                         }
                     default:
                         return
                     }
                 }
             }
         }
     }()
     ```

     synchronization point tests block on (Q-AC002), so by the time a test observes
     `nodeConnRegistered` and proceeds to trigger drain, `writerWG`'s count for this
     connection is already committed ‚Äî the shutdown flush pass's `writerWG.Wait()`
     (below) cannot observe a stale zero for it.
  4. Does NOT register a per-connection `drain.Observer` ‚Äî see Q-SINGLE-OBS below; only
     the single startup observer, registered once before serving, is ever registered.
  5. Returns a behavior-cleanup `func()` that: removes the `InterfaceID` entry from the
     per-node send map; fires `nodeConnHook(nodeConnRemoved, h.IfaceID)` if
     `nodeConnHook != nil`; calls `nc.doneOnce.Do(func() { close(nc.done) })` ‚Äî one of the
     two possible `done`-close triggers (harmless no-op if the router-wide shutdown-flush
     pass already closed it first). `send` itself is NEVER closed ‚Äî see the invariant
     above.
- `ServeConn` is the sole reader of `conn`. No mutex is required (net.TCPConn read/write

### Shutdown Ordering Guarantee (Q-SEAM ‚Äî binding, v1.3, extended v1.4 per F-DW-SP4-004, tail REORDERED v1.5 per F-DW-SP5-001; the bounded flush-phase mechanism established v1.3 is unchanged)

**The problem:** Q3.P1's best-effort delivery means the single drain observer returns as
soon as it has QUEUED the DRAIN frame to every live node's `send` channel (a non-blocking
`select`/`default`) ‚Äî `drainCoord.Wait` returning `nil` says only "queuing finished," not
"every writer goroutine has flushed its queued frame to the wire." The existing shutdown
block calls `ingressCancel()` immediately after `drainCoord.Wait(drainCtx)` returns, which
tears down `ingressCtx` and closes every accepted connection's underlying `net.Conn`.
Nothing orders the writer goroutine's `conn.Write(DRAIN)` before that close ‚Äî under
scheduling pressure the `net.Conn` can close before the writer even reaches its
`case frame := <-nc.send:` branch, and the node reads EOF instead of the DRAIN frame.
This is the AC-004/AC-001 read-determinism dependency both discharge-traces rely on.

**v1.5 correction (F-DW-SP5-001, HIGH):** the v1.4 tail ‚Äî `... ‚Üí ingressCancel() ‚Üí
writerWG.Wait() UNBOUNDED ‚Üí (existing dataWG.Wait())` ‚Äî reopened the SAME defect class
F-DW-SP4-004 was built to close, at a DIFFERENT pair of racing call sites, and its
safety-justification comment was wrong about the actual exit mechanism. `ingressCancel()`
is a `context.CancelFunc` ‚Äî it only SIGNALS `ingressCtx`; the listener close
(`ln.Close()`) runs ASYNCHRONOUSLY in `netingress.Serve`'s own watcher goroutine, woken
off the same cancellation with no ordering guarantee relative to `ingressCancel()`
returning. A connection already racing `ln.Accept()` can still be admitted AFTER
`ingressCancel()` executes, so `OnAccept` can still fire ‚Äî from the freshly spawned
per-conn goroutine, per the Goroutine pin above ‚Äî and call `writerWG.Add(1)`
CONCURRENTLY with the already-parked unbounded `writerWG.Wait()`: a `sync.WaitGroup`
Add-concurrent-with-Wait misuse, moved rather than eliminated. Separately, "any blocked
writer's `conn.Write` errors out and it exits promptly" mis-describes a writer still
PARKED in `select { case frame := <-nc.send: ...; case <-nc.done: ... }` with an empty
`send` and a not-yet-closed `done` ‚Äî that writer calls `conn.Write` ZERO times until
`done` closes, and `done` closes only via this connection's behavior-cleanup `func()`,
run from the per-conn goroutine's `defer` chain ‚Äî a goroutine `netingress.Serve`'s own
internal `wg` joins (inside its `ln.Accept()` error branch) BEFORE `Serve` itself
returns. `Serve` is the goroutine `dataWG` tracks; `dataWG.Wait()` is the ONLY existing
join of that completion. The v1.4 tail ran the final unbounded `writerWG.Wait()` BEFORE
`dataWG.Wait()`, so its completion silently depended on goroutines (and their
`done`-closing cleanup) the sequence had not yet joined. Not a deadlock ‚Äî
`ServeConn`'s own ctx-cancel handling eventually drives every per-conn goroutine to
completion regardless ‚Äî but the stated mechanism was wrong.

**Ruled ‚Äî insert a bounded flush phase between `Wait` and `ingressCancel()`, THEN join
`dataWG` before the final unbounded `writerWG` join (binding ‚Äî v1.5, F-DW-SP5-001,
reorders the tail relative to v1.4; the bounded flush phase itself is unchanged from
v1.3):**

```
Signal ‚Üí Wait ‚Üí close all live nodes' done (router-wide flush pass) ‚Üí
writerWG.Wait() bounded by drainFlushTimeout ‚Üí ingressCancel() ‚Üí
dataWG.Wait() ‚Üí writerWG.Wait() UNBOUNDED
```

The shutdown block gains two new steps between the existing `drainCoord.Wait(drainCtx)`
call and `ingressCancel()` (unchanged from v1.3), and the existing `dataWG.Wait()` call
(already present pre-story) now runs BEFORE one new final step (v1.5, F-DW-SP5-001,

```go

    nc.doneOnce.Do(func() { close(nc.done) })
    return true
```

**`drainFlushTimeout` ‚Äî RESOLVED, no longer PROVISIONAL (binding ‚Äî v1.4, pass-4

merely the mechanism. A derived value (e.g., a fraction of `cfg.DrainTimeout`) was

(subtracting from a budget that has already reached zero by the time this phase runs)
or, worse, yields ~0 remaining headroom on the EC-003 forced-exit path specifically
(Q-AC005) ‚Äî exactly the path where a bounded-but-nonzero flush window matters most.
200ms is a defensible hard ceiling on ADDED shutdown latency: writers flush in parallel
(one goroutine per node, all racing the same deadline concurrently, not serially), and
this story's drain observer queues exactly one frame per node (the DRAIN frame) ‚Äî a
single ~48-byte `conn.Write` per writer, which completes in microseconds under normal
scheduling; 200ms is headroom for scheduler noise, not an expected steady-state cost.

It is deliberately NOT `cfg.DrainTimeout` ‚Äî that budget is already spent by the time
this phase runs; this is a second, much shorter bound purely for flushing
already-queued, already-small frames to a socket that should accept them near-instantly
under normal conditions. **Caveat (binding ‚Äî v1.4):** any FUTURE test that asserts a
tight ceiling on total shutdown elapsed time must budget `+drainFlushTimeout` (200ms)
into that ceiling, since the flush phase now unconditionally runs on every `runRouter`
shutdown, whether or not any node is connected.

**Why `sync.Once` and not a direct `close(nc.done)` here:** this router-wide pass and each
connection's own behavior-cleanup are two INDEPENDENT triggers that can both fire for the
same node ‚Äî e.g. a node that happens to disconnect on its own at the same moment the
router is draining. `doneOnce` (created alongside `done`, stored in the `nodeConn` entry)
generalizes the single-closer fix to any number of trigger sites, present or future,
without reintroducing a double-close panic.

### Single Startup Observer (Q-SINGLE-OBS ‚Äî binding, amended v1.3)

**Binding (per placement note Q-SINGLE-OBS, F-DW-SP1-007, amended v1.3 F-DW-SP3-001/005/

```go

        drainObserverFiredHook()

    sendMap.Range(func(key, value any) bool {
        nc := value.(*nodeConn)
        frame := assembleDrainFrame(...)

        case nc.send <- frame:
        default: // channel full ‚Äî best-effort (never closed, per the single-closer/
                 // no-send-after-close invariant above; this send cannot panic)
        }

    })
}

```

The observer then returns (context deadline is the bound ‚Äî Q3.P1 BINDING option (2)).

`drainCoord.RegisterObserver`. Only the single startup observer is registered.

**Consequence for `drain.Drain` API:** No deregister API is needed. `RegisterObserver`
retains its current shape.
