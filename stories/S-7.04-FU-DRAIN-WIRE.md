# S-7.04-FU-DRAIN-WIRE: DRAIN-over-SVTN Wire Propagation

## Narrative

- **As an** operator gracefully shutting down a PE router
- **I want** connected nodes to receive a DRAIN signal over their SVTN channels before
  the router disconnects
- **So that** nodes can migrate active sessions to alternate routers within the drain
  window, rather than losing them abruptly

## Context

S-7.04 (merged PR #101/#103) delivered `internal/drain.Drain` with `RegisterObserver`,
`Signal`, `Wait`, and the `drainCoordHook` test-injection mechanism. The coordinator's
liveness model (fan-out-then-race under a bounded child context, EC-003 ErrTimeout) is
complete and tested. However, `runRouter` currently has zero registered observers: no
per-node connection identity model exists at the DRAIN observer call site.

S-BL.PE-RECEIVE-LOOP (merged PR #118 @ `e940fc2`, 2026-07-11) delivered the per-connection
receive goroutine, `upstreamdial.Connector.SetFrameCallback`, and the
`FrameArrivalHandler.OnFrameArrival` wiring. It also emitted FO-RECV-FWD-001: the nil
`ForwardFunc` at `mgmt_wire.go`:549 must be replaced when the interface set widens.

**v1.1 clarification (Q2-AMENDED):** FO-RECV-FWD-001 is NOT consumed by this story. The
nil `ForwardFunc` stays; the `interfaceSet` stays `[]{peIfaceID}`. DRAIN is a
router-originated broadcast dispatched directly from the drain observer closure to per-node
send channels — it does NOT route through `OnFrameArrival`. The E-FWD-001 invariant is
preserved. FO-RECV-FWD-001 remains deferred to the future data-frame-widening story.

This story lands three deliverables:

1. **DRAIN frame on the wire** — A single startup drain observer (registered once at
   `runRouter` setup) iterates the live per-node send map at `Signal` time. For each
   connected node, it assembles a `FrameTypeCtl` (0x03) frame with a 4-byte payload
   (`control_type = 0x01`, DRAIN opcode) and non-blocking sends it to the node's send
   channel. The per-node send map is a `sync.Map` owned by `runRouter`; entries are
   added via the `netingress.ServeConfig.OnAccept` hook and removed on conn close.

2. **netingress OnAccept seam** — `netingress.Serve` gains a `ServeConfig` parameter
   carrying an `OnAccept func(conn net.Conn, h NodeHandle) func()` hook. `runRouter`
   supplies the hook to allocate `routing.InterfaceID`, create the per-node send channel,
   start the writer goroutine, and register the channel in the send map. Cleanup
   (channel close + map remove) is deferred on conn close. This is the binding Q-SEAM
   design.

3. **Per-observer panic recovery and EC-003 production-path evidence** — `drain.Signal`'s
   fan-out goroutine currently has no `defer recover()`. This story adds the recovery
   wrapper. EC-003 is evidenced via `drainCoordHook` + blocked observer +
   `cfg.DrainTimeout=100ms` (the real drain-timeout injection seam per Q-AC005).

**Previous Story Intelligence.** `S-BL.PE-RECEIVE-LOOP` completed 4 adversarial passes
(21 findings, all dispositioned) before convergence. Key lessons carried forward:
(1) Every symbol in ACs must be grep-verified at `e940fc2` or marked "(new — defined by
this story)" — the placement note's Appendix A performed this sweep; use it.
(2) Avoid line-number citations in story prose — use mechanism-anchor descriptions.
(3) The "ctl discrimination" lives in the netingress `route` RouteFn closure — NOT the
PE-upstream `FrameFn` closure, which stays frozen per Q2-AMENDED — and must silently
ignore unknown `control_type` values (FO-DRAIN-WIRE-001 forward-compat rule) — the
RESYNC opcode (0x02) is RESERVED but not dispatched here; failing to add this guard was
a recurring error class in S-7.04-FU-PE-CONNECTOR (F-P11-001 lesson).
(4) Production send-path seams need full round-trip pins — a test that only registers
an observer without verifying the DRAIN frame arrives on the wire is a false positive.
(5) All drainCoord-behavior tests use the real-runRouter pattern (`startRunRouterWithConfig`,
verified `router_drain_test.go`:94-114 at `e940fc2`) + `drainCoordHook` to capture the
`*drain.Drain`; do NOT use `testenv.SendDrainSignal` for runRouter's coordinator (it
drives testenv's separate `drainCtrl` — a different `drain.Drain` instance).

## Design Constraints

### DRAIN Frame Wire Schema (Q1 — binding)

**Binding (per placement note Q1).**

The DRAIN signal is carried as a `FrameTypeCtl = 0x03` frame (existing constant, verified
`internal/frame/frame.go` at `e940fc2`). No new `FrameType` constant is introduced. The
DRAIN payload schema is a fixed 4-byte control message (BC-2.01.008 v1.0, schema home):

```
byte[0]   control_type   = 0x01   // DRAIN opcode
byte[1]   version        = 0x01   // protocol version; allows future extension
byte[2:4] reserved       = 0x0000 // zero-filled; receiver MUST ignore
```

Rationale: `FrameTypeCtl` is the existing discriminator for authenticated control frames.
Adding `FrameTypeDrain = 0x07` would require widening `Valid()` past `FrameTypePEConnect =
0x06` — the guard S-BL.PE-RECEIVE-LOOP erected. Using `FrameTypeCtl` with a
`control_type` payload byte is the correct design: outer header identifies the wire
family (`ctl`); the control-message opcode lives in the payload. BC-2.01.008 is the
authoritative schema home; BC-2.01.004 Inv-2 carries the terminal-consumer carve-out
cross-reference.

**Assembly path:** The observer closure constructs the 4-byte payload and calls
`frame.EncodeOuterHeader` + `append(ehdr[:], payload...)`. The `Envelope` fields
(SVTN, SrcAddr, DstAddr, FrameAuthKey) are sourced from the per-node connection context
stored in the `runRouter` connection registry using the zero-envelope pattern identical to
S-7.04-FU-PE-CONNECTOR (FO-DRAIN-WIRE-002 notes that full Ed25519 bootstrap is a
session-bootstrap follow-on). `frame.EncodeOuterHeader` is an EXISTING function at
`e940fc2` (not new — defined by this story).

**Wire-format spec-pair obligation (same-commit — binding):** This story introduces the
`control_type` discriminator byte in the `FrameTypeCtl` payload schema. The following
same-commit amendments are required:

1. ARCH-02 § "Outer Header Format" (NOT §3.1 — ARCH-02 has no §3.1, corrected v1.2,
   F-DW-SP2-009) frame_type table: add a note that `ctl (0x03)` payloads carry a
   `control_type` byte discriminator; enumerate opcode `0x01 = DRAIN`.
2. BC-2.01.004 Postcondition 2 (outer-header layout): add the same `control_type` note
   under the `frame_type = ctl` row (with cross-ref to BC-2.01.008 as schema home).

These are a NARROWER obligation than the FrameTypePEConnect case (F-SP14-001) — the
frame_type byte itself is unchanged (0x03 already listed); only the payload schema
documentation is new. The same-commit rule (F-P8-008) still applies.

**Forward-compatibility silent-ignore rule (FO-DRAIN-WIRE-001, binding; anchors to
BC-2.01.008 Postcondition 4; guard home firmed v1.2, F-DW-SP2-003):** Any `FrameTypeCtl`
frame arriving at the netingress `route` RouteFn closure (`mgmt_wire.go`, the inbound
per-node path) with an unrecognized `control_type` value MUST be silently ignored:
logged (naming the unrecognized byte) and dropped — the `route` closure returns `nil`
directly, with no forwarding to `routing.RouteFrame`/`OnFrameArrival`. This guard does
NOT live in the PE-upstream `FrameFn` closure, which is frozen per Q2-AMENDED and
carries no code change this story. The DRAIN receive path MUST NOT fail on unknown
opcodes. `control_type = 0x02` (RESYNC) is reserved and dispatched by
`S-BL.RESYNC-FRAME` — NOT this story. EC-003 in BC-2.01.008 specifies that RESYNC
received before S-BL.RESYNC-FRAME ships is silently ignored.

`netingress.Serve` gains an `OnAccept` hook via a `ServeConfig` struct parameter.
`NodeHandle` is passed INTO `OnAccept` as a parameter — by Go's own calling
convention, the CALLER of `OnAccept` (i.e. `netingress.Serve`) constructs it before

any `NodeHandle` field itself:

```go

// NodeHandle is the write handle for an accepted node connection, fully
// populated by netingress BEFORE OnAccept is called. OnAccept receives it as
// a finished value; it does not construct any of its fields.
type NodeHandle struct {
    IfaceID routing.InterfaceID
    Send    chan []byte    // netingress-created. NEVER closed by any goroutine,
                             // for the lifetime of the process — see the
                             // single-closer/no-send-after-close invariant
                             // below (Q-SEAM v1.2, F-DW-SP2-002).
    Done    chan struct{}  // netingress-created. Closed EXACTLY ONCE, always
                             // through a per-node doneOnce sync.Once — never
                             // call close(h.Done) directly. See "Shutdown
                             // ordering guarantee" below for the two trigger
                             // paths that race to close it (v1.3, F-DW-SP3-005).
}

type ServeConfig struct {
    // OnAccept is called for each ADMITTED connection — i.e. one that has
    // cleared the CWE-770 concurrency-cap semaphore (netingress.go:191-197)
    // — NEVER for a connection this Serve loop sheds (v1.4, F-DW-SP4-003;
    // see "Admission-gating" below). It fires after netingress has allocated
    // the connection's InterfaceID and created its Send/Done channels
    // (NodeHandle is fully populated), and before ServeConn starts. The
    // returned func() (behavior cleanup) is called in a defer inside the
    // per-conn goroutine when ServeConn returns. Every OnAccept call is
    // paired 1:1 with exactly one guaranteed cleanup invocation. If nil, no
    // hook fires and netingress allocates nothing for that connection
    // (zero-cost when unused); a shed connection likewise allocates nothing
    // and calls neither OnAccept nor cleanup, regardless of whether OnAccept
    // is nil.
    OnAccept func(conn net.Conn, h NodeHandle) func()

    // IfaceIDSeed is the first InterfaceID netingress allocates for an
    // admitted connection when OnAccept is non-nil; subsequent admitted
    // connections get IfaceIDSeed+1, IfaceIDSeed+2, ... via an internal
    // atomic counter. Shed connections do not consume a value from this
    // counter. Zero defaults to 2 (preserves the reserved peIfaceID =
    // routing.InterfaceID(1) and keeps a zero-value ServeConfig
    // runtime-compatible — see the compatibility note below for what that
    // does and does not cover). New v1.3, F-DW-SP3-001.
    IfaceIDSeed routing.InterfaceID
}

// func Serve(ctx context.Context, ln net.Listener, route RouteFn,
```

**Admission-gating — OnAccept fires only for admitted connections (binding — v1.4,
F-DW-SP4-003):** the story-stub language "OnAccept fires for each accepted
connection ... before ServeConn starts" read literally as firing for every TCP-level

before that point; it never reaches `cfg.OnAccept`, allocates none of `NodeHandle`'s
fields, and therefore has no writer goroutine, no send-map entry, and no
`nodeConnHook` firing to clean up — the `OnAccept`↔cleanup pairing is 1:1 by
construction. Without this gating, a shed connection could otherwise leak a phantom
`writerWG` entry that no cleanup path ever retires, and every subsequent router
shutdown would pay the full `drainFlushTimeout` waiting on it for nothing.

**Per-node send map ownership (ownership split binding — v1.3, F-DW-SP3-001):**

- **Ownership split.** `netingress.Serve` owns DATA creation: it allocates `ifaceID`
  from an atomic counter seeded by `ServeConfig.IfaceIDSeed` (default 2; `peIfaceID = 1`
  stays reserved for the PE upstream interface), creates `send := make(chan []byte, 32)`
  and `done := make(chan struct{})`, populates `h := NodeHandle{IfaceID: ifaceID, Send:
  send, Done: done}`, and calls `cfg.OnAccept(conn, h)`. `runRouter`'s `OnAccept` closure
  owns BEHAVIOR only: it wraps `h.Send`/`h.Done` in a `*nodeConn` (see "Send-map value
  type" below), stores that in the per-node send map under `h.IfaceID`, fires
  `nodeConnHook`, and starts the writer goroutine — it never re-creates or mutates the
  identity of `send`/`done`, only stores and selects on the ones netingress handed it.
- **Send-map value type (binding — v1.3, F-DW-SP3-001/005; extended v1.6,
  F-DW-SP6-001):** because the `done`-close trigger has two independent call sites (this
  connection's own teardown, and the router-wide drain-shutdown flush pass — see
  "Shutdown ordering guarantee" below), the per-node send map's value type is
  `*nodeConn`, not a bare `chan []byte`:
  ```go
  type nodeConn struct {
      send         chan []byte
      done         chan struct{}
      doneOnce     *sync.Once
      writerExited chan struct{} // v1.6, F-DW-SP6-001 — closed by the writer
                                  // goroutine itself, via defer, on exit. Single
                                  // closer by construction; no sync.Once
                                  // companion needed (contrast done, which has
                                  // two independent closer call sites). The
                                  // Shutdown ordering guarantee's bounded flush
                                  // phase waits on exactly this channel, per
                                  // snapshotted node, instead of the shared
                                  // writerWG.
  }
  ```
  stored as `sync.Map[routing.InterfaceID, *nodeConn]` (`Store(h.IfaceID, &nodeConn{send:
  h.Send, done: h.Done, doneOnce: new(sync.Once), writerExited: make(chan struct{})})`
  inside `OnAccept`).
- **Single-closer / no-send-after-close invariant (binding — v1.2, extended v1.3,
  F-DW-SP2-002 / F-DW-SP3-001/005):** the per-node `send` channel is NEVER closed, by any
  goroutine, for the lifetime of the process. The per-node `done` channel IS still closed
  exactly once, but now has TWO possible triggers racing to close it: (a) this
  connection's own teardown via the behavior-cleanup `func()` `OnAccept` returns, and (b)
  the router-wide drain-shutdown flush pass, which proactively closes every still-live
  node's `done` BEFORE `ingressCancel` runs (see "Shutdown ordering guarantee" below).
  BOTH triggers call `nc.doneOnce.Do(func() { close(nc.done) })` rather than
  `close(nc.done)` directly — exactly one performs the actual close, the other is a
  harmless no-op. This eliminates a close-race against the single drain observer's
  concurrent `sendMap.Range` (Q-SINGLE-OBS): closing `send` while the observer's
  `select { case nc.send <- frame: default: }` targets it would panic — `default:` only
  guards a full channel, not a closed one — and the resulting panic would be swallowed by
  `Signal`'s `defer recover()` (Q5/AC-005), silently truncating DRAIN delivery to every
  node ordered after the disconnecting one in map-iteration order. `send` becomes
  unreferenced (GC-eligible) once its map entry is deleted and its writer goroutine has
  exited; any frame the drain observer still manages to enqueue on it in the race window
  between `Range` visiting the entry and cleanup deleting it is a harmless best-effort
  send into an abandoned buffered channel (Q3.P1 best-effort ruling) — no consumer reads
  it, and nothing re-closes or re-sends on it.
- **`writerExited` — single-writer close, no `sync.Once` needed (binding — v1.6,
  F-DW-SP6-001):** unlike `done`, `writerExited` has exactly ONE possible closer — the
  writer goroutine's own `defer close(nc.writerExited)` (Writer goroutine ownership, step
  2, below), which by definition of `defer` executes exactly once per goroutine lifetime,
  on that goroutine alone. No other goroutine ever calls `close(nc.writerExited)` or
  sends on it. This is categorically different from `done` (two independent trigger call
  sites, requiring `doneOnce`) — a second closer would be a distinct, larger design
  change this story does not make.
> **v1.4 SUPERSEDED — step ordering only (F-DW-SP4-001, HIGH):** through v1.3, step 1
> below fired `nodeConnHook(nodeConnRegistered, ...)` immediately after `sendMap.Store`,
> BEFORE `writerWG` was incremented or the writer goroutine started (old step 2). Tests
> use `nodeConnHook`'s `nodeConnRegistered` event as the accept/register barrier before
> calling `cancel()` (Q-AC002, AC-001/AC-004 recipes) — so a test could observe the
> hook, call `cancel()`, and reach the shutdown block's flush pass (`writerWG.Wait()`,
> Shutdown ordering guarantee below) BEFORE `writerWG.Add(1)` had run for this
> connection. `sync.WaitGroup.Wait()` returns immediately when the counter is (still)
> zero, so the flush pass would proceed straight to `ingressCancel()` with this
> connection's writer goroutine not yet started — closing the underlying `net.Conn`
> before the writer ever reaches its `case frame := <-nc.send:` branch, reopening the
> AC-004/AC-001 EOF flake via a different race than the one v1.3 fixed. This is also a
> `sync.WaitGroup` Add-concurrent-with-Wait misuse in its own right (`-race`-flaggable):
> nothing ordered this connection's `Add(1)` relative to the flush pass's `Wait()` call.
> **Ruled: reorder so `writerWG.Add(1)` and the writer-goroutine start happen BEFORE
> `nodeConnHook` fires.** The hook becomes the happens-after anchor for the Add+launch:

> `cancel()`. Adopted together with F-DW-SP4-004's new final unbounded

### Shutdown Ordering Guarantee (Q-SEAM — binding, v1.3, extended v1.4 per F-DW-SP4-004, tail REORDERED v1.5 per F-DW-SP5-001, bounded-phase WAIT MECHANISM replaced v1.6 per F-DW-SP6-001, EXTENDED v1.7 per F-DW-SP7-001 with an explicit join of the bounded phase's own goroutines before `runRouter` returns; the final-join ordering `dataWG.Wait() → writerWG.Wait()` established v1.5 is UNCHANGED; the bounded flush-phase's close-every-live-`done` step established v1.3 is UNCHANGED; the v1.6 snapshot-scoped WAIT MECHANISM itself is UNCHANGED — pass 7 CONFIRMED it closes F-DW-SP6-001 without a fifth relocation)

**The problem:** Q3.P1's best-effort delivery means the single drain observer returns as
soon as it has QUEUED the DRAIN frame to every live node's `send` channel (a non-blocking
`select`/`default`) — `drainCoord.Wait` returning `nil` says only "queuing finished," not
"every writer goroutine has flushed its queued frame to the wire." The existing shutdown
block calls `ingressCancel()` immediately after `drainCoord.Wait(drainCtx)` returns, which
tears down `ingressCtx` and closes every accepted connection's underlying `net.Conn`.
Nothing orders the writer goroutine's `conn.Write(DRAIN)` before that close — under
scheduling pressure the `net.Conn` can close before the writer even reaches its
`case frame := <-nc.send:` branch, and the node reads EOF instead of the DRAIN frame.
This is the AC-004/AC-001 read-determinism dependency both discharge-traces rely on.

**v1.5 correction (F-DW-SP5-001, HIGH):** the v1.4 tail — `... → ingressCancel() →
writerWG.Wait() UNBOUNDED → (existing dataWG.Wait())` — reopened the SAME defect class
F-DW-SP4-004 was built to close, at a DIFFERENT pair of racing call sites, and its
safety-justification comment was wrong about the actual exit mechanism. `ingressCancel()`
is a `context.CancelFunc` — it only SIGNALS `ingressCtx`; the listener close
(`ln.Close()`) runs ASYNCHRONOUSLY in `netingress.Serve`'s own watcher goroutine, woken
off the same cancellation with no ordering guarantee relative to `ingressCancel()`
returning. A connection already racing `ln.Accept()` can still be admitted AFTER
`ingressCancel()` executes, so `OnAccept` can still fire — from the freshly spawned
per-conn goroutine, per the Goroutine pin above — and call `writerWG.Add(1)`
CONCURRENTLY with the already-parked unbounded `writerWG.Wait()`: a `sync.WaitGroup`
Add-concurrent-with-Wait misuse, moved rather than eliminated. Separately, "any blocked

### Ctl Receive Guard (Q-CTL-GUARD — binding, guard home firmed v1.2, terminal-consumer basis + logging asymmetry firmed v1.3)

The `FrameTypeCtl` frame receive path in `cmd/switchboard` MUST length-check the payload
before reading `payload[0]`. **Guard home (firmed v1.2): the netingress `route` RouteFn
closure exclusively** (`mgmt_wire.go`'s `route := func(hdr frame.OuterHeader, payload

**Terminal-consumer basis for the unconditional guard (binding — v1.3, F-DW-SP3-004; PO
ruling BC-2.01.008 v1.1 Inv-2 consumed):** this guard applies to EVERY `ctl` frame

inter-router forwarding/relay (a topology where a `ctl` frame could arrive via a
router-facing connection addressed to a downstream router), this guard's unconditional
posture must be re-examined against BC-2.01.008 Precondition 2's general
terminal-consumer language — carry this revisit trigger forward into any follow-on
story that adds router-to-router connections.

### AC-002 — netingress OnAccept seam: per-node connection registers in send map; writer goroutine is sole writer; cleanup deregisters (Q-SEAM binding, ownership split firmed v1.3)

**BC Anchors:** BC-2.09.002 PC-1 (DRAIN reaches each connected node via per-node send
channel); placement note Q-SEAM (F-DW-SP1-001, ownership split F-DW-SP3-001). Traces to
BC-2.09.002 PC-1.

**Precondition:** `runRouter` is running with the `OnAccept` hook wired via `ServeConfig`.
A raw TCP connection is accepted by `netingress.Serve`.

**Postconditions:**

1. `netingress.Serve` owns DATA creation (ownership split, v1.3 — F-DW-SP3-001):
   allocates `IfaceID` from an atomic counter seeded by `ServeConfig.IfaceIDSeed`
   (default 2; `peIfaceID=1` stays reserved), creates `Send chan []byte` (buffered 32,
   NEVER closed by any goroutine — single-closer/no-send-after-close invariant, Q-SEAM
   v1.2/v1.3, F-DW-SP2-002/F-DW-SP3-001/005) and `Done chan struct{}`, and populates a
   fully-built `NodeHandle{IfaceID, Send, Done}` BEFORE calling `OnAccept`. `OnAccept`
   fires ONLY for each ADMITTED connection (i.e. one that has cleared the CWE-770
   concurrency-cap semaphore — v1.4, F-DW-SP4-003, see Q-SEAM's Admission-gating
   subsection) with this pre-populated handle, BEFORE `ServeConn` starts; it does not
   construct any of the handle's fields. `OnAccept` fires from the freshly spawned
   per-conn goroutine, as the FIRST act before `ServeConn` starts — NEVER from the
   `Serve` accept loop itself (Goroutine pin, binding — v1.5, F-DW-SP5-001, pins prior
   latitude); the accept loop's only per-connection work is admission bookkeeping
   (semaphore check, `wg.Add(1)`, goroutine spawn). The returned behavior-cleanup
   `func()` is deferred in that SAME per-conn goroutine, with its `defer` statement
   placed AFTER the goroutine's `defer wg.Done()` in source order — Go's LIFO defer
   ordering guarantees cleanup completes before `wg.Done()` fires, which is the
   mechanism the Shutdown ordering guarantee's `dataWG.Wait()`-completes-Serve
   reasoning depends on. A shed connection allocates none of `NodeHandle`'s fields and
   calls neither `OnAccept` nor cleanup.
2. The `runRouter` `OnAccept` closure owns BEHAVIOR only: wraps `h.Send`/`h.Done` into
   `nc := &nodeConn{send: h.Send, done: h.Done, doneOnce: new(sync.Once), writerExited:
   make(chan struct{})}` (v1.6, F-DW-SP6-001 adds `writerExited`); stores `nc` in the
   per-node `sync.Map` under `h.IfaceID`; increments the router-scoped `writerWG`
   and starts a writer goroutine that `select`s on `nc.send`/`nc.done` — reads and
   writes frames from `nc.send`, and on `nc.done` flushes any already-queued frame
   before exiting (Shutdown ordering guarantee, v1.3, F-DW-SP3-005) rather than
   dropping it — exits when `conn.Write` errors or after the flush completes, closing
   `nc.writerExited` via `defer` on exit (v1.6, F-DW-SP6-001 — single closer by
   construction, no `sync.Once` needed); ONLY THEN fires
   `nodeConnHook(nodeConnRegistered, h.IfaceID)` (Q-AC002) — AFTER `writerWG.Add(1)`
   and the writer goroutine's `go` statement have both executed (reordered v1.4,
   F-DW-SP4-001: through v1.3 the hook fired immediately after `Store`, before the
   writer goroutine was counted, which could let a test's `cancel()` race the shutdown
   flush pass's `writerWG.Wait()` against a still-zero counter — see the Writer
   goroutine ownership v1.4 supersession banner in Design Constraints for the full
   derivation).
3. The writer goroutine is the SOLE writer to `conn`; `ServeConn` is the sole reader.
4. When `conn` closes (ServeConn returns), the behavior-cleanup `func()` returned by
   `OnAccept` is called: the `sync.Map` entry for `h.IfaceID` is deleted;
   `nodeConnHook(nodeConnRemoved, h.IfaceID)` fires (Q-AC002) immediately after the
   `Delete`; `nc.doneOnce.Do(func() { close(nc.done) })` is called — one of two possible
   `done`-close triggers (the other being the router-wide shutdown-flush pass; harmless
   no-op if that pass already closed it first, v1.3, F-DW-SP3-005). `send` is NEVER
   closed — see the single-closer/no-send-after-close invariant, Q-SEAM v1.2/v1.3,
   F-DW-SP2-002/F-DW-SP3-001/005. `writerExited` is NEVER touched by cleanup (v1.6,
   F-DW-SP6-001) — it is closed exclusively by the writer goroutine's own `defer`.
5. **Negative postcondition (binding — Q2-AMENDED):** The `SetFrameCallback` closure in
   `mgmt_wire.go` MUST NOT be changed by this story. `interfaceSet` stays
   `[]{peIfaceID}`; `ForwardFunc` stays `nil`. FO-RECV-FWD-001 is NOT consumed. The
   `TestRunRouter_ForwardFunc_WiredToNodeSendMap` test from v1.0 is REMOVED and replaced
   by lifecycle tests (see below).
6. **Observability (Q-AC002, binding — v1.2, variable names aligned v1.3):** tests MUST
   NOT assert directly on the contents of the unexported per-node `sync.Map`
   (`sync.Map[routing.InterfaceID, *nodeConn]` as of v1.3). Registration and removal are
   observed exclusively via the package-level `nodeConnHook(event, ifaceID)` test hook
   (mirrors the existing `drainCoordHook` precedent), fired synchronously from `OnAccept`
   (`nodeConnRegistered`) and `cleanup` (`nodeConnRemoved`); nil in production (no-op).

**Test names (respecified v1.2 against `nodeConnHook`, F-DW-SP2-006; descriptors aligned
v1.3 to the ownership split — neither test reads the `sync.Map` directly):**

- `TestNetingress_OnAccept_RegistersNodeHandle` (unit/integration, `cmd/switchboard/router_drain_wire_test.go` — accepts a simulated inbound conn; asserts `OnAccept` fires before `ServeConn`; asserts `NodeHandle.IfaceID >= 2` — now legitimately satisfiable, since `netingress.Serve` itself allocates the `IfaceID` from the `ServeConfig.IfaceIDSeed`-seeded counter (v1.3 ownership split); asserts `nodeConnHook` records a `nodeConnRegistered` event for the `IfaceID` — the send-map-registration observable, per Q-AC002)
