# S-4.02: Upstream Idempotent Replay Window

> **Execute:** `/vsdd-factory:deliver-story S-4.02`

## Narrative

- **As an** access node
- **I want to** receive upstream keystroke frames that carry the last N keystrokes
- **So that** transient path failures do not cause permanently lost keystrokes, and duplicate deliveries are suppressed

## Acceptance Criteria

### AC-001 (traces to BC-2.02.004 postcondition 1)
`Replay.OnUpstream(frame)` never delivers the same sequence number twice. Second delivery of the same seq returns `ErrAlreadyDelivered`.
- **Test:** `TestReplay_NoDuplicateDelivery`

### AC-002 (traces to BC-2.02.004 postcondition 2)
`Replay.OnUpstream` delivers keystrokes in sequence order. If seq N+1 arrives before N (out-of-order), N+1 is buffered and delivered after N arrives.
- **Test:** `TestReplay_InOrderDelivery`

### AC-003 (traces to BC-2.02.004 invariant 1)
The replay window carries the last N keystrokes (where N is configured). Frames older than the window are discarded without error.
- **Test:** `TestReplay_WindowBoundary`

### AC-004 (traces to BC-2.02.004 postcondition 3)
Keystroke-to-echo p99 latency is ≤ 100ms under single-path conditions (benchmark gate for VP-042).
- **Test:** `BenchmarkReplay_KeystrokeLatency`

## Edge Cases

| ID | Scenario | Expected Behavior |
|----|----------|-------------------|
| EC-001 | Sequence number exactly at window boundary | Window-oldest entry is evicted; new entry accepted |
| EC-002 | All N frames in window re-sent | All N deduplicated (no double delivery) |
| EC-003 | Gap in sequence (missing frame N) | N+1 through N+K buffered; delivered in order when N arrives or window expires |
