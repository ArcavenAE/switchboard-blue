# S-7.04-FU-SIGHUP-RELOAD — DELIVERY

## What Landed

This story closes the SIGHUP runtime reload path that S-7.04 deferred. It
delivers a fully fail-closed config reload cycle inside `runRouter`, wired
end-to-end from `main.go`'s signal loop through the two-case select, through
a re-parse + re-validate sequence, through a diff against the live
`upstreamRouters` local variable, and out as a `mode=PE`/`mode=E` re-emission
to the writer. No daemon restart is required for a topology change to take
effect at the mode-signal layer.

Four composed changes:

1. **`cmd/switchboard/main.go`** — adds a dedicated `sighupCh := make(chan
   os.Signal, 1)` channel with `signal.Notify(sighupCh, syscall.SIGHUP)` +
   `defer signal.Stop(sighupCh)` in the `"router"` case body, alongside (not
   replacing) the existing `signal.NotifyContext` for SIGTERM/SIGINT. The two
   channels are orthogonal: `ctx` is cancelled by termination signals;
   `sighupCh` receives reload events. No race between them. The `runRouter`
   call is updated to pass `*configPath` and `sighupCh` as the two new
   trailing parameters (architect Q1/Q3 ruling).

2. **`cmd/switchboard/mgmt_wire.go`** — `runRouter` signature extended with
   `configPath string` and `sighupCh <-chan os.Signal`. The single-case
   `<-ctx.Done()` run loop becomes a two-case select. The `sighupCh` case
   implements the fail-closed reload sequence: `config.LoadFile(configPath)` →
   `loaded.Validate()` → `upstreamRoutersFor(loaded)` diff against local
   `upstreamRouters` → re-emit or log EC-004 failure format. The `cfg` pointer
   parameter is never mutated during reload; the only state mutations are the
   write to `w` and the assignment to the `upstreamRouters` local variable,
   both in the single `runRouter` goroutine after a fully-successful validation
   sequence. The reload path does not touch `ingressCtx`, `ingressCancel`,
   `dataWG`, `drainCoord`, `mgmtSrv`, `mgmtWG`, or the parent `ctx` (architect
   Q5 ruling).

3. **`cmd/switchboard/router_config.go`** — adds `equalStringSlices(a, b
   []string) bool` helper for the reload-diff logic in `mgmt_wire.go`.

4. **`internal/testenv/testenv.go`** — extends `RouterHandle` with a
   transitional `SetSighupCh` post-hoc setter (the `chan<- os.Signal`
   reference is wired after construction) and a `SendReloadSignal(t
   testing.TB)` 1-arg helper. The handle forwards signals on the held channel
   but does not track real router state. Construction-time wiring and
   real-state tracking are deferred to the PE-CONNECTOR-era testenv integration.
   This is an addition inside the test package only — no production surface is
   exposed (architect Q6 ruling).

## Declared Divergences

The following two intentional divergences from the story outline's stated or
implied verification shapes are declared per the declared-divergence protocol.
Each carries a class and rationale.

**Divergence 1 — AC-003 verification shape (class: not-core)**

AC-003 session non-interruption is verified via a read-probe (the open TCP
connection remains alive and readable across reload; `runRouter` does not
return) rather than a write-byte round-trip through an active session during
the reload window. The story outline's AC-003 outline states "write a byte;
read it back, or check conn state" as alternatives; the shipped test uses
connection liveness detection rather than a symmetric write-read exchange
during the reload window itself. The rationale: a symmetric write-read
through a live session during the SIGHUP window introduces a timing-coupled
race between the reload emission scan and the I/O exchange — non-deterministic
under `go test -race`. Proving that `runRouter` has not returned (goroutine
still in run loop) and that the held TCP connection has not received EOF or
connection-reset is the load-bearing behavioral property BC-2.09.001 PC-4
requires. Write-path continuity is additionally covered by drain-test call
sites remaining green across the signature change.

**Divergence 2 — `testenv.SendReloadSignal` arity (class: unmet-deps)**

The story outline's earlier description of the testenv extension implied a
2-arg `SendReloadSignal(t, cfgPath)` shape. The shipped function has 1-arg
signature `SendReloadSignal(t testing.TB)`. The config path is fixed at
`runRouter` goroutine start via the `configPath string` parameter; it is not
a runtime input to the send-signal operation. Passing it again at signal-send
time would require the testenv handle to own the router's reload path, which
is the PE-CONNECTOR-era construction-time wiring deferred obligation. Dropping
the `cfgPath` parameter was first recorded as adversary pass-1 F-011 and
formalized in story v1.5. The 1-arg shape is the correct seam for the
transitional handle (forward-signals-only, no state tracking).

## Forward Obligations → S-7.04-FU-PE-CONNECTOR

The following five obligations are anchored to S-7.04-FU-PE-CONNECTOR and are
not defects in this story's delivery. Each carries a deferral class.

**FO-1 — Order-vs-set upstream diff semantics (class: not-core)**
`equalStringSlices` performs a positional (order-sensitive) comparison. If
`config.LoadFile` returns upstream addresses in a different order than the
previous config (despite identical addresses), a spurious `mode=PE`
re-emission is triggered. The ruling on whether upstream diffs should be
set-equal (order-insensitive) or positional is deferred to PE-CONNECTOR
elaboration, where the outbound dial loop's connection-management semantics
will make the correct choice evident. Convergence summary forward obligation
O2 (6th confirmation at pass 14); adversary finding F-P1-007 first raised.

**FO-2 — `RouterHandle.Mode()` seam construction-time wiring (class: unmet-deps)**
`SetSighupCh` is a post-hoc setter on `RouterHandle` — the handle forwards
signals but does not track real router state. Construction-time wiring (the
handle owns the channel at construction, with real `Mode()` state tracking
derived from the reload path) depends on PE-CONNECTOR-era testenv integration
that has not landed. Until then, `RouterHandle.Mode()` direct assertion
remains deferred per the AC-004 PC-2 deferral note (story v1.2). Convergence
summary forward obligation O2 (Mode seam).

**FO-3 — Cross-BC Note EC-003/E-CFG-001 rendering nuance (class: keep-on-task)**
BC-2.09.001 EC-003 (invalid upstream addr on reload) is satisfied structurally
via `Validate` but renders under E-CFG-001 with E-CFG-003 detail text, not the
literal E-CFG-003 code. This cross-BC rendering nuance — how the error detail
appears in a PE-mode context and whether a BC amendment is warranted — is
deferred to S-7.04-FU-PE-CONNECTOR elaboration, which owns the PE-mode
outbound connector and will need to address E-CFG-003 rendering in that context.
Adversary pass-7 O3; convergence summary forward obligation O3.

**FO-4 — `upstreamRouters` single-goroutine assumption → concurrency design (class: unmet-deps)**
`upstreamRouters` is a local variable in the `runRouter` goroutine, safe for
single-goroutine read/write today. When the PE outbound connector lands
(S-7.04-FU-PE-CONNECTOR), the dial goroutine loop will need to read
`upstreamRouters` concurrently with the reload case that writes it. A mutex
or channel-mediated update design will be needed at that point. The
current seam — a local variable only the run loop touches — is correct for
this story's scope. Convergence summary forward obligation O4; adversary
pass-8 finding P8-O1.

**FO-5 — VP-038 full activation (class: unmet-deps)**
VP-038 ("E→PE via config-only", "Same binary runs in both E and PE modes",
"Sessions not interrupted during config reload") is discharged at seam level
by AC-004's emission-based observable. The harness-ready form (the testenv
seam can drive the reload via `SendReloadSignal`) is verified. Full production
activation — where the live `RouterHandle.Mode()` state and the PE outbound
connector evidence confirm the property at the production level, not just the
seam level — lands with S-7.04-FU-PE-CONNECTOR. Convergence summary forward
obligation O5.

## Drift Rows Parked → S-BL.CLI-SURFACE-COMPLETION

The following two drift rows are adjudicated as out of scope for this story
(class: keep-on-task) and parked at S-BL.CLI-SURFACE-COMPLETION.

**DRIFT-SIGHUP-MODE-ASYMMETRY** — `kill -HUP` terminates access, console,
and control daemon modes (default Go SIGHUP behavior — no handler registered,
so the default signal disposition applies); only the router mode handles SIGHUP
explicitly via the dedicated `sighupCh` channel. This asymmetry is a UX gap
for operators who may expect SIGHUP to be universally safe across all daemon
modes. Resolving it requires adding `signal.Notify` handling to the other
signal-loop branches in `main.go`, which is a CLI-surface concern beyond this
story's scope. Parked with justification class keep-on-task: this story's
obligation is the router reload path only; the other modes do not have a
reload concept yet.

**DRIFT-SIGHUP-INERT-RELOAD-UX** — A valid SIGHUP that changes only non-upstream
config fields (e.g., `drain_timeout`, `listen_addr`) parses and validates
successfully but produces no operator-visible output — the diff detects no
upstream change, so neither a mode re-emission nor a success confirmation is
emitted to the writer. From the operator's perspective, the signal is silently
absorbed. This is a UX gap for operator workflows that rely on visible reload
confirmation. Resolving it requires a defined format for "reload processed,
no mode change" log emission and a decision on whether non-upstream field
reloads should take effect at runtime. Parked at keep-on-task: the story's
scope is upstream-router reload and mode re-emission; a reload-acknowledgment
verb is a CLI-surface concern.

## Sentinel / Spec Impact

**Wire protocol:** unchanged. SIGHUP reload emits a mode line to stderr
(writer) only. No wire-protocol change, no config schema change, no `paths.list`
or `router.status` RPC schema change. SPEC-1..SPEC-5 sentinels (unreachable-
daemon and usage-error paths) are unaffected.

**Startup-line format:** the reload re-emission uses the identical format as
the startup emission from S-7.04 (`mode=PE upstream_routers=[...]` /
`mode=E (no upstream_routers configured)`). The format is `key=value` pairs —
stable enough for operator eyeballing; not a machine-consumed surface.

**runRouter call signature:** the two new trailing parameters (`configPath
string`, `sighupCh <-chan os.Signal`) affect all in-package test call sites.
All five sites in `mgmt_wire_test.go` and all `runRouter` calls in
`router_drain_test.go` have been updated. No change to the production binary's
CLI surface or external behavior beyond the new reload capability.
