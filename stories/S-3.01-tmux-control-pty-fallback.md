# S-3.01: Tmux Control Mode Integration with PTY Fallback

> **Execute:** `/vsdd-factory:deliver-story S-3.01`

## Narrative

- **As an** access node
- **I want to** connect to the local tmux server via control mode and publish sessions over the SVTN, falling back to PTY proxy mode when control mode is unavailable
- **So that** console operators can attach to named terminal sessions without knowing the access node's hostname, even on hosts without tmux installed

## Acceptance Criteria

### AC-001 (traces to BC-2.04.001 PC-1)
When the access node starts and tmux is available, `TmuxControlMode.Connect()` establishes an active control mode connection to the local tmux server (`tmux -C`).
- **Test:** `TestTmuxControlMode_Connect_EstablishesConnection`

### AC-002 (traces to BC-2.04.001 PC-2)
After `TmuxControlMode.Connect()` succeeds, all current tmux sessions are enumerated and published; the returned session list matches what `tmux ls` reports.
- **Test:** `TestTmuxControlMode_Connect_EnumeratesSessions`

### AC-003 (traces to BC-2.04.001 PC-3 + PC-4)
New tmux sessions created after startup are automatically discovered and published; closed sessions are automatically unpublished. Both events are processed via control mode event subscription (`%session-created` / `%session-closed`).
- **Test:** `TestTmuxControlMode_SessionLifecycleEvents`

### AC-004 (traces to BC-2.04.001 PC-5)
Session output events from control mode (`%output`) feed the downstream half-channel; at least 99% of events emitted by the tmux control mode session are delivered to the downstream stream (integration test with a real tmux session).
- **Test:** `TestTmuxControlMode_OutputEventsFeedDownstream`

### AC-005 (traces to BC-2.04.002 PC-1)
When `TmuxControlMode.Connect()` fails (tmux not found in PATH, socket absent, or `-C` flag not supported), the access node enters PTY proxy mode automatically. The fallback is triggered only at initial connection — not mid-session.
- **Test:** `TestTmuxControlMode_FallbackToPTY_OnInitialFailure`

### AC-006 (traces to BC-2.04.002 PC-2 + PC-3)
In PTY proxy mode, the access node publishes the PTY session under a synthetic name (e.g., `"pty-<pid>"`), and writes a log entry of the form `"tmux control mode unavailable; using PTY proxy mode. Functionality limited: no structured session metadata, no content-type detection."` No silent failure.
- **Test:** `TestPTYProxy_PublishesSessionAndLogs`

### AC-007 (traces to BC-2.04.002 EC-004)
When both tmux control mode and PTY device are unavailable, `Connect()` returns `E-SYS-001` ("PTY device unavailable; cannot start access node") and the access node exits with a non-zero status. The failure is never silent.
- **Test:** `TestTmuxControlMode_NoPTY_ReturnsErrSysOne`

## Edge Cases

| ID | Source | Description | Expected Behavior |
|----|--------|-------------|-------------------|
| EC-001 | BC-2.04.001 EC-001 / EC-004 | tmux not found in PATH | PTY fallback triggered; log: "tmux not found; using PTY proxy" |
| EC-002 | BC-2.04.001 EC-002 | Control mode connection drops mid-session | Reconnect attempted; if reconnect fails, NOT an automatic PTY fallback (fallback is only at initial connect per ADR-010) — session marked unavailable |
| EC-003 | BC-2.04.001 EC-003 | tmux server has no sessions on startup | Connect succeeds; session list empty; no sessions published until sessions are created |
| EC-004 | BC-2.04.002 EC-001 | tmux exists but old version (no `-CC` support) | PTY fallback; log: "tmux version does not support -CC flag" |
| EC-005 | BC-2.04.002 EC-004 | tmux unavailable AND PTY device unavailable | E-SYS-001 returned; non-zero exit; never silent |
