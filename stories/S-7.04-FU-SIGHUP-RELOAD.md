# S-7.04-FU-SIGHUP-RELOAD: SIGHUP Config-Reload for Router Daemon

## Narrative

- **As an** operator managing a running E router
- **I want to** add `upstream_routers` entries to the config file, send SIGHUP,
  and have the router graduate to PE mode without a full daemon restart
- **So that** active sessions are preserved during topology changes
  (BC-2.09.001 PC-2/PC-4)

## Context

S-7.04 delivered the application-point seam for this story: `upstreamRoutersFor(cfg)` in
`cmd/switchboard/router_config.go` returns the upstream router addresses, and `runRouter`
emits `mode=E` or `mode=PE upstream_routers=[...]` at startup. The drain coordinator
(`internal/drain`) is wired. The only missing piece is the SIGHUP handler that drives
re-emission without a daemon restart.

`cmd/switchboard/main.go` signal loop (line ~118) currently handles SIGTERM and SIGINT only,
via `signal.NotifyContext`. Adding SIGHUP requires a **dedicated `sighupCh`** channel using
`signal.Notify` (not `NotifyContext`) so the reload signal does not cancel the daemon's parent
context — which would trigger the graceful-drain shutdown path rather than a reload.

The reload must be fail-closed: a bad config file must leave the daemon on its current config
with a specific log entry (BC-2.09.003 EC-004).

S-6.04 (draft stub for SIGHUP config reload with fail-closed safety) is **closed-subsumed** by
this story as of 2026-07-06. Every obligation S-6.04 carried — BC-2.09.003 Inv-3, EC-004, and
all three of S-6.04's ACs — is fully covered by this story's ACs. See
`.factory/decisions/S-6.04-disposition-ruling.md` for the obligation-by-obligation trace.

VP-028 and VP-029, formerly traced in S-6.04, remain **proven at `internal/config`
startup-validation scope** and do not transfer here. `vp_traces` stays `[VP-038]` per the
PO disposition ruling §"VP-028 and VP-029 Disposition".

## Design Constraint: runRouter Signature Change

**Binding.** Verbatim from the architect placement note
(`.factory/decisions/S-7.04-FU-SIGHUP-RELOAD-placement-note.md` §Q2–Q3):

Current (`mgmt_wire.go:400`):

```go
func runRouter(ctx context.Context, w io.Writer, cfg *config.Config) error
```

After this story:

```go
func runRouter(ctx context.Context, w io.Writer, cfg *config.Config,
               configPath string, sighupCh <-chan os.Signal) error
```

Call site update (`main.go:120`):

```go
return runRouter(ctx, os.Stderr, cfg, *configPath, sighupCh)
```

All existing `runRouter` callers in tests must add the two new parameters. For tests that do
not exercise reload: `configPath: ""` and `sighupCh: make(chan os.Signal, 1)` (channel never
written; the reload path only reads `configPath` when SIGHUP is received, so an empty path
during a non-reload test triggers no I/O).

The `sighupCh` channel is buffered (capacity 1) per `signal.Notify` best practice — a missed
SIGHUP while a reload is in progress is silently absorbed. Created in the `"router"` case body
of `main.go`, after the existing `signal.NotifyContext` call, alongside its cleanup:

```go
sighupCh := make(chan os.Signal, 1)
signal.Notify(sighupCh, syscall.SIGHUP)
defer signal.Stop(sighupCh)
```

The existing `signal.NotifyContext` at `main.go:118` is **not modified**. The two channels are
orthogonal: `ctx` is cancelled by termination signals; `sighupCh` receives reload events. No
race between them.

## Acceptance Criteria

### AC-001 — SIGHUP capture and conditional mode re-emission

**BC Anchors:** BC-2.09.001 PC-1 (reload → PE mode when UpstreamRouters changes), EC-002
(sessions maintained during config reload)

**Precondition:** `runRouter` is executing in E mode (`upstreamRouters` local variable is
empty). A valid YAML config file with one or more `upstream_routers` entries is written to a
`t.TempDir()` path. `configPath` names that path.

**Postconditions:**

1. `sighupCh <- syscall.SIGHUP` causes the reload select-case to execute.
2. `config.LoadFile(configPath)` returns a valid `*Config`; `loaded.Validate()` returns nil.
3. `upstreamRoutersFor(loaded)` returns a non-empty slice.
4. `fmt.Fprintf(w, ...)` writes a `mode=PE upstream_routers=[...]` line to the writer,
   identical in format to the startup emission from S-7.04.
5. The `upstreamRouters` local variable in `runRouter` is updated to the new slice.
6. The `cfg` pointer parameter is **not mutated** — the reload operates on the fresh `loaded`
   struct; `cfg` remains the startup-time config throughout.
7. `runRouter` has **not returned**; the daemon continues executing the run loop.

**Atomicity:** The only state mutations are (4) the write to `w` and (5) the assignment to the
`upstreamRouters` local variable. Both occur in the single `runRouter` goroutine, after a
fully-successful LoadFile + Validate + diff sequence. No partial application is possible
because the old `upstreamRouters` value is replaced only after the new config is validated.

**Test name:** `TestRunRouter_SIGHUPReload_EtoPE`
**Test level:** integration
**Test file:** `cmd/switchboard/router_sighup_test.go`

### AC-002 — Fail-closed reload on bad config (BC-2.09.003 EC-004)

**BC Anchors:** BC-2.09.003 Inv-3 (config reload subject to same validation as startup; bad
reload leaves daemon on previous config), EC-004

**Precondition:** `runRouter` is running. A YAML config file that fails `Config.Validate`
(e.g., `listen_addr: "notaport"` which triggers E-CFG-002) is on disk at `configPath`.

**Postconditions:**

1. `sighupCh <- syscall.SIGHUP` causes the reload select-case to execute.
2. Either `config.LoadFile(configPath)` or `loaded.Validate()` returns a non-nil error.
3. The daemon logs exactly the following to `w` (verbatim from BC-2.09.003 EC-004):
   ```
   config reload failed: <err>; continuing with previous config
   ```
   where `<err>` is the error returned by `LoadFile` or `Validate`.
4. The `cfg` pointer parameter is **not mutated**; the `upstreamRouters` local variable is
   **unchanged**.
5. `runRouter` has **not returned**; the daemon continues on the previous config.
6. No sessions are disrupted (ingress listener, mgmt server, and all accepted connections
   remain serving).

**E-CFG-001 taxonomy note:** When `loaded.Validate()` fails, the error it returns already
carries E-CFG-001 format text (`"config error: <field>: <problem>. Fix: <suggestion>"`). The
reload log wraps that error as `<err>` inside the EC-004 message format, so the E-CFG-001
taxonomic information is preserved inside the logged string. The outer wrapper message is the
EC-004 reload-failure format — not the bare E-CFG-001 startup-exit format. Similarly, a
`LoadFile` I/O error (E-CFG-004/E-CFG-005) is wrapped verbatim as `<err>`.

**Test name:** `TestRunRouter_SIGHUPReload_BadConfig_FailClosed`
**Test level:** integration
**Test file:** `cmd/switchboard/router_sighup_test.go`

### AC-003 — Active session non-interruption during valid reload

**BC Anchor:** BC-2.09.001 PC-4 (active sessions not interrupted during config reload)

**Precondition:** `runRouter` is running with a live TCP connection dialed to the ingress
listener (held open by the test). A SIGHUP with a valid PE config is sent.

**Postconditions:**

1. After SIGHUP completes and the `mode=PE` line is emitted, the open TCP connection is **not
   closed** by the daemon (no EOF or connection-reset on the test-held conn).
2. The following constructs are **not touched** by the reload path (per architect placement
   note §Q5):
   - `ingressCtx` / `ingressCancel` (`mgmt_wire.go:~459`) — cancelling tears down the
     data-plane accept loop and closes all accepted connections
   - `dataWG` (`mgmt_wire.go:~464`) — waiting on it during reload would stall active sessions
   - `drainCoord` (`mgmt_wire.go:~489`) — signalling triggers a drain broadcast; reload is
     not a shutdown event
   - `mgmtSrv` / `mgmtWG` (`mgmt_wire.go:~450-451`) — management server continues serving
   - parent `ctx` (the `runRouter` parameter) — cancelling this initiates graceful shutdown
3. Only `upstreamRouters` (local variable) and a write to `w` (the log writer) are affected
   by the reload path. `os.Stderr` is safe for concurrent writes; startup-emission and reload-
   emission may interleave on the log stream without data loss.

### AC-004 — VP-038 activation: E→PE via SIGHUP without process restart

**VP Anchor:** VP-038 ("PE mode activated when upstream_routers is non-empty"; "Same binary
runs in both E and PE modes"; "Sessions not interrupted during config reload")

**Precondition:** A `testenv.RouterHandle` is started (E mode, no upstream routers). The
`RouterHandle` holds a reference to the `sighupCh` used by the underlying `runRouter` call,
exposed via a test-only accessor.

**Postconditions:**

1. After writing a PE config to disk and sending on the `sighupCh`, the handler observes
   `mode=PE` emitted to the writer.
2. `env.RouterHandle.Mode()` returns `testenv.ModePE` without the `runRouter` goroutine
   having returned (no process restart).
3. The test does **not** call `syscall.Kill` or issue a real OS signal; the in-process
   `sighupCh` channel injection is the test seam (same pattern used by
   `TestRunRouter_ForcedExitPastDrainTimeout` via `drainCoordHook`).

**testenv extension required (test package only, no production surface change):**
`testenv.RouterHandle` must hold a `chan<- os.Signal` reference set when the handle is
constructed (the same channel passed as `sighupCh` to the underlying `runRouter` call). A
`SendReloadSignal(t testing.TB, cfgPath string)` helper (or equivalent) writes the config path
to `RouterHandle`'s internal state and sends `syscall.SIGHUP` on the held channel, allowing
tests to drive the reload seam without managing a separate channel outside the rig. This is
an addition inside `internal/testenv/testenv.go` — not a production surface change.

Existing `testenv.RouterHandle.Restart(t, cfg)` at `testenv.go:269` reconfigures the in-process
mode by writing to `r.mode` directly. The VP-038 test must drive the mode change via `sighupCh`
(the signal path), not `Restart` — to prove the signal handler fires. `Restart` may be used
to synchronise the handle's mode view after the reload emission is observed in the writer.

failure leaves daemon on previous config". This is an overstatement: VP-028 and VP-029 are
proven against `internal/config`'s `Config.Validate` startup-validation behavior and their

daemon-level reload integration logic.

This is pre-existing authoring drift in the BC, identified in:
`.factory/decisions/S-6.04-disposition-ruling.md`
§"Spec Note: BC-2.09.003 Verification Properties Table Drift"

Remediation (narrowing the BC-2.09.003 VP table row, or adding a new VP for the reload
integration property) is being handled by the spec-steward in the same burst as this
elaboration. No change to VP-028 or VP-029 documents is required by this story.

The fail-closed reload obligation (BC-2.09.003 Inv-3/EC-004) is satisfied by the AC-002
integration test in this story, which is a behavioral integration test at the daemon level —
distinct from the unit-level startup-validation property covered by VP-028/VP-029.

### Task 3 — Green: implement reload dispatch

Replace the `case <-sighupCh: // TODO` stub with the fail-closed reload sequence:

```go
case <-sighupCh:
    loaded, err := config.LoadFile(configPath)
    if err != nil {
        fmt.Fprintf(w, "config reload failed: %s; continuing with previous config\n", err)
        continue
    }
    if err := loaded.Validate(); err != nil {
        fmt.Fprintf(w, "config reload failed: %s; continuing with previous config\n", err)
        continue
    }
    newUpstreams := upstreamRoutersFor(loaded)
    // diff: re-emit only if the upstream list changed
    if !equalStringSlices(upstreamRouters, newUpstreams) {
        upstreamRouters = newUpstreams
        if len(upstreamRouters) > 0 {
            fmt.Fprintf(w, "mode=PE upstream_routers=%v\n", upstreamRouters)
        } else {
            fmt.Fprintf(w, "mode=E (no upstream_routers configured)\n")
        }
    }
```

Note: `cfg` (the `runRouter` parameter) is **never mutated**. All reload operations are
against `loaded` (a fresh struct from `LoadFile`). The `upstreamRouters` local variable (the
resolved `[]string` from the startup call to `upstreamRoutersFor(cfg)`) is the only piece of
state that changes — and only after full validation succeeds.

Gate: all four new tests green; all pre-existing tests remain green; `just test-race` clean;
