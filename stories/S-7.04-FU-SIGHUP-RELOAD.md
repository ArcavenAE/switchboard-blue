# S-7.04-FU-SIGHUP-RELOAD: SIGHUP Config-Reload for Router Daemon

> **STATUS: BACKLOG STUB.** This story is the S-7.04 AC-001 residual. Acceptance
> criteria, file structure, and task list will be fleshed out when the story is
> scheduled. The ACs below are sketched to define scope and BC traceability.

## Narrative

- **As an** operator managing a running E router
- **I want to** add `upstream_routers` entries to the config file, send SIGHUP,
  and have the router graduate to PE mode without a full daemon restart
- **So that** active sessions are preserved during topology changes
  (BC-2.09.001 PC-2/PC-4)

## Context

S-7.04 delivered the application-point seam for this story: `upstreamRoutersFor(cfg)`
in `cmd/switchboard/router_config.go` returns the upstream router addresses, and
`runRouter` emits `mode=E` or `mode=PE upstream_routers=[...]` at startup. The drain
coordinator (`internal/drain`) is wired. The only missing piece is the SIGHUP handler
that drives re-emission without a daemon restart.

`cmd/switchboard/main.go` signal loop currently handles SIGTERM and SIGINT only.
Adding SIGHUP requires signal capture and a reload-dispatch path into runRouter.
The reload must be fail-closed: a bad config file must leave the daemon on its current
config (BC-2.09.003 EC-004).

`S-6.04` is a draft stub for SIGHUP config reload with fail-closed safety. This story
subsumes S-6.04's scope for the router daemon; whether S-6.04 is closed or narrowed is
decided at scheduling time.

## Sketched Acceptance Criteria

> ACs are illustrative. Exact scope, test names, and BC postcondition references will
> be confirmed at scheduling time by the product-owner and architect.

**AC-001 (BC-2.09.001 PC-1):** Signal loop captures SIGHUP. On receipt: re-parses
config from disk and validates. If config is valid and `UpstreamRouters` has changed
(added or modified), `runRouter` re-emits mode startup line reflecting the new
PE/E status without restarting the daemon or dropping existing connections.

**AC-002 (BC-2.09.003 EC-004):** If the reloaded config fails `Config.Validate`,
the daemon logs the error in E-CFG-001 format and continues running on the previous
config. The reload is atomic: either the new config is fully applied or the old
config is preserved. No partial application.

**AC-003 (BC-2.09.001 PC-4):** Active sessions are not interrupted during a valid
config reload. The mode-emission seam fires on the writer; existing net-ingress
goroutines and data waitgroups are not cancelled.

**AC-004 (VP-038):** Integration test asserts that after a SIGHUP with a config that
adds upstream_routers, the daemon emits `mode=PE` on the writer without terminating
the process. Pairs with the existing `TestRunRouter_EmitsUpstreamModePE` unit test
which asserts the seam at startup.

## Non-Goals

- Does not implement the outbound PE connector (TCP dial loop). That is `S-7.04-FU-PE-CONNECTOR`.
- Does not implement per-session SIGHUP liveness or drain-during-reload. The reload
  is for mode-transition emission only; the full drain-on-PE-shutdown is `S-7.04-FU-DRAIN-WIRE`.
- Does not change Config.Validate logic — that is already correct.
