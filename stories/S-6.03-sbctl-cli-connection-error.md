# S-6.03: sbctl Client Auth, Flag Parsing, JSON Envelope, and Connection Error Reporting

> **Execute:** `/vsdd-factory:deliver-story S-6.03`

## Scope Note (v2.0 re-scope)

This story implements the **client side** of the ADR-012 management plane in
`cmd/sbctl` only. The daemon-side management server (`internal/mgmt`) is
implemented by S-W5.01. The E2E integration harness is S-W5.02. This story
does NOT need the server to exist — `Authenticate()` is unit-tested with a
`net.Pipe` mock server.

**Scope in:** `cmd/sbctl/main.go` (flag parsing, subcommand routing, JSON
envelope output), `cmd/sbctl/client.go` (connection, `Authenticate()`, RPC
dispatch).

**Scope out:** `internal/mgmt` (S-W5.01); e2e across all four daemon types
(S-W5.02).

## Narrative

- **As an** operator
- **I want to** use a single `sbctl` binary with OpenSSH key authentication that
  gracefully handles connection failures and authentication failures
- **So that** I can operate the Switchboard management plane securely and get
  clear, actionable errors when something goes wrong

## Acceptance Criteria

### AC-001 (traces to BC-2.07.002 postcondition 2 — OpenSSH key load)
`sbctl --key <path>` loads an Ed25519 private key from an OpenSSH PEM file using
`golang.org/x/crypto/ssh.ParsePrivateKey`. The key file is read with
`io.LimitReader(f, 1<<16)` (64 KiB limit). The private key is extracted as
`ed25519.PrivateKey` and passed to `Authenticate()`. The key is never logged
or serialized.
- **Test:** `TestSbctl_KeyLoading_Ed25519` — verify that a well-formed Ed25519
  OpenSSH private key file is successfully loaded and the resulting
  `ed25519.PrivateKey` has the correct length (64 bytes). Verify that a file
  larger than 64 KiB is rejected with an error.

### AC-002 (traces to BC-2.07.002 postcondition 2 — Authenticate() fail-closed, VP-067; Ruling 2)
`Authenticate(ctx context.Context, conn net.Conn, privKey ed25519.PrivateKey) error`
in `cmd/sbctl/client.go` is fail-closed per ADR-012 §Authenticate() FAIL-CLOSED
Contract. `context.Context` is always the first parameter (go.md rule 7). The
function derives the read deadline from the context:
1. Compute effective deadline: if `ctx.Deadline()` is set use it; else
   `time.Now().Add(mgmt.HandshakeTimeout)`. Call `conn.SetReadDeadline(deadline)`.
2. Read CHALLENGE message via `json.NewDecoder(io.LimitReader(conn, 1<<16))`.
3. Decode `nonce_bytes` from the base64url `nonce` field; return error if absent
   or not valid base64url-encoded 32 bytes.
4. Send `{"type":"challenge_response","nonce_sig":"<base64url sig>","pubkey":"<base64url pubkey>"}`.
5. Read AUTH_OK or AUTH_FAIL response (deadline still active).
6. Return `nil` **only** if `{"type":"auth_ok"}` was received and decoded.
7. Return non-nil error for: connection error, malformed CHALLENGE, AUTH_FAIL,
   deadline exceeded, any other response type, truncated stream, oversized response.

The `daemon_sig` field in the CHALLENGE is decoded but its signature is NOT
verified in MVP. This is a documented, spec-permitted deferral: trust-on-first-use
for MVP; verification deferred to a post-MVP hardening pass (ARCH-12 §Authenticate()
FAIL-CLOSED Contract step 4 note). This deferral is explicit, not a silent drop.

The call site in `cmd/sbctl/main.go` passes a context with the `--timeout` deadline:
```go
ctx, cancel := context.WithTimeout(context.Background(), timeout)
defer cancel()
conn, err := net.DialContext(ctx, "unix", target)
// ...
err = Authenticate(ctx, conn, privKey)
```
- **Test:** `TestAuthenticate_FailClosed_VP067` — table-driven test covering
  all failure sub-cases from VP-067.md plus the happy-path AUTH_OK. Each sub-case
  uses `net.Pipe` as the transport. Sub-cases MUST include:
  (a) happy-path AUTH_OK → nil; (b) AUTH_FAIL response → non-nil; (c) malformed
  CHALLENGE JSON → non-nil; (d) missing `nonce` field → non-nil; (e) oversized
  CHALLENGE (> 64 KiB) → non-nil; (f) connection closed mid-handshake → non-nil;
  (g) wrong `"type"` in response → non-nil; (h) oversized AUTH response → non-nil;
  **(i) deadline expiry — server sends nothing; verify non-nil error returned
  (uses a context with a 50 ms timeout).**

### AC-003 (traces to BC-2.07.002 postcondition 4 — auth failure exit; BC-2.07.003 EC-005 — key load failure, Ruling 5)
When `Authenticate()` returns a non-nil error (any cause), sbctl prints
`E-ADM-010 "authentication failed"` to stderr and exits with code 1. No stdout
output is produced.

When the key file at `--key` does not exist, is larger than 64 KiB, is malformed
(not valid OpenSSH PEM), or contains a non-Ed25519 key type, sbctl prints
`E-CFG-010 "key load failed: <path>: <reason>"` to stderr and exits 1. **No
connection attempt is made** — the key is loaded and validated before any `net.Dial`.
- **Test:** `TestSbctl_AuthFailure_ExitsOneWithEADM010` — mock server that
  sends AUTH_FAIL; verify stderr contains "E-ADM-010" and exit code is 1.
- **Test:** `TestSbctl_KeyLoadFailure_ExitsOneWithECFG010` — supply a `--key`
  path that does not exist; verify stderr contains "E-CFG-010" and exit code is 1;
  verify no TCP/Unix dial was attempted (e.g., target a non-listening address and
  confirm the E-CFG-010 error fires before E-NET-001 could).

### AC-004 (traces to BC-2.07.003 postcondition 1 — E-NET-001 connection error, VP-030; BC-2.07.003 EC-006 — RPC dispatch failure, Ruling 5)
When `sbctl` cannot connect to the daemon (connection refused, socket absent,
timeout), it prints `E-NET-001 "daemon unreachable: <address>: <reason>"` to
stderr and exits with code 1. No stdout output is produced.

**E-NET-001 scope (Ruling Y — BC-2.07.003 Inv-4 v1.5 reconciliation):** `E-NET-001`
is explicitly correct for TWO cases:
(a) `net.Dial`/`net.DialContext` failure — daemon connection refused or DNS failure;
(b) handshake read-deadline timeout — the daemon accepted the TCP connection but did
not complete the ADR-012 challenge-response handshake within the timeout budget
(treated as unreachable per BC-2.07.003 Inv-2). Message format for case (b):
`"daemon unreachable: <address>: connection timed out"`. This reconciles the prior
Inv-2 vs. Inv-4 conflict: both point the operator to the right target.
`E-NET-001` MUST NOT be emitted for key-load failures (E-CFG-010) or post-auth
(post-AUTH_OK) dispatch failures (E-RPC-001) — those failure modes have their own
codes and MUST NOT share E-NET-001.

When authentication succeeded (AUTH_OK received) but the subsequent RPC request
fails (server returns `"ok":false`, response decode fails, or connection drops
mid-RPC), sbctl prints `E-RPC-001 "rpc failed: <command>: <reason>"` to stderr
and exits 1. No stdout output is produced.
- **Test:** `TestSbctl_ConnectionRefused_ExitsOneWithENET001_VP030` — target
  a non-listening address; verify stderr contains "E-NET-001" and exit code 1.
- **Test:** `TestSbctl_RPCDispatchFailure_ExitsOneWithERPC001` — mock server that
  completes the AUTH_OK handshake then returns `{"ok":false,"error":{"code":"E-RPC-001","message":"rpc failed: router.status: unknown command"}}`; verify
  sbctl exits 1 and stderr contains "E-RPC-001".

### AC-005 (traces to BC-2.07.003 postcondition 3 — no partial output)
When the daemon is unreachable, sbctl produces zero stdout output. The error
message appears only on stderr.
- **Test:** `TestSbctl_NoStdoutOnConnectionFailure` — verify that stdout is
  empty when connection is refused.

### AC-006 (traces to BC-2.07.002 postcondition 3 — JSON envelope output)
All sbctl subcommands accept `--json` flag and produce output conforming to the
JSON envelope from `interface-definitions.md §JSON Output Schema`:
`{"ok":true,"error":null,"data":{...}}` on success,
`{"ok":false,"error":{"code":"E-xxx","message":"..."},"data":null}` on failure.
- **Test:** `TestSbctl_JSONEnvelopeFormat` — verify that `--json` produces a
  valid JSON object with `ok`, `error`, and `data` fields for both success and
  error outcomes.

### AC-007 (traces to BC-2.07.003 invariant 2 — connection timeout)
`sbctl` does not hang indefinitely when the daemon is unreachable. The connection
timeout defaults to 5 seconds and is configurable via `--timeout`. After timeout,
sbctl exits with E-NET-001.
- **Test:** `TestSbctl_ConnectionTimeout` — use a listener that accepts but
  never reads; set `--timeout=100ms`; verify sbctl exits with E-NET-001 within
  500ms.

### AC-008 (traces to BC-2.07.003 EC-007 + Precondition 3 — tilde/home expansion for `--key` path)
The `--key` path (default `~/.ssh/id_ed25519`) MUST have leading `~/` expanded via
`os.UserHomeDir()` **before** any file-open or dial attempt. `~username` (other-user)
forms are explicitly out of scope and are treated as literals. Two distinct failure
sub-cases apply (BC-2.07.003 EC-007):

- **(a) `os.UserHomeDir()` itself returns an error:** E-CFG-010 is emitted with the
  **original (unexpanded) path** in the message — `"key load failed: ~/.ssh/id_ed25519:
  <reason>"`. No connection attempt is made.
- **(b) Expansion succeeded but the file cannot be read (absent, oversized, malformed,
  wrong key type):** E-CFG-010 is emitted with the **expanded path** — `"key load
  failed: /home/operator/.ssh/id_ed25519: <reason>"`. No connection attempt is made.

The expansion check and key load MUST precede any `net.Dial` call (ordering requirement
from BC-2.07.003 Precondition 3 and EC-007).

- **Test:** `TestSbctl_TildeExpansion_DefaultKey` — table-driven, covering:
  1. Happy path: mock `os.UserHomeDir()` returning a temp dir; place a valid Ed25519
     key at `<tempdir>/.ssh/id_ed25519`; call `loadEd25519Key("~/.ssh/id_ed25519")`;
     verify the key loads successfully (no E-CFG-010) and the expanded path was used.
  2. Sub-case (a): mock `os.UserHomeDir()` returning an error; call
     `loadEd25519Key("~/.ssh/id_ed25519")`; verify E-CFG-010 is returned and the
     error message contains the **original** path `~/.ssh/id_ed25519`.
  3. Sub-case (b): mock `os.UserHomeDir()` returning a temp dir; do NOT create the
     key file; call `loadEd25519Key("~/.ssh/id_ed25519")`; verify E-CFG-010 is
     returned and the error message contains the **expanded** path
     `<tempdir>/.ssh/id_ed25519`.
  4. `~username` literal: `loadEd25519Key("~root/.ssh/id_ed25519")` — verify the path
     is used verbatim (no expansion attempted), producing E-CFG-010 with the literal
     path `~root/.ssh/id_ed25519` in the message (out-of-scope form).

### AC-009 (architecture compliance — os.Exit only in main(), Ruling 5)
`connectAndRun` (or equivalent dispatch function) in `cmd/sbctl/client.go` MUST
return an `error`; it MUST NOT call `os.Exit` directly. Only `main()` maps errors
to exit codes and calls `os.Exit`. This preserves testability and complies with
go.md rule "No `log.Fatal` / `os.Exit` outside `main()`".
- **Test:** `TestSbctl_ConnectAndRun_ReturnsError` — call `connectAndRun` (or
  equivalent) in-process with a mock that returns an auth failure; verify it returns
  a non-nil error rather than calling `os.Exit`. (Verify by inspection + test — if
  `os.Exit` were called in-test the test process would exit, making the test
  self-evidently broken.)

### AC-010 (traces to ADR-012 §3 step 6 / BC-2.07.002 PC-3 — RPC envelope wire type, Ruling M)
The `dispatch()` function in `cmd/sbctl/client.go` MUST set `rpcRequestMsg.Type`
to `"request"` (not `"rpc_request"` or any other string). The server (`internal/mgmt`)
expects `"type":"request"` per ADR-012 §3 step 6 and closes the connection on any other
type string after authentication — a wire-type mismatch silently causes every RPC to
fail with a connection-closed E-RPC-001 decode error on the client side.

The server response envelope carries `"type":"response"`. The `dispatch()` function
MUST accept a `{"type":"response",...}` server response as a successful RPC reply.
`"rpc_request"` and `"rpc_response"` are NOT valid wire types and MUST NOT appear in
any `dispatch()` implementation.

This is a **one-line fix** in `cmd/sbctl/client.go`:
```go
// BEFORE (wrong — server rejects silently after auth):
Type: "rpc_request",
// AFTER (correct per ADR-012 §3 step 6):
Type: "request",
```

**Receiving-side wire-type validation (Ruling U — ARCH-12 v1.5 / BC-2.07.002 PC-3):**
`dispatch()` MUST validate `resp.Type == "response"` after decoding. Any other value
(e.g., `"rpc_response"`, `"auth_fail"`, `""`) is rejected with a non-nil error
containing the received type string (E-RPC-001). A wrong-type response with `"ok":true`
MUST NOT be silently accepted — the type check takes precedence over the ok-flag check.
The canonical guard (before the `if !resp.OK` check):
```go
if resp.Type != "response" {
    return nil, fmt.Errorf("rpc failed: %s: unexpected response type %q (want \"response\")", command, resp.Type)
}
```
The existing fixture in `TestSbctl_RPCDispatchFailure_ExitsOneWithERPC001` MUST be
corrected: change the mock server response from `"type":"rpc_response"` to
`"type":"response"` — the test exercises the `"ok":false` path, which only works with
a valid response type.

**Non-constant request ID and echo check (Ruling X — ARCH-12 v1.5 / BC-2.07.002 PC-3 /
ADR-012 §3 step 6):**
`dispatch()` MUST generate a non-constant request `id` (e.g., `fmt.Sprintf("%x", time.Now().UnixNano())`).
After decoding the response, `dispatch()` MUST verify `resp.ID == req.ID`. A mismatch
returns E-RPC-001. For a single-RPC-per-connection client the practical impact is low,
but the spec-mandated echo check is explicitly required per ADR-012 §3 step 6.

- **Test:** `TestDispatch_EmitsCorrectWireType` — use `net.Pipe` as transport; call
  `dispatch()` with a mock server side that reads the raw JSON bytes from the pipe;
  assert `strings.Contains(rawJSON, `"type":"request"`)`. Verify the mock returns
  `{"type":"response","id":"<id>","ok":true,"data":{}}` (echoing the request ID) and
  `dispatch()` decodes it as a successful call (non-nil data, nil error). This test
  fails before the fix and passes after.
- **Test:** `TestDispatch_AcceptsResponseType` — mock server sends
  `{"type":"response","id":"1","ok":true,"data":{"status":"ok"}}`; verify `dispatch()`
  returns non-nil data and nil error (confirming `"type":"response"` is the accepted
  server wire type).
- **Test:** `TestDispatch_RejectsWrongResponseType` (Ruling U) — mock server sends
  `{"type":"rpc_response","id":"1","ok":true,"data":{}}`; verify `dispatch()` returns
  a non-nil error containing `"unexpected response type"`. Verify that a correct-type
  response with `"ok":false` produces a different error (E-RPC-001 from the ok-false
  path, not the type-mismatch path).
- **Test:** `TestDispatch_IDEchoEnforced` (Ruling X) — mock server sends a response
  with a different `id` than the request (e.g., request sends `id="abc"`, server
  responds with `id="xyz"`); verify `dispatch()` returns a non-nil error.

### AC-011 (traces to BC-2.07.003 Invariant 2, ADR-012 §7, Ruling V — dispatch read deadline + ctx-first signature)
`dispatch()` in `cmd/sbctl/client.go` MUST accept `context.Context` as its first
parameter (go.md rule 7). Before decoding the RPC response, `dispatch()` MUST call
`conn.SetReadDeadline` derived from `ctx.Deadline()` — or 30 s as fallback if the
context has no deadline (symmetric with `HandshakeTimeout`/`RPCIdleTimeout` on the
server side). The deadline is cleared via `defer` after `dispatch` returns.
All call sites (`connectAndRun`) MUST pass `ctx`.

This closes the residual concern from ARCH-12 §Ruling F (v1.4): a caller passing
`context.Background()` (no deadline) would have no deadline on the RPC response
phase after auth completes. With this fix, `dispatch()` explicitly re-arms the
deadline for every caller.

Canonical signature:
```go
// BEFORE:
func dispatch(conn net.Conn, command string, args any) (json.RawMessage, error)
// AFTER (go.md rule 7 — ctx first):
func dispatch(ctx context.Context, conn net.Conn, command string, args any) (json.RawMessage, error)
```
Canonical read-deadline block (before the response `Decode`):
```go
responseDeadline, ok := ctx.Deadline()
if !ok {
    responseDeadline = time.Now().UTC().Add(30 * time.Second)
}
if err := conn.SetReadDeadline(responseDeadline); err != nil {
    return nil, fmt.Errorf("rpc failed: %s: set read deadline: %w", command, err)
}
defer func() { _ = conn.SetReadDeadline(time.Time{}) }()
```
(Cites BC-2.07.003 Invariant 2 / Ruling V, ARCH-12 v1.5.)
- **Test:** `TestDispatch_RespReadDeadlineEnforced` — create a mock server that
  completes AUTH_OK then hangs (never writes the RPC response); pass a context with
  a 50 ms deadline to `dispatch()`; verify `dispatch()` returns a non-nil error within
  ~200 ms (deadline fires, not blocked indefinitely).

**Test-quality note (Ruling W — ARCH-12 v1.5):**
Three sites in `cmd/sbctl/client_test.go` call `t.Fatalf` (or helpers that call it)
from inside `go func()` goroutines:
1. `VP067_g` — `wellFormedChallenge(t)` called inside the mock server goroutine.
2. `VP067_h` — same pattern.
3. `TestAuthenticate_PrivKeyNeverTransmitted` — `freshNonce(t)` inside `go func()`.

`t.Fatalf` calls `runtime.Goexit()` on the calling goroutine. When called from a
non-test goroutine, it exits that goroutine — NOT the test goroutine — leaving the
test in an indeterminate state (may appear to hang or give a false pass).

**Test-writers MUST** hoist all calls to helpers that may invoke `t.Fatalf`/`t.Fatal`/
`t.Error`/`t.Log` into the test goroutine BEFORE spawning `go func()`, capturing the
results by value, then passing them into the goroutine as parameters or via closure
over already-computed values. Example:
```go
// AFTER (correct):
challenge := wellFormedChallenge(t)  // t.Fatalf fires in test goroutine if this fails
go func(c challengeMsg) {
    _ = json.NewEncoder(conn).Encode(c)
}(challenge)
```
No production code change is required.

### AC-012 (traces to BC-2.07.002 EC-003 — no-subcommand / --help → exit 0)
When `sbctl` is invoked with no subcommand arguments (i.e., `flag.Args()` returns an empty slice), it prints the usage/help text to **stdout** and exits **0**. When `--help` or `-h` is passed, sbctl explicitly redirects `flag.CommandLine` output to `os.Stdout` (via `flag.CommandLine.SetOutput(os.Stdout)` before `flag.Parse`), so the Go `flag` package's help text is printed to stdout and the process exits 0. The no-subcommand path MUST NOT print to stderr and MUST NOT exit 2.

The `unknown subcommand` path (a non-empty first argument that is not a registered subcommand) is unaffected — it correctly prints to stderr and exits 2; that behaviour is not changed by this AC.

- **Test:** `TestSbctl_NoSubcommand_ExitsZero` — invoke the no-subcommand branch with no positional arguments; verify exit code is 0 and at least one line of help text is written to stdout (not stderr).
- **Test:** `TestSbctl_HelpFlag_ExitsZeroStdout` — invoke sbctl with `--help` and separately with `-h`; verify exit code is 0, stdout is non-empty, and stderr is empty for both invocations.
