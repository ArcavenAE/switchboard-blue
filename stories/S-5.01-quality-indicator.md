# S-5.01: Green/Yellow/Red Quality Indicator with Hysteresis

> **Execute:** `/vsdd-factory:deliver-story S-5.01`

## Narrative

- **As a** console operator
- **I want to** see a green/yellow/red quality indicator that reflects actual path latency and loss
- **So that** I know immediately when my session quality is degraded and when it recovers

## Acceptance Criteria

### AC-001 (traces to BC-2.06.001 postcondition 1)
`QualityIndicator.Update(rtt_ms, loss_pct)` returns `Green` when `rtt_ms ≤ 100` and `loss_pct ≤ 5`, `Yellow` when `rtt_ms ≤ 500` and `loss_pct ≤ 20`, and `Red` otherwise (thresholds per NFR-001).
- **Test:** `TestQualityIndicator_ThresholdClassification`

### AC-002 (traces to BC-2.06.001 invariant 1)
Transitions require 3 consecutive measurements below the threshold before upgrading (hysteresis = 3). A single good measurement after sustained degradation does not upgrade the indicator.
- **Test:** `TestQualityIndicator_HysteresisUpgrade`

### AC-003 (traces to BC-2.06.002 postcondition 1)
A missing expected frame within the tick deadline is registered as a degradation signal via `QualityIndicator.OnMissingFrame()`. After 3 consecutive missing-frame events, the indicator downgrades.
- **Test:** `TestQualityIndicator_MissingFrameDowngrade`

### AC-004 (traces to BC-2.06.002 postcondition 2)
Degradation always goes down (green → yellow → red, never skip steps in downgrade direction); upgrade can only happen after hysteresis of 3 consecutive measurements at the new level.
- **Test:** `TestQualityIndicator_DegradationOnlyGoesDown`
