# S-6.01: Config Parsing and Validation with Actionable Startup Errors

> **Execute:** `/vsdd-factory:deliver-story S-6.01`

## Narrative

- **As a** router operator
- **I want to** receive a clear, actionable error message when my config file has invalid settings
- **So that** I can fix the problem immediately without having to read source code or logs

## Acceptance Criteria

### AC-001 (traces to BC-2.09.003 postcondition 1)
`Config.Validate()` returns a descriptive error identifying the invalid field and its value when `tick_interval` is outside the range [5ms, 50ms].
- **Test:** `TestConfigValidate_RejectsOutOfRangeTickInterval`

### AC-002 (traces to BC-2.09.003 postcondition 2 and invariant 4; the exhaustive all-fields reporting is a story-level strengthening of the E-CFG-001 contract)
`Config.Validate()` collects and reports ALL validation errors together — including E-CFG-001 (missing required fields), E-CFG-002 (invalid listen_addr), E-CFG-003 (invalid upstream_routers addr), E-CFG-006 (non-positive drain_timeout), and E-CFG-007 (non-positive keepalive_interval) — when multiple fields are invalid simultaneously (Inv-4: exhaustive reporting).
- **Test:** `TestConfigValidate_RejectsMissingRequiredFields`

### AC-003 (traces to BC-2.09.003 postconditions 1 and 2)
The router daemon exits with `E-CFG-001` and prints the validation error to stderr when `Config.Validate()` fails at startup. Exit code is 1.
- **Test:** `TestRouterStartup_ExitsWithActionableError`

### AC-004 (traces to BC-2.09.003 invariant 1)
`Config.Validate()` is called before any network sockets are opened; no partial initialization occurs on invalid config.
- **Test:** `TestConfigValidate_BeforeSocketOpen`

### AC-005 (traces to BC-2.09.003 postcondition 5)
`Config.Validate()` rejects a `listen_addr` that is not a valid `host:port` (as determined by `net.ResolveTCPAddr` or equivalent) and returns an error matching E-CFG-002 format: `"config error: listen_addr: '<value>' is not a valid host:port. Fix: use '<ip>:<port>' format, e.g. '0.0.0.0:9090'"`, naming the invalid value.
- **Test:** `TestConfigValidate_RejectsInvalidListenAddrFormat`

### AC-006 (traces to BC-2.09.003 postcondition 6)
`Config.Validate()` rejects any `upstream_routers` entry whose `addr` is not a valid `host:port` and returns an error matching E-CFG-003 format: `"config error: upstream_routers[<N>].addr: '<value>' is not a valid host:port. Fix: use '<ip>:<port>' format, e.g. '10.0.0.1:9090'"`, naming the array index (0-based) and invalid value.
- **Test:** `TestConfigValidate_RejectsInvalidUpstreamRouterAddr`

### AC-007 (traces to BC-2.09.003 postcondition 7)
`Config.Validate()` rejects `drain_timeout` <= 0 with an error matching E-CFG-006 format: `"config error: drain_timeout: must be > 0; got '<value>'. Fix: set to a positive duration, e.g. '10s'"`.
- **Test:** `TestConfigValidate_RejectsNonPositiveDrainTimeout`

### AC-008 (traces to BC-2.09.003 postcondition 8)
`Config.Validate()` rejects `keepalive_interval` <= 0 with an error matching E-CFG-007 format: `"config error: keepalive_interval: must be > 0; got '<value>'. Fix: set to a positive duration, e.g. '1s'"`.
- **Test:** `TestConfigValidate_RejectsNonPositiveKeepaliveInterval`

### AC-009 (traces to BC-2.09.003 PC-9 / Inv-5)
When `--config` is supplied and validation passes, `cmd/switchboard` sources the half-channel tick interval from `cfg.TickInterval` (passed to `halfchannel.New`) instead of the hardcoded `10ms` default; the validated `cfg` is threaded to the daemon runtime and not discarded. The daemon MUST NOT silently fall back to the hardcoded `10*time.Millisecond` constant when a config file is supplied (Inv-5, applicable fields only).
- **Test:** `TestConfigTickIntervalApplied` — construct/run the daemon access path with a config whose `tick_interval` differs from the 10ms default (e.g. 20ms) and assert `halfchannel.New` receives that value.

> **Scope note (Deferred Application — BC-2.09.003 v1.3 DEFERRED-APPLICATION):** AC-009 covers ONLY `tick_interval`. The following fields are VALIDATED by AC-005–AC-008 (validation errors still cause exit 1), but their APPLICATION to subsystems is explicitly deferred:
> - `listen_addr` binding (no TCP listener exists on develop) — owned by **S-BL.NI** (network-ingress listener story, pending authorship; flagged in STORY-INDEX)
> - `drain_timeout`, `upstream_routers`, `keepalive_interval` application — owned by **S-7.04** (Wave 7, PE graduation and graceful drain)
>
> These are not gaps — they are tracked forward dependencies with named owning stories.
