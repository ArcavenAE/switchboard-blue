# S-6.04: SIGHUP Config Reload with Fail-Closed Safety

> **STATUS: DRAFT.** This story is a placeholder for future scheduling — not yet assigned
> to a wave. Acceptance criteria are intentionally lightweight (sketch-level). When
> promoted to a wave, story-writer will flesh out full ACs, tasks, file structure, and
> architecture mapping.
>
> Anchored to: BC-2.09.003 Inv-3 and EC-004.

## Narrative

- **As a** router operator
- **I want to** reload the daemon configuration by sending SIGHUP, without restarting the daemon
- **So that** I can update settings at runtime while ensuring a bad config file never disrupts an in-flight session

## Behavioral Contract Anchors

This story implements the SIGHUP/reload branch of BC-2.09.003:

| BC Reference | Requirement |
|-------------|-------------|
| BC-2.09.003 Inv-3 | Config reload (SIGHUP) is subject to the same validation contract as startup: a bad reload config leaves the daemon running on the previous (valid) config. |
| BC-2.09.003 EC-004 | When SIGHUP triggers a config reload and the new config fails validation, the daemon logs `"config reload failed: <error>; continuing with previous config"` and the previous config remains active. No sessions are disrupted. |

## Acceptance Criteria (sketch — expand when scheduled)

### AC-001 (traces to BC-2.09.003 EC-004)
When the daemon receives SIGHUP, it re-reads and re-validates the config file at the path originally supplied via `--config`. If the new config fails `Config.Validate()`, the daemon logs the error to stderr (format: `"config reload failed: <error>; continuing with previous config"`) and continues running with the previously-valid config. No sessions are disrupted.

### AC-002 (traces to BC-2.09.003 Inv-3)
When the daemon receives SIGHUP and the new config passes validation, the daemon applies the new config to applicable subsystems. (Exact subsystem list TBD by architect when scheduled — at minimum: `tick_interval` per PC-9/AC-009 precedent from S-6.01.)

### AC-003 (traces to BC-2.09.003 Inv-1 / fail-closed)
The daemon MUST NOT enter a partially-updated state during reload. Either the full new config is applied atomically (all applicable fields), or the reload is aborted and the previous config remains fully in effect.

## Architecture Notes (to be confirmed when scheduled)

- Signal handling belongs in `cmd/switchboard/main.go` (or a dedicated signal handler goroutine).
- `internal/config` reload logic (re-read file + re-validate) is pure-core; the signal plumbing is effectful.
- Atomic config swap: the daemon must hold the current config behind a `sync/atomic.Value` or protected by a mutex so in-flight requests see a consistent config during the swap.
- ARCH-06 §Config File Validation documents the binding sequence. Reload must follow the same `loadConfigFile → Validate` chain before swapping.
