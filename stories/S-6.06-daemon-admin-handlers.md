# S-6.06: Daemon-Side Admin RPC Handlers

> **Execute:** `/vsdd-factory:deliver-story S-6.06`

## Scope Note

S-6.02 shipped the `SVTNManager` methods (`RegisterKey`, `RevokeKey`, `ExpireKey`) and
`sbctl admin` CLI commands. However, the `cmd/switchboard` daemon never registered
`mgmt.Handler` entries for those RPCs ŌĆö every call site passed `startMgmtServer(..., nil)`.
`mgmt_wire.go:259` contains `TODO(CR-002)` documenting this gap.

Adversary Pass 1 on S-6.02 (finding F-001, CRITICAL) confirmed: admin RPC commands
invoked via `sbctl admin` against a live daemon return `E-RPC-010 "unknown command"`
because handlers are nil. CLI tests pass only because they use a fake/stub server.

This story wires the missing handler layer. It does not re-implement SVTNManager methods
(owned by S-6.02) nor the sbctl CLI dispatch (owned by S-6.02/S-6.03). It only adds the
`[]mgmt.Handler` builder in `cmd/switchboard` that bridges incoming RPC envelopes to the
already-existing SVTNManager methods.

Per the HUMAN DECISION (CR-W5-SCOPE-SPLIT), S-6.02 ships with an explicit deferral note
and S-W5.02 (e2e harness) depends on this story before it can exercise admin RPCs end-to-end.

## Acceptance Criteria

### AC-001 (traces to BC-2.05.004 postconditions 1ŌĆō3)
`cmd/switchboard/admin_handlers.go` exports a function with signature
`BuildAdminHandlers(m *svtnmgmt.SVTNManager, ops *mgmt.OperatorKeySet) []mgmt.Handler` that returns handlers for
`admin.key.register`, `admin.key.revoke`, `admin.key.expire`, and `admin.key.list-keys`.
The `ops *mgmt.OperatorKeySet` parameter is required for EC-006 bootstrap-grant authority: an operator-set member may call `admin.key.register` for a SVTN with no active control key; passing `nil` is equivalent to an empty OperatorKeySet.
Each handler:
- Parses the JSON args field from the incoming `mgmt.Request`
- Calls the appropriate `SVTNManager` method
- Maps sentinel errors to wire-level error codes (see Error Code Map section)
- Returns a `mgmt.Response` with `ok: true` and serialized result on success

Tests (unit, stdlib `testing`, table-driven, no testify):
- `TestBuildAdminHandlers_KeyRegister_HappyPath` ŌĆö response payload contains `key_fingerprint` and `timestamp` fields (BC-2.05.004 PC-4)
- `TestBuildAdminHandlers_KeyRevoke_HappyPath` ŌĆö response payload contains `key_fingerprint` and `timestamp` fields (BC-2.05.004 PC-4)
- `TestBuildAdminHandlers_KeyExpire_HappyPath` ŌĆö response payload contains `key_fingerprint` and `timestamp` fields (BC-2.05.004 PC-4)
- `TestBuildAdminHandlers_ListKeys_HappyPath`
- `TestBuildAdminHandlers_KeyRegister_ErrorMapping` (ErrSVTNNotFound ŌåÆ E-SVTN-003)
- `TestBuildAdminHandlers_KeyRevoke_ErrorMapping` (ErrKeyNotRegistered ŌåÆ E-ADM-013; ErrRoleMismatch ŌåÆ E-ADM-019)

### AC-002 (traces to BC-2.05.004 PC-2; HOLD-001 hybrid pattern; ADR-004)
The `admin.key.revoke` handler:
1. Unmarshals `role` from the wire JSON args field (canonical field name; CLI sends `role` per interface-definitions.md v1.1 F-002 ruling)
2. Converts it to `admission.KeyRole`
3. Passes it as `currentRole` to `SVTNManager.RevokeKey`

End-to-end tests using a real `mgmt.Server` + real Unix socket (NOT a fake server):
- `TestE2E_AdminRevoke_RoleMismatch`: caller claims `console` role, key is `control` ŌåÆ
  response contains `E-ADM-019`
- `TestE2E_AdminRevoke_ControlWithoutConfirm`: control-to-control revocation without
  `"confirm":true` in args ŌåÆ `E-ADM-018`
- `TestE2E_AdminRevoke_ControlWithConfirm`: control-to-control with `"confirm":true` ŌåÆ
  success; key no longer present in `AdmittedKeySet`

### AC-003 (traces to BC-2.05.004 PC-1, PC-3)
`admin.key.register`, `admin.key.expire`, and `admin.key.list-keys` handlers are each
registered and exercised in end-to-end tests against a real `mgmt.Server` + real
Unix socket.

Tests:
- `TestE2E_AdminRegister_HappyPath`: registers key; verifies it appears in list-keys response
- `TestE2E_AdminExpire_HappyPath`: sets TTL; verifies expiry timestamp set on key
- `TestE2E_AdminListKeys_HappyPath`: registers two keys; list-keys returns both

### AC-004 (traces to BC-2.05.004 PC-1; ADR-004 role-exclusion)
Handler registration is gated by daemon mode per ADR-004 role-exclusion:
- Control-mode daemon: all four admin handlers registered via `BuildAdminHandlers`
- Access, console, and router-mode daemons: `BuildAdminHandlers` is NOT called; those
  daemon-mode entry points pass a nil or empty handler slice for admin commands

Tests assert per-daemon-mode behavior:
- `TestControlMode_AdminHandlersRegistered`: sends `admin.key.register` to control
  daemon socket; gets `ok: true` (or a domain error, not `E-RPC-010`)
- `TestAccessMode_AdminHandlersNotRegistered`: sends `admin.key.register` to
  access daemon socket; gets `E-RPC-010 "unknown command"`
- Same pattern for console and router modes

### AC-005 (traces to BC-2.05.004 PC-3; defense-in-depth; DI-003)
Server-side duration validation in the `admin.key.expire` handler:
- Handler re-parses the `after` duration field from the raw JSON args (even though
  sbctl CLI also validates it ŌĆö defense-in-depth)
- Rejects `ttl <= 0` with `E-CFG-001`
- Rejects `ttl > 100*365*24*time.Hour` (100 years) with `E-CFG-001`

Test bypasses sbctl CLI (posts JSON directly to Unix socket):
- `TestE2E_AdminExpire_ServerRejectsTTLNegative`: POST `{"after":"-1h"}` directly to
  socket ŌåÆ response contains `E-CFG-001`
- `TestE2E_AdminExpire_ServerRejectsTTLZero`: POST `{"after":"0s"}` ŌåÆ `E-CFG-001`
- `TestE2E_AdminExpire_ServerRejectsTTLTooLong`: POST `{"after":"876001h"}` (>100 years)
  ŌåÆ `E-CFG-001`

### AC-006 (traces to BC-2.05.004 Precondition 1 / DI-001 ŌĆö admission-control authority per ADR-004)
Caller-key-role enforcement at the handler layer: `admin.key.register`, `admin.key.revoke`, and `admin.key.expire` handlers verify that the requesting key has management authority (control-role) before dispatching to SVTNManager. Caller identity is resolved server-side from the authenticated handshake pubkey via mgmt context ŌĆö never from request payload. A non-control-role key (e.g., console or access role) that invokes any of these mutating handlers receives `E-ADM-009 "insufficient authority for operation <cmd>: key <fingerprint> has role <role>"`.

**Scope exclusion (F-L2-003):** `admin.key.list-keys` is a read-only operation open to any admitted role and is NOT gated by this authority check. A console-role or access-role key calling `admin.key.list-keys` MUST receive `ok: true`, NOT E-ADM-009.

**Handler gate vs. SVTNManager gate precedence (F-L2-002):** The handler-layer authority gate fires BEFORE dispatch to SVTNManager. A console-role caller attempting `admin.key.revoke` receives E-ADM-009 from the handler; `SVTNManager.RevokeKey` is never invoked. E-ADM-011 (hierarchy violation within SVTNManager) is not reachable via the mgmt RPC path when this gate is in place.

Tests (e2e, real mgmt.Server + real Unix socket, `//go:build integration` tag):
- `TestE2E_AdminKeyRegister_RoleInsufficient`: console-role caller invokes `admin.key.register` ŌåÆ response contains `E-ADM-009`
- `TestE2E_AdminKeyRevoke_RoleInsufficient`: access-role caller invokes `admin.key.revoke` ŌåÆ response contains `E-ADM-009`
- `TestE2E_AdminListKeys_AnyRole`: console-role caller invokes `admin.key.list-keys` ŌåÆ response contains `ok: true` (not E-ADM-009)

> **Error code note (F-009):** E-ADM-009 is used per BC-2.05.004 Canonical Test Vectors ("Key operation by node without management authority ŌåÆ E-ADM-009"). Verify E-ADM-009 exists in error-taxonomy v3.0; it is listed there as the insufficient-authority sentinel. Do not invent a new code.
