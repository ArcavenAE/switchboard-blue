# S-4.04: Split-Horizon Loop Prevention in internal/routing

> **Execute:** `/vsdd-factory:deliver-story S-4.04`

## Narrative

- **As a** router
- **I want to** prevent frames from being forwarded back toward the interface they arrived on
- **So that** routing loops do not amplify traffic or cause infinite frame forwarding

## Acceptance Criteria

### AC-001 (traces to BC-2.02.008 postcondition 1)
`SplitHorizon.Forward(frame, arrival_interface_id, interface_set)` does not forward the frame on the `arrival_interface_id` interface.
- **Test:** `TestSplitHorizon_NoForwardTowardArrivalInterface`

### AC-002 (traces to BC-2.02.008 postcondition 2)
`SplitHorizon.Forward` forwards the frame on all other interfaces in the interface set.
- **Test:** `TestSplitHorizon_ForwardOnAllOtherInterfaces`

### AC-003 (traces to BC-2.02.008 invariant 1)
The router code never parses channel header payload; the fuzz test confirms that injecting arbitrary bytes into the channel header section does not affect routing decisions.
- **Test:** `FuzzSplitHorizon_ChannelHeaderOpaque`

### AC-004 (traces to BC-2.02.009 postcondition 1)
The router `OnFrameArrival` path consults the compound-key `(checksum, arrival_interface_id)` `DropCache` (from `internal/multipath`) before forwarding. When the key is already present, the frame is silently discarded (loop duplicate suppression). When the key is absent, it is added and the frame is forwarded.
- **Test:** `TestBC_2_02_009_Router_DropCacheWiring`

### AC-005 (traces to BC-2.02.009 postcondition 2 and EC-002)
EC-005 collision-event logging: the router's `OnFrameArrival` path injects a logger so that drop-cache hits on the same compound key are logged as potential collision events for investigation. Logger injection follows the `internal/routing.WithLogger` pattern established in S-2.02.

Logging MUST be rate-limited or sampled so that log volume is BOUNDED under a sustained stream of hits on the same compound key (routing loop, replayed-frame flood — BC-2.02.009 EC-002). The first occurrence of a collision on a given compound key MUST always be logged so operators are alerted. Subsequent rapid identical-key hits MUST produce far fewer than N log lines for N hits. Acceptable implementation strategies (any one is sufficient; do not mandate a specific one):
- Token-bucket per compound key (e.g. refill rate 1/min, burst 1)
- Log-first-then-sample: log the first hit, then 1-in-N thereafter (e.g. N=100 or configurable)
- Periodic coalesced summary: log the first hit immediately; suppress subsequent hits for a window (e.g. 30 s), then emit a single summary line with the suppressed count

The rate-limiting or sampling state is per compound key `(checksum, arrival_interface_id)`, scoped to the lifetime of the `OnFrameArrival` handler.

Justification: BC-2.02.009 EC-002 frames the loop/flood defense as bounded cache + hit-rate metric. Logging every hit under a flood violates CWE-779 (log injection / unbounded log output). Rate-limited logging is consistent with EC-002's bounded-defense model and is a story-level refinement of HOW EC-005 logging behaves; no BC change is required.

- **Test:** `TestBC_2_02_009_Router_CollisionLogRateLimited`
  - Assert: first drop-cache hit on a given compound key produces at least one log line (first hit logged).
  - Assert: N rapid subsequent identical-key hits (e.g. N=1000) produce fewer than N log lines (bounded — log count must be ≪ N, e.g. ≤ max(2, N/100)).

### AC-006 (traces to BC-2.02.009 postcondition 1 and BC-2.02.008 postcondition 2)
`OnFrameArrival` composes DropCache suppression (AC-004) and split-horizon forwarding (AC-001/AC-002) into a single end-to-end frame-arrival handler, exactly as ARCH-03 §Duplicate-and-Race describes:
- (a) **DropCache MISS:** `OnFrameArrival` adds the compound key `(checksum, arrival_interface_id)` to the DropCache, then calls `SplitHorizon.Forward(frame, arrival_interface_id, interface_set)`, and the frame is forwarded on all interfaces in the set except `arrival_interface_id`.
- (b) **DropCache HIT:** `OnFrameArrival` silently discards the frame and does NOT invoke `SplitHorizon.Forward`; no forwarding occurs (existing AC-004 behavior preserved).

`SplitHorizon.Forward` MUST have at least one non-test caller: the `OnFrameArrival` handler.
- **Test:** `TestOnFrameArrival_ForwardsViaSplitHorizon_AfterDropCacheMiss`

## Scope Transfer Note

BC-2.02.009 router-side wiring was transferred to this story per the product-owner's
Pass-2 scope ruling (`.factory/cycles/cycle-1/S-4.01/adjudications/pass2-bc009-scope.md`,
Ruling 1 — F-H1). S-4.01 delivers the `DropCache` primitive (AC-006, AC-007 hit counter).
S-4.04 owns wiring `DropCache` into the router `OnFrameArrival` forwarding path (AC-004)
and injecting a logger for EC-005 collision-event logging (AC-005), because both
BC-2.02.008 (split-horizon) and BC-2.02.009 (drop-cache loop suppression) describe
router-level frame-arrival handling and the ARCH-03 §Duplicate-and-Race `OnFrameArrival`
pseudocode treats them as a single frame-arrival handler.

**End-to-end composition is IN SCOPE for S-4.04 (not deferred).** Per Wave-4 Phase-3
adversarial reconverge ruling, `OnFrameArrival` must compose `SplitHorizon.Forward`
directly: after a DropCache miss the handler calls `SplitHorizon.Forward`; on a hit
it discards without forwarding. This is required by AC-006 and ensures `SplitHorizon.Forward`
has a non-test caller in production code.

**VP traceability:** VP-025 (DropCache capacity/suppression proptest) is canonically
owned by S-4.01 and anchored to `internal/multipath`; it is NOT claimed by this story.
S-4.04 delivers the router-side WIRING of BC-2.02.009 (AC-004/AC-005/AC-006), whose
behavioral coverage lives in the `TestBC_2_02_009_Router_*` tests in
`internal/routing/on_frame_arrival_test.go`. The story's `vp_traces` (VP-011, VP-015)
correctly reflect only the split-horizon VPs (BC-2.02.008) that S-4.04 owns.

## Edge Cases

| ID | Scenario | Expected Behavior |
|----|----------|-------------------|
| EC-001 | Only one interface in set (arrival interface) | Frame is dropped entirely (no forward possible) |
| EC-002 | Frame arrives with unknown interface ID | Treated as a new interface; forwarded on all known interfaces |
| EC-003 | Same frame (same checksum + arrival_interface_id) arrives twice | Second arrival silently discarded; DropCache hit counter incremented (BC-2.02.009) |
| EC-004 | Two different frames share a checksum on the same interface (hash collision) | Legitimate frame incorrectly suppressed; event logged via injected logger as potential collision (BC-2.02.009 EC-005) |
| EC-005 | Sustained flood of same-key drop-cache hits (routing loop or replayed-frame attack, BC-2.02.009 EC-002) | Log output MUST be bounded: first hit logged immediately; subsequent rapid hits suppressed or sampled (rate-limited per AC-005); log lines produced ≪ number of hits (CWE-779 mitigation) |
