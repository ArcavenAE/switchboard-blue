# S-4.04: Split-Horizon Loop Prevention in internal/routing

> **Execute:** `/vsdd-factory:deliver-story S-4.04`

## Narrative

- **As a** router
- **I want to** prevent frames from being forwarded back toward the interface they arrived on
- **So that** routing loops do not amplify traffic or cause infinite frame forwarding

## Acceptance Criteria

### AC-001 (traces to BC-2.02.008 postcondition 1)
`SplitHorizon.Forward(frame, arrival_interface_id, interface_set)` does not forward the frame on the `arrival_interface_id` interface.
- **Test:** `TestSplitHorizon_NoForwardTowardArrivalInterface`

### AC-002 (traces to BC-2.02.008 postcondition 2)
`SplitHorizon.Forward` forwards the frame on all other interfaces in the interface set.
- **Test:** `TestSplitHorizon_ForwardOnAllOtherInterfaces`

### AC-003 (traces to BC-2.02.008 invariant 1)
The router code never parses channel header payload; the fuzz test confirms that injecting arbitrary bytes into the channel header section does not affect routing decisions.
- **Test:** `FuzzSplitHorizon_ChannelHeaderOpaque`

### AC-004 (traces to BC-2.02.009 postcondition 1)
The router `OnFrameArrival` path consults the compound-key `(checksum, arrival_interface_id)` `DropCache` (from `internal/multipath`) before forwarding. When the key is already present, the frame is silently discarded (loop duplicate suppression). When the key is absent, it is added and the frame is forwarded.
- **Test:** `TestBC_2_02_009_Router_DropCacheWiring`

### AC-005 (traces to BC-2.02.009 postcondition 2 and EC-002)
EC-005 collision-event logging: the router's `OnFrameArrival` path injects a logger so that drop-cache hits are logged as potential collision events for investigation. Logger injection follows the `internal/routing.WithLogger` pattern established in S-2.02.

**(a) BOUNDED MEMORY for collision-tracking state.** Any per-key rate-limit or collision-tracking structure used to implement log sampling MUST be bounded. The structure MUST be capped to at most the DropCache capacity (default 10,000 entries, BC-2.02.009 precondition 1) or a fixed implementation-chosen maximum. A distinct attacker-controlled stream of `(checksum, arrival_interface_id)` keys MUST NOT cause the tracking structure to grow without bound. No monotonically-growing map (CWE-401/400). The tracking structure lives in `internal/routing` — do NOT modify `internal/multipath.DropCache` to accommodate it; if the implementer judges folding the counter into DropCache is cleaner, that is a separate cross-story change.

**(b) BOUNDED AGGREGATE log volume across ALL keys.** Under a sustained flood of DISTINCT compound keys, aggregate collision-log output across ALL keys MUST also be bounded. N distinct-key collisions MUST produce far fewer than N log lines in aggregate. Acceptable strategies for the aggregate bound (any one is sufficient; do not mandate a specific one):
- A global token bucket or leaky bucket on collision-log emission (e.g. burst 10, refill rate 1/s)
- A global sampling budget (e.g. emit at most 1-in-100 log lines globally when aggregate rate exceeds a threshold)
- A global periodic coalesced summary (e.g. suppress excess lines for 30 s, then emit one summary with the suppressed count)

**(c) BEST-EFFORT first-occurrence observability.** The first occurrence of a collision on a given compound key SHOULD be logged when capacity allows — that is, when the bounded tracking window has not yet evicted that key and the aggregate log budget permits. This guarantee is explicitly BEST-EFFORT because bounded memory precludes tracking an unbounded distinct key space. Operators MUST NOT rely on every first occurrence being logged under adversarial distinct-key floods.

**(d) Per-key flood (same key repeated) MUST still be bounded.** N rapid identical-key hits (e.g. N=1000) MUST produce far fewer than N log lines for that key (preserves v1.3 same-key behavior).

Justification: BC-2.02.009 EC-002 frames the loop/flood defense as bounded cache + hit-rate metric. Rate-limit tracking keyed on an attacker-controlled, unbounded key space violates CWE-401/400 (unbounded memory) and CWE-779 (aggregate log-spam DoS). A bounded tracking structure + aggregate rate limit is consistent with, and strengthens, EC-002's bounded-defense model. This is a story-level refinement of HOW EC-005 logging behaves; no BC change is required. EC-002 already frames the defense as bounded.

- **Test:** `TestBC_2_02_009_Router_CollisionLogRateLimited`
  - Assert: first drop-cache hit on a given compound key produces at least one log line (first hit logged — best-effort, same key repeated).
  - Assert: N rapid subsequent identical-key hits (e.g. N=1000) produce fewer than N log lines for that key (bounded per-key — log count must be ≪ N, e.g. ≤ max(2, N/100)).

- **Test:** `TestBC_2_02_009_Router_CollisionLog_DistinctKeyFlood_Bounded`
  - Setup: generate K distinct compound keys (e.g. K=500) and send one drop-cache hit per key in rapid succession.
  - Assert (i): total aggregate log lines emitted is far less than K (bounded aggregate — e.g. ≤ max(10, K/50)).
  - Assert (ii): the tracking structure size (number of tracked keys) does not exceed its declared cap (e.g. DropCache capacity or the implementation's fixed maximum) — memory does not grow without limit regardless of K.

### AC-006 (traces to BC-2.02.009 postcondition 1 and BC-2.02.008 postcondition 2)
`OnFrameArrival` composes DropCache suppression (AC-004) and split-horizon forwarding (AC-001/AC-002) into a single end-to-end frame-arrival handler, exactly as ARCH-03 §Duplicate-and-Race describes:
- (a) **DropCache MISS:** `OnFrameArrival` adds the compound key `(checksum, arrival_interface_id)` to the DropCache, then calls `SplitHorizon.Forward(frame, arrival_interface_id, interface_set)`, and the frame is forwarded on all interfaces in the set except `arrival_interface_id`.
- (b) **DropCache HIT:** `OnFrameArrival` silently discards the frame and does NOT invoke `SplitHorizon.Forward`; no forwarding occurs (existing AC-004 behavior preserved).

`SplitHorizon.Forward` MUST have at least one non-test caller: the `OnFrameArrival` handler.
- **Test:** `TestOnFrameArrival_ForwardsViaSplitHorizon_AfterDropCacheMiss`

### AC-007 (traces to BC-2.02.008 postcondition 3)
When `SplitHorizon.Forward` determines that the only interface in the set is the arrival interface (all eligible paths are split-horizon-blocked), `OnFrameArrival` MUST drop the frame AND emit exactly one E-FWD-001 log event via its injected logger. The log line MUST include the arrival interface ID and the checksum of the dropped frame, consistent with the field set used by the EC-005 drop-cache-hit log line. No forward call occurs.

- **Test:** `TestOnFrameArrival_AllPathsSplitHorizon_LogsEFWD001`
  - Setup: construct a `FrameArrivalHandler` with a capturing logger and a fresh `DropCache`; call `OnFrameArrival` with an `interfaceSet` containing only the `arrivalIface`.
  - Assert (i): the return value is `ErrAllPathsSplitHorizon` (frame dropped).
  - Assert (ii): the capturing logger received exactly one log line containing `"E-FWD-001"`, the arrival interface ID, and the frame checksum.
  - Assert (iii): the forward function `fn` was never called (no forward occurred).

pseudocode treats them as a single frame-arrival handler.

**End-to-end composition is IN SCOPE for S-4.04 (not deferred).** Per Wave-4 Phase-3
adversarial reconverge ruling, `OnFrameArrival` must compose `SplitHorizon.Forward`
directly: after a DropCache miss the handler calls `SplitHorizon.Forward`; on a hit
it discards without forwarding. This is required by AC-006 and ensures `SplitHorizon.Forward`
has a non-test caller in production code.

**VP traceability:** VP-025 (DropCache capacity/suppression proptest) is canonically
owned by S-4.01 and anchored to `internal/multipath`; it is NOT claimed by this story.
S-4.04 delivers the router-side WIRING of BC-2.02.009 (AC-004/AC-005/AC-006), whose
behavioral coverage lives in the `TestBC_2_02_009_Router_*` tests in

correctly reflect only the split-horizon VPs (BC-2.02.008) that S-4.04 owns.

## Edge Cases

| ID | Scenario | Expected Behavior |
|----|----------|-------------------|
| EC-001 | Only one interface in set (arrival interface) | Frame is dropped entirely (no forward possible) AND an E-FWD-001 event is logged via the injected logger with arrival interface ID and frame checksum (BC-2.02.008 PC-3; AC-007) |
| EC-002 | Frame arrives with unknown interface ID | Treated as a new interface; forwarded on all known interfaces |
| EC-003 | Same frame (same checksum + arrival_interface_id) arrives twice | Second arrival silently discarded; DropCache hit counter incremented (BC-2.02.009) |
| EC-004 | Two different frames share a checksum on the same interface (hash collision) | Legitimate frame incorrectly suppressed; event logged via injected logger as potential collision (BC-2.02.009 EC-005) |
| EC-005 | Sustained flood of same-key drop-cache hits (routing loop or replayed-frame attack, BC-2.02.009 EC-002) | Per-key log output MUST be bounded: first hit on a given key logged immediately (best-effort, within aggregate budget); subsequent rapid identical-key hits suppressed or sampled (rate-limited per AC-005-d); log lines produced ≪ number of hits (CWE-779 mitigation) |
| EC-006 | Sustained flood of DISTINCT compound keys (attacker-varied checksums/interface IDs — CWE-401/400 + CWE-779) | Tracking structure MUST NOT grow without bound: capped to DropCache capacity or fixed maximum; LRU or equivalent eviction removes oldest entries. Aggregate log output MUST be bounded: N distinct-key collisions produce far fewer than N log lines via global rate-limit/token-bucket/sampling (AC-005-b). First-occurrence logging is best-effort: once tracking capacity is full, first-occurrence guarantee is relaxed — operators must not rely on it under adversarial distinct-key floods (AC-005-c). |
