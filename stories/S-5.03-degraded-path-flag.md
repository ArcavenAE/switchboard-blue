# S-5.03: Flag Paths as Degraded When EWMA RTT Exceeds 200ms Threshold

> **Execute:** `/vsdd-factory:deliver-story S-5.03`

## Narrative

- **As a** quality-observability consumer (internal/metrics)
- **I want to** read a pre-computed `degraded` boolean from `PathSnapshot`
- **So that** the sbctl output `status` field can be set to `degraded` when a path's EWMA RTT has crossed 200ms, without duplicating threshold logic outside `internal/paths`

## Acceptance Criteria

### AC-001 (traces to BC-2.02.003 postcondition 5)
`PathTracker.IsDegraded()` returns `true` after `OnProbe` is called with an EWMA-smoothed RTT that has converged above `DegradedRTTThresholdMS` (200.0ms). `Snapshot().Degraded` reflects the same boolean.
- **Test:** `TestBC_2_02_003_PathTracker_IsDegraded` — table-driven; include at least: (a) RTT always below threshold → IsDegraded=false, (b) RTT always above threshold → IsDegraded=true, (c) RTT transitions from above to below → IsDegraded follows EWMA convergence.

### AC-002 (traces to BC-2.02.003 postcondition 5 — recovery branch)
`PathTracker.IsDegraded()` returns `false` after subsequent `OnProbe` calls drive the EWMA RTT back below `DegradedRTTThresholdMS`. Recovery is symmetric — the flag clears on the same tick where the EWMA drops below threshold.
- **Test:** included as case (c) in `TestBC_2_02_003_PathTracker_IsDegraded`; also cover recovery after sustained degradation (≥5 above-threshold probes followed by ≥5 below-threshold probes).

### AC-003 (traces to BC-2.02.003 postcondition 6 — reactivation branch)
On path reactivation (`resetRTT` called with the arrival RTT of the first successful probe), the `degraded` flag is set from the reactivating probe's RTT: `degraded = arrivalRTTMS > DegradedRTTThresholdMS`. A reactivating probe at 250ms → degraded=true; at 150ms → degraded=false.
- **Test:** `TestBC_2_02_003_PathTracker_IsDegraded_OnReactivation`

### AC-004 (traces to BC-2.02.003 invariant 2 — concurrency safety)
Concurrent calls to `IsDegraded()` and `OnProbe()` under the Go race detector (`-race`) produce no data races. `Snapshot()` under concurrent `OnProbe()` also produces no data races.
- **Test:** `TestBC_2_02_003_PathTracker_IsDegraded_Race` — launch N goroutines calling `OnProbe` and M goroutines calling `IsDegraded`/`Snapshot()` concurrently; run with `go test -race`.

### AC-005 (traces to BC-2.02.003 postcondition 5 — property test; anchors VP-063)
For any RTT sample sequence, `IsDegraded()` is true if and only if the EWMA-smoothed RTT exceeds `DegradedRTTThresholdMS` (200.0ms). Recovery below the threshold clears the flag with no hysteresis or additional delay. This property is verified via gopter property tests.
- **Test:** `TestProp_IsDegraded_TracksEWMAThreshold` and `TestProp_IsDegraded_RecoveryClears` (proptest harness per VP-063 proof harness skeleton). VP-063 is a Wave-5 functional property (exercisable during S-5.03 TDD, not Phase-6 deferred).

## Scope Note

This story is confined to `internal/paths`. The changes are:

1. Add constant `DegradedRTTThresholdMS = 200.0` (float64) to `internal/paths`.
2. Add `degraded bool` field to `PathTracker` struct (shares existing `mu sync.Mutex`).
3. Extend `OnProbe` success branch: after updating `ewmaRTTMS`, set `t.degraded = t.ewmaRTTMS > DegradedRTTThresholdMS`. Loss events do not touch `degraded` (no RTT measured on miss — consistent with existing semantics).
4. Extend `resetRTT` (called on reactivation and first probe): set `t.degraded = arrivalRTTMS > DegradedRTTThresholdMS`.
5. Add `IsDegraded() bool` accessor (takes `t.mu`; returns `t.degraded`).
6. Introduce `PathSnapshot` value type with fields: `EWMARTTMs float64`, `LossPct float64`, `Active bool`, `Degraded bool`, `P99RTTMs float64`. Add `Snapshot() PathSnapshot` method that takes `t.mu` once and returns a consistent copy. Note: `P99RTTMs` is wired in S-5.02 (the histogram field is added there); for S-5.03, `Snapshot()` may set `P99RTTMs: 0` as a placeholder if the histogram is not yet present, or both stories may be sequenced so S-5.03 runs first and S-5.02 extends the struct.

No changes to `internal/metrics`, `cmd/sbctl`, or any other package. The composition contract with S-5.01 is: once S-5.03 merges, S-5.01 switches `internal/metrics` from individual accessor calls to `Snapshot()` to read `Snapshot().Degraded` when populating the per-path `status` field.

## Composition Note

S-5.01 (`internal/metrics`) reads `Snapshot().Degraded` to set the per-path `status` field in the quality-indicator metrics snapshot exposed to sbctl. S-5.01 does not evaluate `DegradedRTTThresholdMS` directly — it reads the pre-computed boolean. The threshold evaluation lives entirely in `internal/paths`. This is the composition contract described in ARCH-03 v1.5 §Degraded-Path Flag Design.

## Edge Cases

| ID | Scenario | Expected Behavior |
|----|----------|-------------------|
| EC-001 | RTT exactly at threshold (200.0ms) | 200.0 > 200.0 is false → degraded=false. Boundary is exclusive: `> DegradedRTTThresholdMS`. |
| EC-002 | EWMA RTT oscillates near threshold | Flag follows EWMA value on each successful probe; no hysteresis on the degraded flag itself (hysteresis is a quality-indicator concern in internal/metrics, not a PathTracker concern). |
| EC-003 | Path deactivated (missed keep-alives) then reactivated | `resetRTT` sets `degraded` from the reactivating RTT. No stale flag from before deactivation. |
| EC-004 | Loss event (no RTT measured) | `degraded` is not updated on loss event — only on successful probe RTT measurement. |
