# S-6.05: SVTN Destroy Lifecycle: SVTNManager.Destroy + sbctl admin svtn destroy

> **Execute:** `/vsdd-factory:deliver-story S-6.05`

## Scope Note

This story owns **only** BC-2.07.001 postcondition 3 (Destroy). It was deferred
from S-6.02 per CR-009 product-owner ruling (2026-06-29): S-6.02's AC table never
claimed Destroy; adding it mid-delivery would expand an in-flight 8-point story
and stall the Wave-5 serialization chain. Destroy is not on any E-phase critical
path (no Wave-5 story depends on it), so Wave 6 is the correct placement.

**RULING-W6TB-A (2026-07-01):** `admin.svtn.destroy` uses the **general control-role
handler gate** (`resolveAndVerifyCallerRole`), NOT the bootstrap-only gate used by
`admin.svtn.create`. See `.factory/decisions/RULING-W6TB-A-svtn-destroy-authority.md`.

VP-048 properties 2 and 3:
- Property 2: "svtn destroy ‚Üí SVTN no longer appears in list ‚àß nodes cannot admit to the destroyed SVTN" ‚Äî verified by this story's integration test.
- Property 3: "non-control key cannot destroy SVTN" ‚Äî verified by AC-004 via handler gate.

## Narrative

- **As a** control node operator
- **I want to** destroy an SVTN via `sbctl admin svtn destroy`
- **So that** decommissioned networks are fully cleaned up: all admitted keys
  removed, all active sessions terminated, and the SVTN ID freed

## Acceptance Criteria

### AC-001 (traces to BC-2.07.001 postcondition 3)
`SVTNManager.Destroy(ctx, svtnName)` removes all admitted keys for the SVTN from
the router's key set and frees the SVTN ID. Returns `E-SVTN-003`
(`"SVTN not found: <name>"`) if the SVTN does not exist.
- **Test:** `TestSVTNManager_Destroy_RemovesAllKeys`

### AC-002 (traces to BC-2.07.001 postcondition 3 + EC-002)
`SVTNManager.Destroy(ctx, svtnName)` with active sessions terminates all sessions
with notification before destroying the SVTN. Admitted nodes receive
session-terminated signals.
- **Test:** `TestSVTNManager_Destroy_WithActiveSessions`

### AC-003 (traces to BC-2.07.001 postcondition 3)
`sbctl admin svtn destroy --name <svtn-name> --confirm <svtn-short-id>` invokes
`SVTNManager.Destroy` and prints confirmation to stdout on success. Exits with
non-zero on E-SVTN-003 (SVTN not found). The `--confirm=<svtn-short-id>` flag is
required per interface-definitions.md v1.1 ¬ß117 and ADR-004; when omitted the
command enters interactive mode prompting `Type SVTN-<short-id> to confirm:`.
- **Test:** `TestSbctlAdmin_SVTNDestroy_HappyPath`, `TestSbctlAdmin_SVTNDestroy_NotFound`,
  `TestSbctlAdmin_SVTNDestroy_ConfirmGate`

### AC-004 (traces to BC-2.07.001 invariant 3 ‚Äî handler gate; RULING-W6TB-A)
The `admin.svtn.destroy` handler calls `resolveAndVerifyCallerRole` before invoking
`SVTNManager.Destroy` (S-6.07 pattern). A non-control caller (console/readonly role)
attempting destroy via the mgmt RPC receives an error response with code `E-RPC-011`
and message containing `"E-ADM-009"` (the handler gate fires; the SVTN is not
destroyed). `ErrDestroyUnauthorized` / `E-ADM-011 Variant 2` is a Go-API-layer
defense-in-depth check at the `SVTNManager.Destroy` boundary ‚Äî it is only surfaced
in unit tests that call `SVTNManager.Destroy` directly without the handler layer.
- **Test (RPC path):** `TestSbctlAdmin_SVTNDestroy_RequiresControlRole` ‚Äî asserts
  E-RPC-011 wrapping E-ADM-009 in wire response; SVTN unchanged
- **Test (Go-API path):** `TestSVTNManager_Destroy_ErrDestroyUnauthorized` ‚Äî calls
  `SVTNManager.Destroy` directly with non-control caller; asserts
  `errors.Is(err, svtnmgmt.ErrDestroyUnauthorized)`

### AC-005 (traces to BC-2.07.001 invariant 3 + RULING-W6TB-A genesis re-open)
Destroying the last SVTN (such that `HasAnySVTN()` returns false) re-opens the
genesis carve-out for a subsequent `admin.svtn.create` with the bootstrap key.
This is permitted recovery semantics (not privilege escalation): a subsequent
`admin.svtn.create` call with the bootstrap key succeeds via the genesis carve-out
exactly as on first initialization.
- **Test:** `TestSVTNManager_Destroy_GenesisReopened` ‚Äî destroy last SVTN, then
  verify a subsequent `Create` with bootstrap key succeeds.

## Error Code Table (Implementer Reference)

Per RULING-W6TB-A:

| Caller path | Gate that fires | Observable error code |
|-------------|----------------|-----------------------|
| Non-control key via mgmt RPC | `resolveAndVerifyCallerRole` (handler layer) | E-RPC-011 wrapping E-ADM-009 |
| Non-control key via direct Go API | `SVTNManager.Destroy` defense-in-depth | `ErrDestroyUnauthorized` (E-ADM-011 Variant 2) |
| SVTN not found (any caller) | `SVTNManager.Destroy` | `ErrSVTNNotFound` (E-SVTN-003) |

Error string canonical forms (Ruling-11/12):
- `ErrSVTNNotFound`: `"SVTN not found: <name>"` (colon-space; `<name>` not `<id>`)
- `ErrDestroyUnauthorized`: `"destroy: caller is not a control-role key"`

## Edge Cases

| ID | Scenario | Expected Behavior |
|----|----------|-------------------|
| EC-001 | Destroy SVTN that does not exist | E-SVTN-003 `"SVTN not found: <name>"`; no action |
| EC-002 | Destroy SVTN with active sessions | Sessions terminated with notification; then SVTN destroyed (per BC-2.07.001 EC-002) |
| EC-003 | Destroy SVTN with multiple admitted nodes on multiple routers (PE topology) | Termination signals sent to all admitted nodes; propagation to other routers (per BC-2.07.001 EC-004) |
| EC-004 | Non-control key attempts destroy via mgmt RPC | E-RPC-011 wrapping E-ADM-009; SVTN unchanged |
| EC-005 | Non-control key invokes SVTNManager.Destroy directly (Go API) | `ErrDestroyUnauthorized` (E-ADM-011 Variant 2); SVTN unchanged |
| EC-006 | Destroy last SVTN ‚Üí subsequent admin.svtn.create with bootstrap key | Genesis carve-out re-opened; Create succeeds (recovery semantics, RULING-W6TB-A ¬ß4) |
