# S-6.05: SVTN Destroy Lifecycle: SVTNManager.Destroy + sbctl admin svtn destroy

> **Execute:** `/vsdd-factory:deliver-story S-6.05`

## Scope Note

This story owns **only** BC-2.07.001 postcondition 3 (Destroy). It was deferred
from S-6.02 per CR-009 product-owner ruling (2026-06-29): S-6.02's AC table never
claimed Destroy; adding it mid-delivery would expand an in-flight 8-point story
and stall the Wave-5 serialization chain. Destroy is not on any E-phase critical
path (no Wave-5 story depends on it), so Wave 6 is the correct placement.

**RULING-W6TB-A (2026-07-01):** `admin.svtn.destroy` uses the **general control-role
handler gate** (`resolveAndVerifyCallerRole`), NOT the bootstrap-only gate used by
`admin.svtn.create`. See `.factory/decisions/RULING-W6TB-A-svtn-destroy-authority.md`.

VP-048 properties 2 and 3:
- Property 2: "svtn destroy → SVTN no longer appears in list ∧ nodes cannot admit to the destroyed SVTN" — verified by this story's integration test.
- Property 3: "non-control key cannot destroy SVTN" — verified by AC-004 via handler gate.

## Narrative

- **As a** control node operator
- **I want to** destroy an SVTN via `sbctl admin svtn destroy`
- **So that** decommissioned networks are fully cleaned up: all admitted keys
  removed, all active sessions terminated, and the SVTN ID freed

## Acceptance Criteria

### AC-001 (traces to BC-2.07.001 postcondition 3)
`SVTNManager.Destroy(caller admission.AdmittedKey, svtnName string)` removes all
admitted keys for the SVTN from the router's key set and frees the SVTN ID. Returns
`E-SVTN-003` (`"SVTN not found: <name>"`) if the SVTN does not exist.
- **Test:** `TestSVTNManager_Destroy_RemovesAllKeys`

### AC-002 (traces to BC-2.07.001 PC-3, in-scope portion)
`SVTNManager.Destroy(caller admission.AdmittedKey, svtnName string)` removes the
SVTN entry and purges all admitted keys atomically under lock, rendering the SVTN's
session-set unrecoverable. The downstream session-terminated notification (BC-2.07.001
PC-3 "Nodes receive session-terminated signals") requires cross-subsystem plumbing
between svtnmgmt and `internal/session` which is architecturally forbidden inside
svtnmgmt (Forbidden Dependencies). This downstream notification is deferred to
follow-on story **S-BL.SESSION-DRAIN** (backlog).
- **Test (purge-under-quiescence oracle):** `TestSVTNManager_Destroy_KeyPurgePostcondition`
  — asserts that all admitted keys are absent from the key set after a successful
  `Destroy` call (the quiescent-state postcondition within the story's architectural
  boundary).
- **Test (concurrency-race oracle):** `TestSVTNManager_Destroy_ConcurrentAdmissionIsRaceFree`
  — asserts that concurrent admission attempts during and after `Destroy` do not produce
  data races or leave the key set in a torn state; validates atomic-under-lock
  postcondition (the session-layer-observable consequence for concurrent callers).

### AC-003 (traces to BC-2.07.001 postcondition 3)
`sbctl admin svtn destroy --name <svtn-name> --confirm <svtn-short-id>` invokes
`SVTNManager.Destroy` and prints confirmation to stdout on success. Exits with
non-zero on E-SVTN-003 (SVTN not found). The `--confirm=<svtn-short-id>` flag is
required per interface-definitions.md v1.1 §117 and §125 and ADR-004; when omitted
the command enters interactive mode prompting `Type SVTN-<short-id> to confirm:`
before proceeding. See Confirm-Gate Semantics below for the full flow
(interactive-mode path, `--yes` bypass, flag-supplied static-shape check).
- **Test:** `TestSbctlAdmin_SVTNDestroy_HappyPath`, `TestSbctlAdmin_SVTNDestroy_NotFound`,
  `TestSbctlAdmin_SVTNDestroy_ConfirmGate`

### AC-004 (traces to BC-2.07.001 invariant 3 — Go-API defense-in-depth + handler gate; RULING-W6TB-A §3)
The `admin.svtn.destroy` handler calls `resolveAndVerifyCallerRole` before invoking
`SVTNManager.Destroy` (S-6.07 pattern). A non-control caller (console/readonly role)
attempting destroy via the mgmt RPC receives an error response with code `E-RPC-011`
and message containing `"E-ADM-009"` (the handler gate fires; the SVTN is not
destroyed).

Additionally, `SVTNManager.Destroy(caller admission.AdmittedKey, svtnName string)`
applies `ErrDestroyUnauthorized` (E-ADM-011 Variant 2) as a **defense-in-depth Go-API
check**: when `caller.role != RoleControl`, `Destroy` returns `ErrDestroyUnauthorized`
before any SVTN state is consulted. This is the inner layer that guards against callers
that somehow bypass the handler gate (unit-test path, direct API invocation, or future
refactor that removes the handler gate inadvertently). Per RULING-W6TB-A §3 and
BC-2.07.001 Inv-3.

- **Test (RPC path — behavioral coverage, VP-048 P3):** `TestSbctlAdmin_SVTNDestroy_RequiresControlRole`
  — asserts E-RPC-011 wrapping E-ADM-009 in wire response; SVTN unchanged.
- **Test (Go-API defense-in-depth):** `TestSVTNManager_Destroy_ErrDestroyUnauthorized`
  — direct call `SVTNManager.Destroy(nonControlKey, "svtn-A")` returns
  `ErrDestroyUnauthorized` (E-ADM-011 Variant 2); direct call with control-role key
  succeeds. This is defense-in-depth at the Go API layer per RULING-W6TB-A §3 and
  BC-2.07.001 Inv-3.

### AC-005 (traces to BC-2.07.001 invariant 3 + RULING-W6TB-A genesis re-open)
Destroying the last SVTN (such that `HasAnySVTN()` returns false) re-opens the
genesis carve-out for a subsequent `admin.svtn.create` with the bootstrap key.
This is permitted recovery semantics (not privilege escalation): a subsequent
`admin.svtn.create` call with the bootstrap key succeeds via the genesis carve-out
exactly as on first initialization.
- **Test:** `TestSVTNManager_Destroy_GenesisReopened` — destroy last SVTN, then
  verify a subsequent `Create` with bootstrap key succeeds.

## Confirm-Gate Semantics

Per interface-definitions.md v1.1 §125 and §129 (canonical authority) and ADR-004:

The `--confirm` gate has **three paths**:

1. **Flag-supplied path:** `--confirm=SVTN-<8-hex>` satisfies the check
   non-interactively. The sbctl CLI validates that the supplied value matches the
   pattern `SVTN-[0-9a-f]{8}` before sending the destroy RPC; the operator must
   know and re-type the short-ID they saw at create-time (human-in-loop typing
   ceremony).

2. **Interactive-mode path (flag omitted):** When `--confirm` is not supplied on
   the command line, the command enters interactive mode and prompts
   `Type SVTN-<short-id> to confirm:` before proceeding. The operator types the
   SVTN short ID; the CLI validates the input and dispatches the destroy RPC on
   match. Per §125 and §129.

3. **`--yes` bypass:** `--yes` bypasses the `--confirm` interactive prompt for
   scripted use and emits a warning to stderr:
   `"WARNING: --yes bypasses confirmation; ensure correct --svtn target before scripting"`.
   Cannot be combined with `--confirm` (usage error, exit 2, E-CFG-006). Per §129.

Identity-match enforcement (verifying the confirm token names the same SVTN as
`--name`) is deferred to phase-5 hardening. See drift item
**DRIFT-S605-CONFIRM-IDENTITY** (backlog).

## Error Code Table (Implementer Reference)

Per RULING-W6TB-A:

| Caller path | Gate that fires | Observable error code |
|-------------|----------------|-----------------------|
| Non-control key via mgmt RPC | `resolveAndVerifyCallerRole` (handler layer) | E-RPC-011 wrapping E-ADM-009 |
| Non-control key via direct Go API | `SVTNManager.Destroy` defense-in-depth (`caller.role != RoleControl`) | `ErrDestroyUnauthorized` (E-ADM-011 Variant 2) |
| SVTN not found (any caller) | `SVTNManager.Destroy` | `ErrSVTNNotFound` (E-SVTN-003) |

Error string canonical forms (Ruling-11/12):
- `ErrSVTNNotFound`: `"SVTN not found: <name>"` (colon-space; `<name>` not `<id>`)
- `ErrDestroyUnauthorized`: `"destroy: caller is not a control-role key"`

## Edge Cases

| ID | Scenario | Expected Behavior |
|----|----------|-------------------|
| EC-001 | Destroy SVTN that does not exist | E-SVTN-003 `"SVTN not found: <name>"`; no action |
| EC-002 | Destroy SVTN with active sessions | Sessions terminated with notification; then SVTN destroyed (per BC-2.07.001 EC-002) |
| EC-003 | Destroy SVTN with multiple admitted nodes on multiple routers (PE topology) | Termination signals sent to all admitted nodes; propagation to other routers (per BC-2.07.001 EC-004) |
| EC-004 | Non-control key attempts destroy via mgmt RPC | E-RPC-011 wrapping E-ADM-009; SVTN unchanged |
| EC-005 | Non-control key invokes SVTNManager.Destroy directly (Go API) | `ErrDestroyUnauthorized` (E-ADM-011 Variant 2); SVTN unchanged |
| EC-006 | Destroy last SVTN → subsequent admin.svtn.create with bootstrap key | Genesis carve-out re-opened; Create succeeds (recovery semantics, RULING-W6TB-A §4) |
