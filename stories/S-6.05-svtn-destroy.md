## Narrative

- **As a** control node operator
- **I want to** destroy an SVTN via `sbctl admin svtn destroy`
- **So that** decommissioned networks are fully cleaned up: all admitted keys
  removed, all active sessions terminated, and the SVTN ID freed

## Acceptance Criteria

### AC-001 (traces to BC-2.07.001 postcondition 3)
`SVTNManager.Destroy(caller admission.AdmittedKey, svtnName string)` removes all
admitted keys for the SVTN from the router's key set and frees the SVTN ID. Returns
`E-SVTN-003` (`"SVTN not found: <name>"`) if the SVTN does not exist.
- **Test:** `TestSVTNManager_Destroy_RemovesAllKeys`

### AC-002 (traces to BC-2.07.001 PC-3, in-scope portion)
`SVTNManager.Destroy(caller admission.AdmittedKey, svtnName string)` removes the
SVTN entry and purges all admitted keys atomically under lock, rendering the SVTN's
session-set unrecoverable. The downstream session-terminated notification (BC-2.07.001
PC-3 "Nodes receive session-terminated signals") requires cross-subsystem plumbing
between svtnmgmt and `internal/session` which is architecturally forbidden inside
svtnmgmt (Forbidden Dependencies). This downstream notification is deferred to
follow-on story **S-BL.SESSION-DRAIN** (backlog).
- **Test:** `TestSVTNManager_Destroy_KeyPurgeUnderConcurrentAdmission` â€” asserts
  admission-blocking postcondition (the session-layer-observable consequence within
  the story's architectural boundary).

### AC-003 (traces to BC-2.07.001 postcondition 3)
`sbctl admin svtn destroy --name <svtn-name> --confirm <svtn-short-id>` invokes
`SVTNManager.Destroy` and prints confirmation to stdout on success. Exits with
non-zero on E-SVTN-003 (SVTN not found). The `--confirm=<svtn-short-id>` flag is
required per interface-definitions.md v1.1 Â§117 and ADR-004; when omitted the
command enters interactive mode prompting `Type SVTN-<short-id> to confirm:`.
- **Test:** `TestSbctlAdmin_SVTNDestroy_HappyPath`, `TestSbctlAdmin_SVTNDestroy_NotFound`,
  `TestSbctlAdmin_SVTNDestroy_ConfirmGate`

### AC-004 (traces to BC-2.07.001 invariant 3 â€” Go-API defense-in-depth + handler gate; RULING-W6TB-A Â§3)
The `admin.svtn.destroy` handler calls `resolveAndVerifyCallerRole` before invoking
`SVTNManager.Destroy` (S-6.07 pattern). A non-control caller (console/readonly role)
attempting destroy via the mgmt RPC receives an error response with code `E-RPC-011`
and message containing `"E-ADM-009"` (the handler gate fires; the SVTN is not
destroyed).

Additionally, `SVTNManager.Destroy(caller admission.AdmittedKey, svtnName string)`
applies `ErrDestroyUnauthorized` (E-ADM-011 Variant 2) as a **defense-in-depth Go-API
check**: when `caller.role != RoleControl`, `Destroy` returns `ErrDestroyUnauthorized`
before any SVTN state is consulted. This is the inner layer that guards against callers
that somehow bypass the handler gate (unit-test path, direct API invocation, or future
refactor that removes the handler gate inadvertently). Per RULING-W6TB-A Â§3 and
BC-2.07.001 Inv-3.

- **Test (RPC path â€” behavioral coverage, VP-048 P3):** `TestSbctlAdmin_SVTNDestroy_RequiresControlRole`
  â€” asserts E-RPC-011 wrapping E-ADM-009 in wire response; SVTN unchanged.
- **Test (Go-API defense-in-depth):** `TestSVTNManager_Destroy_ErrDestroyUnauthorized`
  â€” direct call `SVTNManager.Destroy(nonControlKey, "svtn-A")` returns
  `ErrDestroyUnauthorized` (E-ADM-011 Variant 2); direct call with control-role key
  succeeds. This is defense-in-depth at the Go API layer per RULING-W6TB-A Â§3 and
  BC-2.07.001 Inv-3.

### AC-005 (traces to BC-2.07.001 invariant 3 + RULING-W6TB-A genesis re-open)
Destroying the last SVTN (such that `HasAnySVTN()` returns false) re-opens the
genesis carve-out for a subsequent `admin.svtn.create` with the bootstrap key.
This is permitted recovery semantics (not privilege escalation): a subsequent
`admin.svtn.create` call with the bootstrap key succeeds via the genesis carve-out
exactly as on first initialization.
- **Test:** `TestSVTNManager_Destroy_GenesisReopened` â€” destroy last SVTN, then
  verify a subsequent `Create` with bootstrap key succeeds.

## Confirm-Gate Semantics

The `--confirm=SVTN-<8-hex>` gate is a **client-side syntactic-shape check**
enforcing human-in-loop typing ceremony (ADR-004). It is **NOT** a server-side
identity match against the target SVTN â€” the daemon does not receive the confirm
token. The sbctl CLI validates that the value matches the pattern `SVTN-[0-9a-f]{8}`
before sending the destroy RPC; the operator must know and re-type the short-ID
they saw at create-time.

Identity-match enforcement (verifying the confirm token names the same SVTN as
`--name`) is deferred to phase-5 hardening. See drift item
**DRIFT-S605-CONFIRM-IDENTITY** (backlog).

## Error Code Table (Implementer Reference)

Per RULING-W6TB-A:

| Caller path | Gate that fires | Observable error code |
|-------------|----------------|-----------------------|
| Non-control key via mgmt RPC | `resolveAndVerifyCallerRole` (handler layer) | E-RPC-011 wrapping E-ADM-009 |
| Non-control key via direct Go API | `SVTNManager.Destroy` defense-in-depth (`caller.role != RoleControl`) | `ErrDestroyUnauthorized` (E-ADM-011 Variant 2) |
| SVTN not found (any caller) | `SVTNManager.Destroy` | `ErrSVTNNotFound` (E-SVTN-003) |

Error string canonical forms (Ruling-11/12):
- `ErrSVTNNotFound`: `"SVTN not found: <name>"` (colon-space; `<name>` not `<id>`)
- `ErrDestroyUnauthorized`: `"destroy: caller is not a control-role key"`
