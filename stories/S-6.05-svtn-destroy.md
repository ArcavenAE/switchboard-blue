# S-6.05: SVTN Destroy Lifecycle: SVTNManager.Destroy + sbctl admin svtn destroy

> **Execute:** `/vsdd-factory:deliver-story S-6.05`

## Narrative

- **As a** control node operator
- **I want to** destroy an SVTN via `sbctl admin svtn destroy`
- **So that** decommissioned networks are fully cleaned up: all admitted keys
  removed, all active sessions terminated, and the SVTN ID freed

## Acceptance Criteria

### AC-001 (traces to BC-2.07.001 postcondition 3)
`SVTNManager.Destroy(ctx, svtnID)` removes all admitted keys for the SVTN from
the router's key set and frees the SVTN ID. Returns `E-SVTN-003` ("SVTN not
found: <id>") if the SVTN does not exist.
- **Test:** `TestSVTNManager_Destroy_RemovesAllKeys`

### AC-002 (traces to BC-2.07.001 postcondition 3 + EC-002)
`SVTNManager.Destroy(ctx, svtnID)` with active sessions terminates all sessions
with notification before destroying the SVTN. Admitted nodes receive
session-terminated signals.
- **Test:** `TestSVTNManager_Destroy_WithActiveSessions`

### AC-003 (traces to BC-2.07.001 postcondition 3)
`sbctl admin svtn destroy --id <svtn-id>` invokes `SVTNManager.Destroy` and
prints confirmation to stdout on success. Exits with non-zero on E-SVTN-003
(SVTN not found).
- **Test:** `TestSbctlAdmin_SVTNDestroy_HappyPath`, `TestSbctlAdmin_SVTNDestroy_NotFound`

### AC-004 (traces to BC-2.07.001 invariant 3)
Only control-role keys may invoke destroy. A non-control key attempting destroy
receives `E-ADM-011` (authorization error) and the SVTN is not destroyed.
- **Test:** `TestSbctlAdmin_SVTNDestroy_RequiresControlRole`

## Edge Cases

| ID | Scenario | Expected Behavior |
|----|----------|-------------------|
| EC-001 | Destroy SVTN that does not exist | E-SVTN-003 "SVTN not found: <id>"; no action |
| EC-002 | Destroy SVTN with active sessions | Sessions terminated with notification; then SVTN destroyed (per BC-2.07.001 EC-002) |
| EC-003 | Destroy SVTN with multiple admitted nodes on multiple routers (PE topology) | Termination signals sent to all admitted nodes; propagation to other routers (per BC-2.07.001 EC-004) |
| EC-004 | Non-control key attempts destroy | E-ADM-011 authorization error; SVTN unchanged |

## Previous Story Intelligence (MANDATORY)

| Story | Key Decisions | Patterns Established | Gotchas Discovered |
|-------|--------------|---------------------|-------------------|
| S-6.02 | SVTNManager struct created; Create + key lifecycle implemented; `internal/frame` imported for `frame.DeriveNodeAddress` in RevokeKey/ExpireKey | `internal/svtnmgmt/svtnmgmt.go` is the boundary type; SVTN registry keyed by **name** (string) not ID (`[16]byte`); `controlPubKey` bootstrapped at Create time | HOLD-001 hybrid currentRole pattern: `RevokeKey` takes `currentRole admission.KeyRole` as a caller-supplied parameter, then cross-checks it against `keySet.Lookup(svtnID, nodeAddr).Role` before the `--confirm` gate. AC-004 for Destroy **must apply the same hybrid pattern**: `Destroy` must check `currentRole == admission.RoleControl` using the same cross-check path. A non-control key gets `ErrRoleMismatch` before the destroy proceeds. Error code E-ADM-011 (authorization error for destroy) must be a new sentinel: `var ErrDestroyUnauthorized = errors.New("destroy: caller is not a control-role key")` — verify this code exists in the error taxonomy before using; if absent, flag as NEW. |
| S-6.02 | Existing methods (`Create`, `RegisterKey`, `RevokeKey`, `ExpireKey`) do NOT take `context.Context` | Value-copy discipline: `SVTN` struct returned by value (go.md rule 12) | `Destroy` should take `ctx context.Context` as first parameter because it must terminate active sessions (potentially async, requires cancellation). This is a deliberate design divergence from the other methods — add a doc comment explaining why. |
