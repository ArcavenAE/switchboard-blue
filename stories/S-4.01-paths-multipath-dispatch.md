# S-4.01: Per-Path RTT/Loss Tracking and Duplicate-and-Race Dispatch

> **Execute:** `/vsdd-factory:deliver-story S-4.01`

## Narrative

- **As a** router
- **I want to** maintain per-path quality metrics via keep-alive probes and send frames on the two fastest paths simultaneously
- **So that** session resilience is maximized under path failures and the first-arriving copy is delivered while duplicates are silently discarded

## Acceptance Criteria

### AC-001 (traces to BC-2.02.003 postcondition 1)
`PathScore(rtt_ms, loss_pct)` ranks paths deterministically: lower RTT and loss produce a higher score; the ranking is transitive.
- **Test:** `TestPathScore_Transitive`

### AC-002 (traces to BC-2.02.003 postcondition 2)
`PathTracker.OnProbe(arrival_rtt_ms, loss_event)` updates the EWMA RTT and loss estimate for the path. After 3 probe arrivals, the score converges.
- **Test:** `TestPathTracker_EWMAConvergence`

### AC-003 (traces to BC-2.02.001 postcondition 1)
`Multipath.Send(frame, path_set)` dispatches the frame on the two highest-scoring paths simultaneously (duplicate-and-race).
- **Test:** `TestMultipath_SendOnTwoFastestPaths`

### AC-004 (traces to BC-2.02.002 postcondition 1)
`Multipath.Receive(frame)` delivers the first-arriving copy and returns `ErrDuplicate` for subsequent copies with the same checksum.
- **Test:** `TestMultipath_FirstCopyDelivered`

### AC-005 (traces to BC-2.02.002 postcondition 2)
Duplicate discards are silent: no error is surfaced to the session layer for a discarded duplicate.
- **Test:** `TestMultipath_DuplicateDiscardedSilently`

### AC-006 (traces to BC-2.02.009 postcondition 1)
`DropCache` never exceeds its configured capacity (LRU eviction); checksum-based lookup is O(1).
- **Test:** `TestDropCache_BoundedCapacity`

### AC-007 (traces to BC-2.02.009 postcondition 2)
`DropCache.Hits()` returns the cumulative count of cache hits (frames suppressed). The counter increments on every `AddIfAbsent` call where the key is already present, and on every `Add` call where the key is already present.
- **Test:** `TestBC_2_02_009_DropCache_HitCounterIncremented`

## Edge Cases

| ID | Scenario | Expected Behavior |
|----|----------|-------------------|
| EC-001 | Only one path available | Send dispatches on single path; no error |
| EC-002 | Both paths have identical score | Either path selected; deterministic tiebreak by path ID |
| EC-003 | DropCache at capacity; new checksum arrives | LRU entry evicted; new checksum inserted |
| EC-004 | Frame arrives 3+ times (looping network) | Second and subsequent arrivals → ErrDuplicate (drop cache hit) |

## Deferrals / Scope Boundary

The following items are explicitly out of scope for S-4.01 per the product-owner's
Pass-2 BC-2.02.009 scope ruling (`.factory/cycles/cycle-1/S-4.01/adjudications/pass2-bc009-scope.md`):

**1. BC-2.02.009 router-side OnFrameArrival / forwarding wiring — DEFERRED to S-4.04**

`DropCache` is delivered here as a standalone exported primitive in `internal/multipath`
per AC-006 and AC-007. No `Forward`, `OnFrameArrival`, or router-path method on `Multipath`
is required by S-4.01. The router-side wiring that consults the compound-key
`(checksum, arrival_interface_id)` drop cache for loop suppression belongs to
`internal/routing` — the layer where `SplitHorizon.Forward` and the `RouteFrame` hot path
live (S-4.04). S-4.04 must add `BC-2.02.009` to its `bc_traces` and implement the
router-side wiring per ARCH-03 §Duplicate-and-Race `OnFrameArrival` pseudocode.

Additionally, the `dropCache *DropCache` field on the `Multipath` struct (if present
from an earlier draft) must be removed as dead code — it implies router-side loop
suppression inside `Multipath.Receive`, which is not the contract. The `DropCache`
type remains as a standalone export; callers (specifically S-4.04) construct it
directly.

**2. BC-2.02.009 EC-005 collision-event logging — DEFERRED to S-4.04**

EC-005 mandates that checksum collisions (two different frames hashing to the same
checksum on the same interface) be logged as a potential collision event for
investigation. Logging is an effectful concern. `internal/multipath` is classified
`pure-core` (see Purity Classification below) and must not acquire an I/O dependency.
EC-005 collision logging must be implemented at the wired boundary layer by injecting
a logger when `DropCache` is wired into the router forwarding path, following the
established `internal/routing.WithLogger` injection pattern. This wired layer is
S-4.04.

**3. BC-2.02.001 EC-003 / postcondition-4 queue-with-timeout + E-NET-002 — OUT OF SCOPE**

BC-2.02.001 EC-003 (queue-with-timeout when no path available) and its associated
`E-NET-002` error raising are not in scope for S-4.01. `Multipath.Send` in the
pure-core layer returns an error on total failure; queueing, timer management, and
`E-NET-002` propagation are effectful downstream concerns. The owning layer is
whichever story wires `Multipath.Send` into the daemon's network-path management
(currently unscheduled; nearest candidate is S-BL.NI or a future Wave-4+ story
handling live-path ingress/egress).

**4. VP-040 end-to-end failover-recovery assertion — DEFERRED to integration harness (same disposition as VP-054)**

VP-040 (`proof_method: e2e`) asserts that multipath failover recovery completes in < 2 s under a live multi-router topology. Verifying this requires the `internal/testenv` multi-router harness, which is outside the pure-core module set (`internal/paths`, `internal/multipath`) for S-4.01. S-4.01 delivers and unit-tests the failover **building blocks** — 3-consecutive-miss deactivation, reactivation on the first successful probe per BC-2.02.003 v1.2, and Rank exclusion of inactive paths — satisfying the unit-level contribution toward VP-040. The VP-040 e2e gate (timing assertion, `//go:build integration`) belongs to the integration harness wave-gate alongside VP-054.
