# S-5.02: Expose Per-Path Metrics via sbctl (Canonical + Alias + p99)

> **Execute:** `/vsdd-factory:deliver-story S-5.02`

## Narrative

- **As an** operator
- **I want to** query per-path RTT, p99 RTT, loss, and quality metrics via `sbctl paths list` (canonical), `sbctl router metrics --svtn=<id>` (canonical), or `sbctl router status --target <router>` (convenience alias)
- **So that** I can diagnose connection quality without having to interpret raw logs, and distinguish per-path latency from application-level problems

## Acceptance Criteria

### AC-001 (traces to BC-2.06.003 postcondition 1 — canonical `sbctl paths list`)
sbctl client correctly serializes and forwards the `paths.list` request and envelopes the daemon response into the BC-2.06.003 PC-1 schema. When `--json` is passed, output is valid JSON conforming to that schema. When ≥10 probes collected, `rtt_p99_ms` is a float64; when <10 probes, `rtt_p99_ms` is the string `"pending"`. Verified against `startCannedDaemon` stub in `cmd/sbctl/router_status_test.go`; end-to-end daemon wiring lands in S-W5.04.
- **Test:** `TestSbctlPathsList_OutputsCanonicalFields`

### AC-002 (traces to BC-2.06.003 postcondition 2 — canonical `sbctl router metrics`)
sbctl client correctly serializes the `router.metrics --svtn=<id>` request and envelopes the daemon response into the BC-2.06.003 PC-2 schema. JSON output when `--json` is passed conforms to that schema. Verified against `startCannedDaemon` stub; daemon-side `RouterMetricsResponse` type and handler registration land in S-W5.04.
- **Test:** `TestSbctlRouterMetrics_OutputsSVTNMetrics`

### AC-003 (traces to BC-2.06.003 postcondition 3 — alias `sbctl router status`)
`sbctl router status --target <router>` is a CLI dispatch shim; JSON structure (minus `quality`) is identical to `sbctl paths list` response; the alias adds a `quality` column (green/yellow/red/pending) per BC-2.06.003 PC-3. There are no divergent code paths in `cmd/sbctl`. The `--target <router>` flag overrides the default daemon address. Verified against `startCannedDaemon` stub.
- **Test:** `TestSbctlRouterStatus_IsAliasForPathsList` — assert that the JSON output (minus `quality`) is structurally identical to `sbctl paths list` output, and that both invoke the same `internal/metrics` query function.

### AC-004 (traces to BC-2.06.003 EC-003 — p99 pending state)
sbctl client correctly passes through the `rtt_p99_ms` union value from the daemon response: string `"pending"` when `SampleCount < 10`, float64 otherwise. Verified via canned-daemon stub that returns both variants. Daemon-side `SampleCount` serialization lands in S-W5.04.
- **Test:** `TestSbctlPathsList_P99Pending_LessThan10Samples`

### AC-005 (traces to BC-2.06.003 PC-1, ARCH-03 v1.6 §p99 RTT Accumulator — histogram accuracy)
The `internal/paths` histogram implementation satisfies `P99RTTMs ≤ true_p99 + max_bucket_width` for synthetic sample distributions. Test cases must cover: (a) all samples in one bucket, (b) samples distributed across the 0–25ms bucket and the 100–150ms bucket, (c) a distribution where the p99 falls in the 150–200ms bucket. This AC is purely internal/paths accuracy; no daemon RPC surface involved.
- **Test:** `TestBC_2_06_003_P99HistogramAccuracy` — table-driven accuracy test asserting the histogram approximation bound. NOTE: a formal accuracy VP for the accumulator is deferred to S-BL.BENCH (VP stub — no VP ID assigned; do not invent one).

### AC-006 (traces to BC-2.06.003 postcondition 4 — JSON and daemon unreachable)
Both `sbctl paths list` and `sbctl router status --target <router>` respect `--json`. When the daemon is unreachable, both return E-NET-001 "daemon unreachable" with exit code 1 (per BC-2.07.003). The daemon-unreachable path exercises real client dial logic and requires no stub — it is fully verifiable without a running daemon.
- **Test:** `TestSbctlMetrics_JSONEnvelope` and `TestSbctlMetrics_DaemonUnreachable`

### AC-008 (traces to BC-2.06.003 v1.7 postcondition 3 + EC-006 — pending-quality sentinel)
The sbctl client-side alias correctly emits `quality: "pending"` when `rtt_p99_ms == "pending"` (BC-2.06.003 v1.7 PC-3 + EC-006). The `quality` column emitted by `sbctl router status` enumerates `green|yellow|red|pending`; the `pending` value is derived from the canned-daemon stub response where p99 is in pending state. This is pure client-side field derivation — `qualityFromPathEntry` in `cmd/sbctl/router_status.go` — verified against `startCannedDaemon`.
- **Test:** `TestSbctlRouterStatus_QualityPending_WhenP99Pending`

## Command Surface Reconciliation (BC-2.06.003 v1.7)

Per BC-2.06.003 v1.7 (Wave-5 reconciliation):
- **Canonical commands:** `sbctl paths list` (PC-1) and `sbctl router metrics --svtn=<id>` (PC-2).
- **Alias:** `sbctl router status --target <router>` is a documented convenience alias for `sbctl paths list`. It is a CLI dispatch shim only. There is exactly one code path in `internal/metrics` serving both invocations — no divergent implementations.
- The `quality` column (green/yellow/red) is added by the alias's output formatter; the underlying data source is identical.

## p99 RTT Scope (ARCH-03 v1.6 Note)

Per ARCH-03 v1.6 §p99 RTT Accumulator:
- A fixed-bucket latency histogram (16 buckets, 0–2000ms) lives in `internal/paths` as `rttHistogram` embedded in `PathTracker`.
- `P99()` is exposed via `PathSnapshot.P99RTTMs float64`.
- `internal/metrics` reads `PathSnapshot.P99RTTMs` from `Snapshot()` and serializes it as `rtt_p99_ms` in JSON output.
- Approximation accuracy: `p99() ≤ true_p99 + max_bucket_width`. Formal VP deferred to S-BL.BENCH (VP stub — no VP ID; do not invent one).

## Scope Boundary

- `sbctl paths list`, `sbctl router metrics`, and `sbctl router status` alias are in scope for S-5.02.
- `sbctl sessions list` quality surfacing (BC-2.06.001 PC-5) deferred to S-7.03 per Pass-3 Ruling 1 (decisions/S-5.02-pass3-scope-ruling.md, commit b45bfc4). Both console and sbctl halves of PC-5 now belong to S-7.03.

### Daemon-Side Deferral (Pass-4 Ruling 1)

Daemon-side `paths.list` / `router.metrics` / `router.status` RPC handlers (`PathsListResponse`, `PathEntry`, `RouterMetricsResponse` types and the JSON wiring of `P99RTTMs` → `rtt_p99_ms` and `Degraded` → `status`) are deferred to S-W5.04. This story delivers the sbctl client-side surface only; ACs are verified against a stub daemon (`cmd/sbctl/router_status_test.go startCannedDaemon`). End-to-end behavior (operator → daemon → real per-path metrics) lands in S-W5.04.

This split follows the same pattern as `sbctl sessions list` quality surfacing (S-7.03 precedent): a clean per-story BC ownership boundary where S-5.02 owns the client and S-W5.04 owns the daemon half.
