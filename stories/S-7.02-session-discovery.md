# S-7.02: SVTN-Scoped Multicast Session Discovery

> **Execute:** `/vsdd-factory:deliver-story S-7.02`

## Narrative

- **As a** console operator
- **I want to** enumerate all available sessions on my SVTN without specifying hostnames or IP addresses
- **So that** I can attach to any session by name, even if I don't know which access node is hosting it

## Acceptance Criteria

### AC-001a (traces to BC-2.03.001 postcondition 3 — state-change trigger)
`Discovery.Advertise(session_presence)` sends a presence advertisement within 1 tick interval when triggered by a session state change (session added, removed, or attachment status changed).
- **Test:** `TestDiscovery_Advertise_OnStateChange`

### AC-001b (traces to BC-2.03.001 postcondition 4 — periodic heartbeat)
`Discovery.Advertise` re-sends the current presence state unconditionally every 30 seconds regardless of whether any state change occurred. The heartbeat is independent of state-change advertisements. The heartbeat timer MUST produce an observable side effect — either an exported counter field or an injected tick channel exposed via `Config` — so the test oracle can assert that N ticks produce exactly N heartbeat events. A no-op tick body that a removed ticker would leave green is **insufficient and explicitly rejected**: if the test passes after removing the ticker, the oracle is broken.
- **Test:** `TestDiscovery_Advertise_PeriodicHeartbeat` — inject tick source via `Config`; advance N ticks; assert counter/channel records N heartbeat events.

### AC-002 (traces to BC-2.03.002 postcondition 1, postcondition 3, invariant 1)
`Discovery.Enumerate()` returns all sessions visible on the local SVTN without requiring hostnames or IP addresses from the caller. The returned set MUST include sessions from at least 2 distinct simulated advertisers — a single-advertiser result is not sufficient to verify BC-2.03.002 PC-3 aggregation. Each entry must carry the advertising node's identity.
- **Oracle assertion:** `len(distinctNodeAddrs(result)) >= 2`
- **Test:** `TestDiscovery_Enumerate_NoHostnameRequired`

### AC-003 (traces to BC-2.03.003 postcondition 1)
Presence advertisement payload includes: session_name, attachment_status (attached/detached), and quality_indicator (green/yellow/red).
- **Test:** `TestDiscovery_Advertisement_RequiredFields`

### AC-004 (traces to BC-2.03.003 invariant 1)
Presence advertisement fields are stable across round-trips: `Encode(Decode(payload)) == payload` for any valid advertisement.
- **Test:** `TestDiscovery_AdvertisementRoundTrip`

### AC-005 (traces to BC-2.03.001 postcondition 5 — HMAC authentication)
Advertisement frames are authenticated using the SVTN HMAC key (via `internal/routing`). A receiver that receives an advertisement with a missing or invalid HMAC tag MUST reject it — it does not update the local session list. `internal/discovery` uses the HMAC capability provided by `internal/routing` (ARCH-08 position 14: discovery imports routing, not hmac directly).
- **Test:** `TestDiscovery_Advertise_HMACAuthenticated` (verify advertisement is rejected when HMAC tag is wrong)

> **DRIFT-W6TBD-001 (scoping placeholder):** `advertisementKey(svtnID) = svtnID` uses the SVTN ID directly as the 16-byte HMAC key. This is internally consistent for the in-process registry model but is NOT admitted-node-scoped secret material. Real multicast authentication requires keying material from the Tier-1 admission layer. This is resolved in S-BL.DISCOVERY-WIRE, which must specify admitted-node HMAC key derivation in a new or amended BC before implementation. The fail-closed rejection behavior (`ErrInvalidHMACTag` on wrong tag) is fully verified in S-7.02.

### AC-006 (traces to BC-2.03.002 invariant 1 — SVTN cross-scope negative)
`Discovery.Enumerate()` on SVTN-A returns ZERO sessions advertised by a node on SVTN-B. A simulated cross-SVTN advertisement (wrong SVTN scope) must not appear in the result set for SVTN-A.
- **Oracle assertion:** `len(sessionsFromSVTNB(svtnAResult)) == 0`
- **Test:** `TestDiscovery_Enumerate_SVTNIsolation`
