# S-7.02: SVTN-Scoped Multicast Session Discovery

> **Execute:** `/vsdd-factory:deliver-story S-7.02`

## Narrative

- **As a** console operator
- **I want to** enumerate all available sessions on my SVTN without specifying hostnames or IP addresses
- **So that** I can attach to any session by name, even if I don't know which access node is hosting it

## Acceptance Criteria

### AC-001a (traces to BC-2.03.001 postcondition 3 — state-change trigger)
`Discovery.Advertise(session_presence)` sends a presence advertisement within 1 tick interval when triggered by a session state change (session added, removed, or attachment status changed).
- **Test:** `TestDiscovery_Advertise_OnStateChange`

### AC-001b (traces to BC-2.03.001 postcondition 4 — periodic heartbeat)
`Discovery.Advertise` re-sends the current presence state unconditionally every 30 seconds regardless of whether any state change occurred. The heartbeat is independent of state-change advertisements.

**Config surface (RULING-W6TB-G):** `Config` gains an optional field `TickSource <-chan time.Time`. When non-nil, `Run()` selects on `TickSource` for heartbeat ticks instead of creating a `time.NewTicker`. Production callers MUST leave this nil; test-only injection seam.

**Nil observer policy (Pass-3 L1 M-1):** `HeartbeatObserver` MUST NOT be silently nil-safe in production. Recommended approach (b): `Discovery` maintains an atomic `uint64` `HeartbeatCount()` accessor unconditionally — a production observability counter that does not require a wired observer. `HeartbeatObserver` (if provided) is called on each heartbeat tick in addition to incrementing the counter. This delivers production observability without wiring burden. `New`/`NewWithConfig` MAY accept a nil observer; the counter still increments.

- **Tests (RULING-W6TB-G dual-test split):**
  - `TestDiscovery_Advertise_PeriodicHeartbeat_ExactN` (**primary oracle**) — inject `Config.TickSource` (buffered `chan time.Time`); send exactly N ticks synchronously; assert `HeartbeatObserver` fires exactly N times. A removed ticker body causes count == 0 and MUST fail (RULING-W6TB-G no-op-removal oracle). No wall-clock sensitivity.
  - `TestDiscovery_Advertise_PeriodicHeartbeat` (**integration sanity**) — real `time.Ticker`; wide tolerance `[N/2, 2N]`; documented as flake-resistance only; detects catastrophic failures (heartbeat never fires). Update test comment to state: "Integration sanity check using real time.Ticker. Wide tolerance [N/2, 2N] is intentional: wall-clock jitter prevents exact counting. Exact-N oracle is in TestDiscovery_Advertise_PeriodicHeartbeat_ExactN (RULING-W6TB-G). This test detects catastrophic failures only."

### AC-002 (traces to BC-2.03.002 postcondition 1, postcondition 3, invariant 1)
`Discovery.Enumerate()` returns all sessions visible on the local SVTN without requiring hostnames or IP addresses from the caller. The returned set MUST include sessions from at least 2 distinct simulated advertisers — a single-advertiser result is not sufficient to verify BC-2.03.002 PC-3 aggregation. Each entry must carry the advertising node's identity.
- **Oracle assertion:** `len(distinctNodeAddrs(result)) >= 2`
- **Test:** `TestDiscovery_Enumerate_NoHostnameRequired`

### AC-003 (traces to BC-2.03.003 postcondition 1)
Presence advertisement payload includes: session_name, attachment_status (attached/detached), and quality_indicator (green/yellow/red).
- **Test:** `TestDiscovery_Advertisement_RequiredFields`

### AC-004 (traces to BC-2.03.003 invariant 1)
Presence advertisement fields are stable across round-trips: `Encode(Decode(payload)) == payload` for any valid advertisement.
- **Test:** `TestDiscovery_AdvertisementRoundTrip`

### AC-004b (traces to BC-2.03.003 postcondition 2 + EC-001 — 255-byte session name cap, Pass-3 L1 M-2)
`Encode` enforces the 255-byte UTF-8 session name limit at encode time.

**Truncation semantics (BC-2.03.003 EC-001 / canonical test vector):** If the input session name exceeds 255 bytes, `Encode` truncates to 252 bytes and appends "…" (U+2026 HORIZONTAL ELLIPSIS, 3 UTF-8 bytes = 255 bytes total). Truncation is applied to the upstream string input at encode time only. `Decode` does not truncate — it trusts that the encoded name is already within bounds.

**VP-055 boundary test cases (Pass-3 L1 M-2):**
- Session name at exactly 255 bytes: `Encode` accepts without truncation (no "…" appended).
- Session name at exactly 256 bytes: `Encode` truncates to 252 bytes + "…" (= 255 bytes total).

- **Test:** `TestDiscovery_Encode_SessionName255ByteCap`

### AC-005 (traces to BC-2.03.001 postcondition 5 — HMAC authentication)
Advertisement frames are authenticated using the SVTN HMAC key (via `internal/routing`). A receiver that receives an advertisement with a missing or invalid HMAC tag MUST reject it — it does not update the local session list. `internal/discovery` uses the HMAC capability provided by `internal/routing` (ARCH-08 position 14: discovery imports routing, not hmac directly).

**Ordering note (RULING-W6TB-H):** `ReceiveAdvertisement` MUST verify HMAC BEFORE the SVTN cross-scope check (`payload.SVTNID != LocalSVTNID`). The HMAC key is derived from the declared `payload.SVTNID` (not from `LocalSVTNID`) so that admitted nodes on other SVTNs are authenticated with their own key and then correctly rejected by the SVTN check. Unauthenticated frames MUST return `ErrInvalidHMACTag` before any SVTN comparison occurs.

Required `ReceiveAdvertisement` ordering:
1. Extract wire HMAC tag.
2. Decode body minimally to read `SVTN_ID` for key derivation.
3. Derive HMAC key: `advertisementKey(payload.SVTNID)` (NOT `d.cfg.LocalSVTNID`).
4. Verify HMAC → return `ErrInvalidHMACTag` on failure.
5. Check `payload.SVTNID != d.cfg.LocalSVTNID` → return `ErrSVTNMismatch`.
6. Store sessions.

**Error semantics:**
- Forged-SVTN attackers (cannot produce valid HMAC): receive `ErrInvalidHMACTag` only.
- Admitted nodes on foreign SVTNs (valid HMAC for their own key, wrong SVTN): receive `ErrSVTNMismatch` after their HMAC verifies.

- **Test:** `TestDiscovery_Advertise_HMACAuthenticated` (verify advertisement is rejected when HMAC tag is wrong → `ErrInvalidHMACTag`)

> **DRIFT-W6TBD-001 (scoping placeholder):** `advertisementKey(svtnID) = svtnID` uses the SVTN ID directly as the 16-byte HMAC key. This is internally consistent for the in-process registry model but is NOT admitted-node-scoped secret material. Real multicast authentication requires keying material from the Tier-1 admission layer. This is resolved in S-BL.DISCOVERY-WIRE, which must specify admitted-node HMAC key derivation in a new or amended BC before implementation. The fail-closed rejection behavior (`ErrInvalidHMACTag` on wrong tag) is fully verified in S-7.02.

### AC-006 (traces to BC-2.03.002 invariant 1 — SVTN cross-scope negative)
`Discovery.Enumerate()` on SVTN-A returns ZERO sessions advertised by a node on SVTN-B. A simulated cross-SVTN advertisement (wrong SVTN scope) must not appear in the result set for SVTN-A.
- **Oracle assertion:** `len(sessionsFromSVTNB(svtnAResult)) == 0`
- **Test:** `TestDiscovery_Enumerate_SVTNIsolation`
