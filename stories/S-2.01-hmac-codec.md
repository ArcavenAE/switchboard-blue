# S-2.01: Implement HMAC-SHA256 Frame Authentication in internal/hmac

> **Execute:** `/vsdd-factory:deliver-story S-2.01`

## Narrative

- **As a** router
- **I want to** authenticate every incoming frame using HMAC-SHA256 with a per-(node, SVTN) HKDF-derived key
- **So that** only frames from legitimately admitted nodes are forwarded — unauthenticated or forged frames are dropped at the first router boundary

## Acceptance Criteria

### AC-001 (traces to BC-2.05.005 precondition 2 (tag = first 8 bytes of HMAC-SHA256))
`ComputeHMAC(key, frame_bytes)` produces an 8-byte (64-bit) truncated HMAC-SHA256 tag.
- **Test:** `TestComputeHMAC_EightByteTag`

### AC-002 (traces to BC-2.05.005 postcondition 1 (HMAC verification succeeds → frame forwarded))
`VerifyHMAC(key, frame_bytes, tag)` returns true when tag matches the expected HMAC-SHA256 truncated to 8 bytes.
- **Test:** `TestVerifyHMAC_ValidTag`

### AC-003 (traces to BC-2.05.005 postcondition 2 (HMAC verification fails → frame dropped))
`VerifyHMAC` returns false for any single-bit flip in the frame payload.
- **Test:** `FuzzVerifyHMAC_SingleBitFlip`

### AC-004 (traces to BC-2.05.005 postcondition 2 (HMAC verification fails on wrong key → frame dropped))
`VerifyHMAC` returns false when a different key is used to verify a tag computed with the correct key.
- **Test:** `TestVerifyHMAC_WrongKey`

### AC-005 (traces to BC-2.05.005 precondition 2 (HKDF-SHA256 keying definition))
`DeriveKey(node_admission_pubkey, svtn_id)` uses HKDF-SHA256 with `node_admission_pubkey` as IKM to produce a per-(node, SVTN) key. Same inputs always produce same output (deterministic).
- **Test:** `TestDeriveKey_Deterministic`
- **Note (amended 2026-06-24):** Inline HKDF via `crypto/hmac` + `crypto/sha256` (RFC 5869 Extract + single Expand iteration) is permitted in place of `golang.org/x/crypto/hkdf`. If inline is used, add `TestDeriveKey_RFC5869_KAT` covering the RFC 5869 §A.1 known-answer vector.

## Edge Cases

| ID | Scenario | Expected Behavior |
|----|----------|-------------------|
| EC-001 | Empty frame bytes | ComputeHMAC still produces a valid 8-byte tag |
| EC-002 | SVTN ID of all zeros | DeriveKey accepts; produces a valid key |
| EC-003 | Tag mismatch / corruption | `VerifyHMAC` returns `false` without panic. The `[TagSize]byte` signature makes "tag too short" structurally impossible at compile time; this EC covers the equivalent runtime case of any tag value (including zero-valued bytes) that does not match the recomputed HMAC. **Test:** `TestVerifyHMAC_ZeroTagRejected` (zero-byte tag is the canonical sample; equivalent for any non-matching value via constant-time compare). |
