## AC-006: Bounded Reads (VP-066)

Tests: TestMgmtServer_BoundedRead_VP066, FuzzMgmtServer_BoundedRead_VP066

Observable behavior:

Sub-case (under_limit_challenge_consumed):
- Small CHALLENGE_RESPONSE (well under MaxMessageBytes=65536) is handled without error.
- Server returns AUTH_FAIL or closes (random key presented) — acceptable; the point is it does NOT hang.

Sub-case (oversized_message_closes_connection):
- Payload of MaxMessageBytes+1 bytes sent to server after consuming CHALLENGE.
- io.LimitReader(conn, MaxMessageBytes) causes json.Decoder to return an error at the limit.
- Server closes the connection cleanly; next read from client returns error (connection closed).
- Process does NOT OOM (VP-066).

Fuzz baseline (4 seed corpus entries):
- Seed#0: valid-looking challenge_response JSON with short nonce_sig/pubkey values.
- Seed#1: 65536+512 bytes of 'A' characters (exceeds MaxMessageBytes).
- Seed#2: "not json at all" — malformed; triggers decode error path.
- Seed#3: raw bytes {0x00, 0xFF, 0xFE} — binary garbage; triggers decode error path.
- All seeds: server does not panic; connection is closed within deadline. Property holds.

Implementation evidence:
- internal/mgmt/mgmt.go: dec := json.NewDecoder(io.LimitReader(conn, MaxMessageBytes))
  applied at EVERY decode site in handleConnection (handshake phase AND RPC phase).
- mgmt.MaxMessageBytes = 1 << 16 = 65536 bytes (exported constant).
